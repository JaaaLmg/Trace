# TRACE 测试计划



## 1. 文档目的

本文档用于指导 TRACE 项目的测试设计、测试执行、缺陷跟踪和测试报告输出。测试计划覆盖后端服务、智能体运行时、工具层、执行器、评测体系、数据集管理、前端页面、Docker Compose 交付栈以及非功能质量要求。后续建立测试框架、编写测试用例、执行回归测试和生成项目测试报告，均以本文档为主要依据。

TRACE 的测试重点不同于普通管理系统。除常规接口、数据模型和前端页面外，还必须验证智能体生成测试的工程闭环、执行结果的结构化记录、反思修复的边界、评测指标的可信度，以及 replay 过程中不污染测试集的规则。测试计划需要同时覆盖“系统能否运行”和“系统给出的评测结论是否可信”两个层面。

本文档参考软件测试计划的通用结构组织内容，包含测试范围、测试对象、测试策略、测试环境、测试数据、测试类型、准入准出、执行安排、风险控制和交付物。具体测试项以《需求分析.md》《系统设计.md》定义的业务流程、模块边界和验收标准为基础，并结合项目迭代中形成的接口合同、测试资产和交付方式进行细化。



## 2. 测试目标

TRACE 测试工作的总目标是证明系统能够稳定完成测试生成、执行、追踪和评测闭环，并且在出现异常时给出可追溯、可解释的结果。测试结果应能支撑项目验收，也能为后续测试报告提供可量化数据。

功能层面，测试应覆盖项目创建、快照管理、运行配置、策略版本、测试计划、测试运行、异步任务、报告导出、数据集管理、实验创建、实验启动、metrics 查询、artifact 查询、cleanup 和前端展示链路。每个主要接口都需要验证正常路径、参数错误、资源不存在、状态冲突和权限或路径边界。

智能体层面，测试应覆盖 Direct、Plan-and-Execute、ReAct + Reflection 三类策略，验证项目分析、上下文构建、提示词输入、结构化输出、Contract Guard、失败上下文回灌、一次反思修复和最终报告。重点关注模型输出不稳定、JSON 结构不合法、目标漂移、弱断言、未知 fixture 和 request/response 字段无证据等场景。

评测层面，测试应证明 clean run、clean replay、variant replay、sampled mutation score、capture matrix 和 EvaluationEvent 的计算口径正确。无效测试集、provider failure、artifact hash mismatch、setup failure、executor failure 和 flaky clean replay 必须与 0 捕获区分开。replay 只能执行冻结测试集，不能调用 LLM。

交付层面，测试应证明本地开发环境和 Docker Compose 环境均可启动、运行和验证。最终测试报告需要包含后端测试结果、前端构建结果、Compose smoke、benchmark suite、UI 彩排、已知风险和剩余缺口。



## 3. 测试范围

本轮测试覆盖 TRACE V1 至 V2.4 的全部交付能力。测试范围按系统架构划分为后端 API、数据模型、智能体运行时、工具层、执行器与运行配置、评测系统、数据集与 mutation、前端可视化、交付脚本和非功能质量。

| 测试域 | 主要对象 | 覆盖重点 | 代表性验证方式 |
| --- | --- | --- | --- |
| 后端 API | projects、test-plans、test-runs、experiments、runtime-profiles、llm-options、evaluation | 请求校验、状态码、资源状态、错误分类、返回结构 | FastAPI TestClient、接口集成测试 |
| 数据模型与迁移 | SQLAlchemy models、Alembic versions、repositories | 表结构、外键、JSON 字段、迁移顺序、种子数据 | schema migration 测试、仓储测试 |
| 智能体运行时 | orchestrator、strategies、prompts、structured output、report | 三策略流程、结构化输出、反思修复、报告内容 | MockLLM 端到端测试、策略单元测试 |
| 工具层 | analyze、write_test、run_pytest、search、lsp、fs_tools | 路径边界、AST 分析、pytest 解析、工具输出结构 | 工具单元测试、临时项目执行测试 |
| 执行与运行配置 | local_subprocess、runtime profile、executor command、rootdir | 独立工作目录、命令拼装、超时、网络策略、preflight | 执行器测试、runtime profile 合同测试 |
| 评测系统 | eval dataset、experiment runner、replay、metrics、events | clean/replay 分层、捕获率、误报率、invalid 分类、LLM 调用隔离 | experiment runner 测试、benchmark smoke |
| 数据集与 mutation | dataset authoring、readiness、seeded bug、variant、mutation discovery | task/bug/variant 合同、probe confirmation、operator 分母 | dataset API 测试、mutation discovery 测试 |
| 前端可视化 | RunConsole、Experiment、Dataset、Runtime、Comparison、Report | 数据加载、状态展示、阻塞原因、cleanup 确认、导出入口 | 类型检查、生产构建、UI 彩排 |
| 交付脚本 | docker-compose、compose_smoke、compose_benchmark | 服务启动、初始化、smoke、benchmark JSON 输出 | PowerShell 脚本执行、Compose 环境验证 |
| 非功能质量 | 安全、可靠性、可观测性、可复现性、兼容性 | 脱敏、幂等、日志字段、artifact hash、配置隔离 | 专项测试、代码审查、故障注入 |

## 4. 测试策略

测试策略采用分层组合方式。底层以单元测试和合同测试保证数据结构、工具输出和业务规则稳定；中层以接口测试和集成测试覆盖 API、数据库、异步任务和执行器；上层以端到端测试、benchmark 和 UI 彩排验证真实业务链路。对于智能体和评测能力，额外引入变异测试思想，通过 seeded bug 和 auto mutation 验证生成测试是否能捕获已知缺陷。

测试金字塔的底部由后端 pytest 测试构成。工具函数、服务函数、Contract Guard、mutation discovery、source context、metrics 计算和 runtime profile 合同都应尽量通过确定性单元测试覆盖。中间层覆盖 FastAPI 路由、数据库事务、Celery eager 执行、experiment runner 和 replay scheduler。顶部保留少量高价值端到端测试，包括 MockLLM run、真实 LLM smoke、Compose smoke 和三策略 benchmark。

前端测试以类型检查、生产构建和手工/半自动 UI 彩排为主。当前前端未引入独立组件测试框架，后续可以基于 Vitest 和 Playwright 增补组件测试与浏览器端回归。现阶段的前端准入要求是 `npm run build` 通过，关键页面按 runbook 完成数据加载、状态展示、按钮可用性和导出入口检查。

测试数据分为三类：第一类是后端单元和接口测试使用的临时数据库、fixture 项目和 MockLLM 数据；第二类是评测使用的 demo dataset、clean snapshot、seeded bug variant 和 auto mutation variant；第三类是交付验证使用的 Docker Compose 初始化数据和外部项目 smoke 数据。不同类型数据不能混用，特别是 demo metrics 不能作为真实 API 失败时的兜底结果。



## 5. 测试类型与方法

### 5.1 单元测试

单元测试用于验证单个函数、类或服务方法的输入输出。后端单元测试使用 pytest 执行，重点覆盖不依赖完整服务启动的业务逻辑，例如路径策略、错误码、环境变量模板脱敏、运行配置规范化、source context 切片、Contract Guard 规则、mutation discovery operator、metrics 聚合和 EvaluationEvent policy。

单元测试需要采用等价类划分、边界值分析和异常路径覆盖。以 runtime profile 为例，local_subprocess 的 network policy、resource limit、replay policy 数值边界都应有正常值、缺省值、非法值和边界值用例。以 Contract Guard 为例，需要覆盖弱断言、缺 fixture 证据、方法路径漂移、request field 无证据、response model 字段不匹配和源码行为矛盾等类别。

### 5.2 接口测试

接口测试覆盖 FastAPI 对外路由，验证请求参数、响应结构、状态码和错误信息。项目、快照、测试计划、测试运行、策略版本、运行配置、数据集、实验、LLM option、cleanup 和 replay cache 都应具备接口测试。

接口测试需要覆盖正常创建、查询列表、查询详情、更新、归档、资源不存在、状态冲突、字段非法、引用关系错误和重复请求等场景。对于创建 experiment、启动 experiment、清理 workspace、confirm mutation 等高风险操作，应验证服务端不会信任前端派生状态，必须重新校验 dataset readiness、runtime profile 归属、LLM option 可选性和 mutation discovery 结果。

### 5.3 集成测试

集成测试用于验证多个模块协同后的行为。TRACE 的核心集成场景包括 API + 数据库、API + Worker、Worker + Agent、Agent + tools、tools + pytest、experiment runner + replay scheduler、cleanup + artifact inventory、source context + Contract Guard。

集成测试应尽量使用真实数据库事务和临时文件目录，避免用过多 mock 掩盖模块边界问题。异步任务可以通过 Celery eager 模式执行，以保证测试速度和可重复性。涉及真实 LLM 的集成测试应单独标记为 smoke，不进入默认快速回归。

### 5.4 端到端测试

端到端测试验证用户视角的完整链路。普通测试生成链路应覆盖：创建项目，创建快照，创建测试计划，触发 run，等待 completed 或 failed，查看 trace、pytest results、artifacts 和 report。评测链路应覆盖：选择 dataset，创建三策略 experiment，启动 draft experiment，等待 completed，查看 metrics、EvaluationEvent、Runtime evidence、Replay evidence、Report/Export 和 cleanup dry-run。

端到端测试分为 MockLLM 和真实 LLM 两类。MockLLM 用于稳定验证系统流程、状态机和数据持久化；真实 LLM 用于验证模型配置、输出结构、Contract Guard retry、token 限制和真实失败形态。真实 LLM smoke 需要与 MockLLM 结果分开记录，避免把确定性流程验证和模型能力验证混为一个结论。

### 5.5 回归测试

回归测试在每个阶段收尾、主链路合并、缺陷修复和交付前执行。后端回归以 `backend/tests` 全量 pytest 为主；前端回归以 `npm run build` 为主；交付回归包括 Compose smoke 和 benchmark suite。对数据库迁移、experiment runner、Contract Guard、runtime profile、mutation denominator、cleanup 和 LLM option 的改动，应追加目标测试后再执行全量回归。

回归测试需要保存命令、执行时间、通过数、失败数、跳过数、warning 类型和失败摘要。测试报告中应说明本次回归覆盖了哪些模块，哪些测试因外部环境缺失被跳过，哪些 warning 属于已知非阻塞问题。

### 5.6 评测有效性测试

评测有效性测试是 TRACE 的重点。测试需要验证 clean run、clean replay、variant replay 的分层关系，确认 false positive 只来自 clean baseline 失败，captured bug 只来自 bug variant replay 的有效 assertion failure。provider failure、setup failure、executor failure、artifact hash mismatch、invalid test set 和 flaky clean replay 均不能计为 0 捕获。

sampled mutation score 的测试应覆盖普通 seeded bug、auto mutation probe passed、auto mutation probe failed、缺失 probe_check、非 auto mutation、未完成 replay 等情况。分母只允许 confirmed 且 probe passed 的 auto mutation variant 进入。mutation operator 测试需要覆盖 comparison negation、arithmetic operator、boolean negation、constant replacement、return value 等已支持类型，并验证服务端 rediscovery 和 probe confirmation。

### 5.7 安全测试

安全测试重点覆盖路径边界、执行边界、配置脱敏和危险输入处理。Agent 工具只能在白名单目录内读写，`write_test_file` 不允许写业务源码、越权目录或绝对路径逃逸。`read_file` 和 `search_code` 需要限制文件大小和返回内容，避免把整个仓库或敏感文件塞给模型。

执行器安全测试需要验证 local_subprocess 的工作目录隔离、超时、命令参数拼接和 rootdir 策略。Docker 相关测试需要验证 Docker 不可用时能返回 preflight warning 或 executor unavailable，环境问题不得计入捕获率。LLM 配置错误响应必须脱敏，不回显密钥、base_url、本地路径或原始坏配置。

### 5.8 可靠性与异常测试

可靠性测试覆盖状态机、幂等、重试、取消和错误恢复。`test_runs.status` 与 `stage` 的组合必须符合生命周期设计，系统错误发生时先保存 run_events 和 trace_steps，再更新最终状态。retry 应创建新的 run 或新的受控 attempt，不能覆盖历史运行。

异常测试应覆盖非法模型输出、JSON 结构不合法、pytest collection error、pytest execution error、依赖安装失败、Runner timeout、artifact 写入失败、数据库冲突、重复 result callback、cleanup 越界路径和 replay cache stale 等场景。每个异常都应产生明确错误码、可读错误摘要和可追溯事件。

### 5.9 性能与容量测试

性能测试以本期交付规模为基准，不做大规模压测承诺。需要关注 API 查询响应、experiment metrics 聚合、replay scheduler 执行时间、artifact inventory 扫描时间、cleanup dry-run 时间和前端页面渲染体验。三策略 benchmark repeat=3 可作为基础容量样本，用于观察 clean runs、replay runs、variant replay records 和 metrics 聚合耗时。

后续若进入 V3 或大规模 replay，需要增加分页、projection cache、队列长度、Worker 并发、数据库索引和 artifact storage 容量测试。本期测试报告可记录当前数据量下的耗时和瓶颈，为后续容量规划提供依据。

### 5.10 兼容性与交付测试

兼容性测试覆盖 Windows 本地开发环境、Python 版本、Node.js 版本、PostgreSQL、Redis 和 Docker Desktop。Docker Compose 测试需要验证 postgres、redis、api、worker、frontend 五个服务可启动，前端可访问，API 可响应，worker 能执行实验任务。

交付测试包括基础 smoke、带 experiment 的 worker/eval smoke、三策略 benchmark 和真实 LLM option 读取。Docker 不可用、镜像缺失、daemon 未启动等环境问题应作为交付环境状态记录，不进入策略捕获率分母。



## 6. 测试环境

测试环境分为本地开发环境、自动化回归环境、Docker Compose 交付环境和真实 LLM smoke 环境。不同环境的目标不同，本地环境侧重快速调试，自动化回归侧重稳定通过，Compose 环境侧重交付复现，真实 LLM 环境侧重模型接入和输出稳定性。

| 环境 | 用途 | 基础组件 | 说明 |
| --- | --- | --- | --- |
| 本地开发环境 | 单元测试、接口测试、目标回归 | Python 3.10+、pytest、PostgreSQL、Redis、Node.js、npm | 适合开发阶段快速执行后端和前端检查 |
| 自动化回归环境 | 阶段验收、合并前回归 | Python、pytest、PostgreSQL、Redis、Node.js | 应固定依赖版本和数据库初始化脚本 |
| Docker Compose 环境 | 软件交付和演示验证 | PostgreSQL、Redis、Backend API、Worker、Frontend | 使用 `docker compose up --build` 启动 |
| 真实 LLM smoke 环境 | 模型配置和真实输出验证 | 后端服务、Worker、真实 LLM 配置 | 密钥从 `.env` 或 `llm.config.json` 注入，结果单独记录 |

环境变量和密钥文件不进入版本库。`.env`、`.env.local`、`backend/llm.config.json` 只在本地或交付环境保存。测试报告不得输出真实 API key、完整 base_url、敏感路径或未脱敏配置。



## 7. 测试数据计划

测试数据需要保证可复现、可隔离、可清理。后端单元和接口测试使用临时数据库和 fixture 数据，每个测试应尽量独立构造项目、快照、策略、运行配置和实验记录。文件系统测试使用临时目录，测试结束后清理生成文件。

评测数据以 demo benchmark suite 为主。数据集至少包含多个 eval task、seeded bug 和 bug variant，覆盖边界条件、比较符翻转、状态码错误、漏校验、返回值错误、布尔逻辑错误等缺陷类型。auto mutation 数据必须带 discovery evidence、patch evidence、probe metadata 和 probe_check 状态。

LLM 测试数据分为 MockLLM fixture 和真实模型响应。MockLLM fixture 要稳定、可重复，适合默认回归；真实模型响应用于 smoke 和人工分析，不作为默认回归的强依赖。真实 LLM 失败时，应记录 provider、模型名、错误分类、是否重试、是否进入 invalid_test_set 或 experiment failed。

前端演示数据只用于静态展示或 UI 开发，不作为真实 API 测试结果。Comparison 页面在 API 模式下只能消费 completed experiment 的真实 metrics，API 失败时应显示错误或空状态。

## 9. 自动化测试框架规划

后端自动化测试以 pytest 为核心。测试目录保持在 `backend/tests`，公共 fixture 放在 `backend/tests/conftest.py`，临时项目放在 `backend/tests/fixtures`。测试按照模块命名，例如 `test_run_pytest.py`、`test_contract_guard.py`、`test_stage4_experiment_runner.py`，便于按风险域选择性执行。

建议将测试分为四个执行集合。第一组是快速单元集合，覆盖工具、schema、Contract Guard、metrics 和 mutation discovery，适合每次提交前执行。第二组是接口与集成集合，覆盖 API、数据库、runtime profile、experiment runner 和 replay scheduler，适合功能完成后执行。第三组是端到端集合，覆盖 MockLLM run、真实 LLM smoke 和 Compose smoke，适合阶段收尾执行。第四组是 benchmark 集合，覆盖三策略 repeat 和 metrics 输出，适合最终验收前执行。

前端当前使用 `npm run build` 作为自动化门禁。后续若补充前端测试框架，建议采用 Vitest 覆盖 API client、状态派生函数和关键组件渲染，采用 Playwright 覆盖 Dataset -> Experiment -> Start -> Metrics / Evidence -> Report 主链路。Playwright 用例应固定 API fixture 或连接本地 Compose 环境，避免依赖 demo fallback。

交付自动化由 `scripts/compose_smoke.ps1` 和 `scripts/compose_benchmark.ps1` 承担。smoke 脚本验证基础服务和可选 experiment run；benchmark 脚本生成带 repeat 的三策略实验和 JSON 证据。测试报告生成工具可以读取 pytest 输出、前端 build 输出、compose 脚本输出和 benchmark JSON，汇总为统一的 Markdown 或 JSON 报告。



## 10. 测试执行计划

测试执行按阶段推进。V1 阶段完成基础表、工具、执行器和 MockLLM 端到端后，执行最小回归；V2.1 阶段增加评测数据模型和 experiment runner 后，执行评测专项测试；V2.2 阶段 A/B 合同收束后，执行执行平台与项目理解专项测试；V2.3 阶段增加 Dataset authoring 和 mutation operator 后，执行数据集与 mutation 专项测试；V2.4 阶段进入交付前，执行全量后端、前端构建、Compose smoke、benchmark 和 UI 彩排。

| 阶段 | 必跑测试 | 目标 |
| --- | --- | --- |
| V1 | 工具单元、API 基础、MockLLM E2E、最小 eval harness | 验证单机闭环 |
| V2.1 | eval harness、experiment runner、clean/replay、LLM runtime config | 验证评测有效性 |
| V2.2 | runtime profile、executor、source context、Contract Guard、Dataset 最小入口 | 验证 A/B 合同 |
| V2.3 | replay cache、artifact cleanup、runtime binding、Dataset readiness、mutation discovery | 验证可维护性和分母合同 |
| V2.4 | 后端全量、前端 build、Compose smoke、Compose benchmark、UI 彩排 | 验证最终交付 |

日常开发中，开发者应先执行受影响模块的目标测试，再根据改动风险决定是否执行更大范围回归。涉及数据库迁移、状态机、评测指标和安全边界的改动，必须执行相关专项测试和后端全量测试。



## 11. 准入、准出与暂停恢复标准

测试准入标准用于判断某个功能是否可以进入系统测试。功能代码需要完成基本自测，接口合同和 schema 已明确，数据库迁移可执行，必要测试数据可构造，运行环境和依赖可启动。涉及真实 LLM 的测试，还需要确认配置文件存在、密钥可用、模型调用预算已获批准。

测试准出标准用于判断阶段是否可以验收。P0 用例必须全部通过；P1 用例如有失败，需要确认不影响主链路并记录规避方案；后端全量测试、前端构建、关键 smoke 和 benchmark 必须有明确结果。所有阻塞缺陷需要修复或关闭，非阻塞缺陷需要写入测试报告的遗留问题。

当出现环境不可用、数据库迁移失败、关键依赖不可安装、真实 LLM 服务不可用、Docker daemon 不可用、测试数据损坏或主链路 P0 用例连续失败时，应暂停对应范围测试。恢复测试前需要完成环境修复、数据重建或缺陷修复，并重新执行受影响范围的回归。



## 12. 缺陷管理

缺陷按严重级别分为 P0、P1、P2、P3。P0 表示主链路不可用、数据损坏、错误指标污染、密钥泄露或执行边界失效；P1 表示核心功能异常但有临时规避；P2 表示局部功能、展示或边界问题；P3 表示文案、样式或低风险优化。

缺陷记录应包含模块、环境、版本、复现步骤、预期结果、实际结果、日志或截图、关联测试用例、严重级别、负责人、修复版本和回归结论。评测相关缺陷还需标明是否影响 capture rate、false positive、metric_status、invalid_test_set 或 `llm_calls=0` 口径。

缺陷关闭前必须完成回归验证。状态机、评测指标、Contract Guard、runtime profile、cleanup 和安全边界缺陷，除复现用例外，还应补充自动化测试防止回归。



## 13. 测试报告输出规范

测试报告应面向项目验收和后续维护，包含测试范围、环境、执行命令、测试结果、缺陷统计、质量评价、风险说明和遗留问题。报告可以用 Markdown 作为人工阅读版本，同时输出 JSON 作为机器可读版本。

建议报告结构如下表。

| 章节 | 内容 |
| --- | --- |
| 测试概述 | 测试版本、测试时间、测试负责人、测试范围 |
| 测试环境 | 操作系统、Python、Node.js、PostgreSQL、Redis、Docker、LLM 配置口径 |
| 执行摘要 | 后端通过数、跳过数、warning、前端构建结果、Compose smoke、benchmark 结果 |
| 功能测试结果 | 按模块列出通过、失败、阻塞和未测项 |
| 评测结果 | dataset、strategy、repeat、clean runs、replay runs、capture rate、false positive、llm_calls |
| 非功能测试结果 | 安全、可靠性、可复现性、交付、性能观察 |
| 缺陷统计 | 按严重级别、模块、状态统计缺陷 |
| 风险与遗留问题 | 未覆盖能力、环境限制、后续建议 |
| 附件 | pytest 输出、build 日志、benchmark JSON、UI 截图、运行报告链接 |

测试报告中的指标必须保留原始来源。例如 benchmark 指标应来自 experiment metrics 或脚本输出 JSON；前端结果应来自 `npm run build` 或 UI 彩排记录；后端结果应来自 pytest 输出。人工结论需要明确依据，不能用概括性描述替代可复查数据。



## 14. 测试交付物

测试交付物包括测试计划、测试用例库、自动化测试脚本、测试数据、测试执行日志、缺陷记录、测试报告和演示证据。自动化测试脚本随代码仓库维护，测试数据和 fixture 应尽量可重复生成，真实密钥和本地环境配置不得作为交付物提交。

本项目至少应保留以下测试交付物：`backend/tests` 后端测试集、`eval/harness` 评测脚本、`eval/demo` demo 数据集、`scripts/compose_smoke.ps1`、`scripts/compose_benchmark.ps1`、前端构建脚本、V2.4 演示 runbook、benchmark JSON 输出和 UI 彩排截图。最终测试报告应引用这些交付物，并说明其覆盖的测试范围。



## 15. 风险与应对

测试风险集中在真实模型不稳定、环境依赖复杂、异步链路难复现、评测口径容易被误读和前端缺少浏览器级自动化。对应措施是：默认回归使用 MockLLM，真实 LLM 只做 smoke；用 Compose 固定交付环境；异步任务在测试中使用 Celery eager；评测指标用 `metric_status` 和 EvaluationEvent 保留原因；前端在现阶段以构建和 runbook 彩排兜底，后续补 Playwright。

另一个风险是测试数据污染。TRACE 会生成测试文件、pytest 报告、artifact、workspace 和 replay cache，如果清理策略不明确，后续测试可能读取到旧结果。测试框架需要统一临时目录、数据库初始化和 cleanup 策略，所有测试报告都要标注使用的数据集、experiment id 或输出文件路径。



## 16. 结论

TRACE 的测试计划需要同时服务软件质量和评测可信度。常规测试保证 API、数据库、前端、执行器和交付栈稳定；评测专项测试保证生成测试的有效性结论可信；非功能测试保证路径边界、脱敏、幂等、可复现和环境限制有明确结果。按照本文档建立测试框架后，团队可以直接执行分层测试、收集自动化结果，并生成覆盖功能、工程质量和评测指标的项目测试报告。
