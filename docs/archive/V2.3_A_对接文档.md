# TRACE V2.3 A 线对接文档

更新日期：2026-06-21

## 1. 范围与结论

V2.3 A 线负责执行平台、runtime profile、replay 调度、cache/cleanup/artifact 生命周期和 metrics evidence 的收束。本轮已经把 V2.2 的最小可用执行链路推进到可对接的运行合同：

- Runtime profile 新增显式 `replay_policy`，用于声明 replay 并发、最大重试次数和退避秒数。
- Experiment 新增 `runtime_profile_bindings`，支持多 project dataset 按 project/task 显式绑定 runtime profile。
- Dataset 新增 runtime binding manifest，A/B 不再依赖前端扫描 project/snapshot 推断绑定关系。
- Replay retry、cache 查询/清理、artifact inventory、workspace cleanup 和 Runtime preflight 已接入 API 与前端。
- Metrics 继续保证 setup/executor/provider/artifact/flaky/invalid failure 不会被压成 0 捕获；replay 继续不调用 LLM，`llm_calls=0`。

## 2. 数据库与 Schema

新增迁移：

```text
backend/alembic/versions/20260621_0008_v23_runtime_bindings_replay_policy.py
```

新增字段：

- `runtime_profiles.replay_policy JSONB NOT NULL DEFAULT '{}'`
- `experiments.runtime_profile_bindings JSONB NOT NULL DEFAULT '{}'`

`replay_policy` 规范化后包含：

```json
{
  "replay_concurrency": 1,
  "max_retries": 0,
  "retry_backoff_seconds": 0
}
```

数值字段会在后端校验。非法数值会返回明确错误；preflight 草稿不会抛 500，而是返回 `status=failed` 和 `profile_contract` 检查失败。

## 3. Runtime Profile 对接

核心 API：

```text
GET  /api/v1/projects/{project_id}/runtime-profiles
POST /api/v1/projects/{project_id}/runtime-profiles
GET  /api/v1/eval-datasets/{dataset_id}/runtime-profiles
GET  /api/v1/eval-datasets/{dataset_id}/runtime-binding-manifest
POST /api/v1/eval-datasets/{dataset_id}/runtime-profiles
GET  /api/v1/runtime-profiles/{runtime_profile_id}
PATCH /api/v1/runtime-profiles/{runtime_profile_id}
POST /api/v1/runtime-profiles/{runtime_profile_id}/archive
POST /api/v1/runtime-profiles/preflight
POST /api/v1/runtime-profiles/{runtime_profile_id}/preflight
```

对接语义：

- `local_subprocess` 不是安全沙箱，不能声明 enforced resource limits，也不能使用非 `default` network policy。
- `docker` profile 必须提供 `image`。
- 单 project dataset 仍可通过 dataset-scoped API 创建 profile。
- 多 project dataset 的 `GET /eval-datasets/{id}/runtime-profiles` 会返回所有相关 project 的可用 profile，供前端做绑定；`POST /eval-datasets/{id}/runtime-profiles` 仍返回 409，避免在 project 不明确时误创建。
- `GET /eval-datasets/{id}/runtime-binding-manifest` 返回 task/project/profile 显式关系，是前端和 B 线 dataset authoring 对接的稳定入口。

## 4. Experiment Runtime Binding

创建 experiment 时可传：

```json
{
  "runtime_profile_id": "default-profile-id",
  "runtime_profile_bindings": {
    "project:project-a": "profile-a",
    "project:project-b": "profile-b",
    "task:task-1": "profile-task-1"
  }
}
```

绑定 key 支持：

- `project:<project_id>`
- `<project_id>`
- `task:<eval_task_id>`
- `<eval_task_id>`

多 project dataset 必须覆盖全部 project。profile 必须属于绑定 project，已归档 profile 不允许用于新 experiment。Runner 执行时按 `task` 优先、再按 `project`、最后按默认 `runtime_profile_id` 解析。

前端实验创建页已支持多 project dataset 的 project/profile 显式绑定；单 project dataset 保持原 runtime profile 下拉体验。

Runtime binding manifest 响应形状：

```json
{
  "dataset_id": "dataset-id",
  "multi_project": true,
  "tasks": [
    {
      "task_id": "task-a",
      "project_id": "project-a",
      "project_snapshot_id": "snapshot-a"
    }
  ],
  "projects": [
    {
      "project_id": "project-a",
      "project_name": "Project A",
      "profiles": []
    }
  ]
}
```

## 5. Replay、Retry 与 Cache

Replay retry 读取 frozen runtime snapshot 中的 `replay_policy`。当前可重试错误：

- `EXECUTOR_UNAVAILABLE`
- `EXECUTOR_TIMEOUT`
- `SETUP_FAILED`
- `PYTEST_REPLAY_ERROR`

每条 replay 的 `executor_metadata` 会记录：

- `retry_policy`
- `retry_attempts`
- `retry_count`

Replay cache API：

```text
GET  /api/v1/experiments/{experiment_id}/replay-cache
POST /api/v1/experiments/{experiment_id}/replay-cache/cleanup
```

cache cleanup 默认只清理 `hit` / `stale` 且位于 `.pytest_tmp_experiments` 下的安全 workspace。对于 cache hit 复用源 replay workspace 的记录，`workspace_manifest.reused_from_replay_id` 会使其跳过清理，避免通过 hit 行删除源 replay evidence。

## 6. Artifact 与 Cleanup

Artifact/lifecycle API：

```text
GET  /api/v1/experiments/{experiment_id}/artifact-inventory
POST /api/v1/experiments/{experiment_id}/cleanup
```

安全边界：

- cleanup 只允许操作 `.pytest_tmp_experiments` 子路径。
- 拒绝清理 experiment work root 本身。
- `keep_failed=true` 时不会删除失败 replay workspace。
- workspace cleanup 和 replay cache cleanup 都会跳过 `workspace_manifest.reused_from_replay_id`，避免通过 cache hit alias 删除源 replay evidence。
- artifact inventory 遇到历史脏数据中的越界 workspace 时不会整体失败，会返回 `safe=false`、`error` 和 `size_bytes=0` 的 workspace 项。
- cleanup 返回 summary，不改变 replay 捕获结果，也不重写历史 metrics。

前端 Experiment Detail 已展示 lifecycle summary、workspace cleanup 和 replay cache cleanup 操作入口。

## 7. Metrics 与 EvaluationEvent

`runtime_execution` 继续作为 metrics 顶层字段返回，并扩展：

- `replay_concurrency`
- `retries`
- `artifact_inventory`
- `replay_cache_counts`
- `setup_status_counts`

EvaluationEvent 扩展了 executor/cache/setup/cleanup 相关类型。它们默认是 audit evidence；是否进入 prompt-safe backfeed 仍由 `evaluation_event_policy` 单独控制。

## 8. 已修复风险点

- 修复 cache hit 行可能删除源 replay workspace 的风险。
- 修复普通 workspace cleanup 可能通过 cache hit alias 删除源 replay workspace 的风险。
- 修复 artifact inventory 遇到历史越界 workspace 时整体失败的问题，现在会标记 unsafe 并继续返回其余 inventory。
- 修复 replay cache 查询不存在 experiment 时静默返回空列表的问题，现在子资源语义保持 404。
- 修复 Runtime preflight 对非法 replay policy 数值不够稳定的问题。
- 修复多 project dataset runtime profile 列表直接 409 导致前端无法完成显式绑定的问题。
- 前端补齐多 project dataset 的 project/profile 绑定 UI，并切换到后端 runtime binding manifest，避免前端扫描 project/snapshot 推断关系。

## 9. 当前保留的大规模扩展项

以下项目不影响当前单机功能完整性，本轮先不展开到大规模实现，但后续进入 V2.3-M3/M4 时需要继续收束：

- 批量实验级 scheduler：当前 experiment runner 仍按 task/strategy/repeat 顺序执行，variant replay 支持单机并发；尚未提供跨 experiment 的统一队列、优先级、取消传播和失败隔离面板。
- Execution plan 一等化：当前 job plan 隐含在 runner 循环和 clean metrics 中；尚未落独立 DB 表或 API。
- Retry 非阻塞调度：当前 retry backoff 使用 worker 内同步等待，适合本地/小规模 smoke；大规模并发时应改为可调度 attempt 或队列延迟。
- Metrics projection cache：当前 metrics 仍即时聚合 replay、event 和 inventory；大实验需要分页、projection snapshot 和失效策略。
- 跨实验/跨数据集 replay cache 治理：当前 cache 查询和清理以 experiment 为主；后续可扩展 dataset/runtime profile/variant 过滤、分页和失效原因聚合。
- cleanup policy 自动执行：runtime profile 已保存 cleanup policy，但当前仍以手动 dry-run/delete 为主；`ttl_days`、`after_success`、`keep_evidence` 自动策略尚未启用。
- 分布式 replay / 跨机器执行：当前 executor 和 replay scheduler 是单机模型。
