# TRACE V2.3 B 线对接文档

## 1. 分工边界

B 线继续负责项目理解、数据集和生成约束：

- source context / retrieval trace
- Contract Guard / 生成与反思约束
- Dataset task / seeded bug / variant authoring
- mutation confirmation / mutation engine v1 的受控推进
- prompt-safe EvaluationEvent 回灌

A 线继续负责执行平台和评测尺子：executor、runtime profile、setup/replay/cache/cleanup、metrics 主体、runtime evidence 和 batch/replay scale。B 线只通过稳定合同消费 A 线输出，不解析 A 线内部 executor payload。

V2.3 B 线已完成 M1 Dataset Authoring 核心闭环，并已把 M2 Mutation Engine v1 收束到可验收最小 operator set。不开启 V3 远程 Runner，不改变 replay 或 captured bug 口径。

当前 B 线本地改动已按小步提交收束；下一步应进入 V2.3-M4 A/B 联合验收，重点核对 A 线 runtime/replay/metrics evidence 与 B 线 Dataset/mutation/Guard 合同是否一致。

## 2. 当前新增对接点

### 2.1 Dataset authoring 编辑闭环

本轮先补 Dataset M1 的最窄缺口：已有创建链路之外，增加 task / seeded bug / variant 的编辑合同。

新增后端 API：

- `PATCH /api/v1/eval-tasks/{task_id}`：更新 `target_scope`、`goal`、`expected_capabilities`。
- `PATCH /api/v1/seeded-bugs/{bug_id}`：更新 `bug_type`、`description`、`expected_detection`。
- `PATCH /api/v1/bug-variants/{variant_id}`：更新 `variant_name`、`mutated_snapshot_id`、普通 `ground_truth`。

variant 编辑刻意不开放 canonical patch 修改。patch 是评测分母和 artifact evidence 的一部分，改 patch 需要重新校验 patch hit、mutated snapshot、probe 和 artifact hash，不能混进 M1 元数据编辑。

普通 variant 更新会保留服务端生成的 `patch_artifact`、`patch_unique_hit` 和 `mutated_snapshot_verification` evidence。更新 `mutated_snapshot_id` 时，服务端会重新验证 mutated snapshot 是否等于 clean snapshot 应用 canonical patch 后的结果。

普通 variant API 仍拒绝 `ground_truth.source=auto_mutation`，auto mutation 只能通过 mutation confirmation 创建，避免绕过 probe confirmation 污染 sampled mutation score 分母。

### 2.2 Dataset UI 编辑入口

Dataset detail 页面在原有 authoring 面板上接入最小编辑体验：

- 可把当前选中 task 加载进表单并更新。
- seeded bug 卡片可加载到表单并更新。
- variant 行可加载到表单并更新 metadata / mutated snapshot / ground truth。
- variant 编辑模式下 patch 字段锁定，只展示为不可编辑状态。

UI 仍只展示已有能力，不把 ordinary variant API 包装成 auto mutation engine。

### 2.3 Dataset task 管理筛选

本轮继续补 Dataset M1 管理体验，在 Dataset detail 的 task 列表增加纯前端派生的 search / readiness filter / 批量状态展示：

- search 可按 task id、snapshot id、goal、target scope、capability、seeded bug 文本定位 task。
- readiness filter 只基于现有 `taskReadinessStatus()`，支持 all / ready / incomplete。
- task 列表标题显示 visible / total，批量状态条显示 ready、incomplete、hidden by filter。
- 该筛选不改变 Dataset API、不改变 readiness 语义、不影响 selected task 的真实数据。

这仍是结构 authoring 体验，不是 replay/metrics 结果展示。

### 2.4 Dataset readiness 后端检查合同

本轮把 Dataset readiness 从纯前端派生下沉为只读 API：

- `GET /api/v1/eval-datasets/{dataset_id}/readiness` 返回 `ready / incomplete`、task 计数、issue 计数和结构化 issue 列表。
- issue 覆盖 dataset 无 task、task 缺 target scope、task 无 seeded bug、seeded bug 无 variant、variant 缺 ground truth。
- 对 `ground_truth.source=auto_mutation` 的 variant，readiness 要求 `probe.probe_check.status=passed`；否则报告 `auto_mutation_probe_not_passed`。
- Dataset detail UI 在 API 数据源下优先展示服务端 readiness，demo 数据源继续使用本地派生结果。

该 API 是 authoring/checking 合同，不读取 replay 结果，不计算 capture，不改变 sampled mutation score 分母。

### 2.5 Mutation confirmation 状态可见性

本轮继续增强 Dataset variant 表格，把原本藏在 ground truth JSON 里的 confirmation/probe evidence 摘出来展示：

- 普通 seeded bug variant 显示为 ordinary，并说明不进入 sampled mutation score 分母。
- `ground_truth.source=auto_mutation` 且 `probe.probe_check.status=passed` 显示为 confirmed。
- auto mutation 缺 `probe_check` 时显示 probe missing，避免把不完整 evidence 看成 confirmed。
- 如果后续出现 failed probe evidence，UI 会显示 probe failed；若 `probe_check` 提供 `reason/message/error/detail`，优先展示失败原因，否则展示 clean / buggy 的 actual / expected 摘要。

该展示只消费已有 `ground_truth` evidence，不新增后端字段，不改变 sampled mutation score 分母口径。Dataset readiness API 对 auto mutation probe 未通过的 issue 也会透出已有失败原因，便于 authoring 阶段定位。

### 2.6 真实 API Contract Guard 重试反馈

本轮处理 `V2.3_B_真实API契约拒绝问题移交.md` 中的真实 API 契约拒绝问题，重点不放松 Guard，而是让一次 structured retry 更可修：

- `build_generate_retry_messages()` 会把 Guard violation 原文和针对性修复动作一起回灌给模型。
- 高频 violation 会被翻译为操作说明：cases 元数据对齐、fixture 证据缺失、request field 证据缺失、弱 route oracle、method/path 不匹配、目标绑定漂移、异常 oracle 缺源码证据、源码行为矛盾等。
- retry prompt 新增输出不变量：必须返回完整 `GenerationOutput` JSON，`test_file_content` 必须是完整 pytest 文件，`cases` 必须和真实测试函数一一对应，不输出 `test_file_path: null`，不切换目标、不删除有证据断言、不新增无证据 fixture/request field/oracle。
- `prompt_bundle()` 冻结 retry 输出不变量，保证历史 run 的 strategy snapshot 可审计。

该改动只提高真实 LLM 修正成功率，不把 Contract Guard 改成 warning，不改变 `PIPELINE_REJECT` 的 invalid_test_set 口径。

### 2.7 sampled mutation score 分母合同

本轮开始收束 V2.3-M2 的最小风险口径，但不扩 mutation operator：只补 sampled mutation score 的分母合同测试。

- 分母只允许 `ground_truth.source=auto_mutation` 且 `ground_truth.probe.probe_check.status=passed` 的 variant。
- `probe_check.status=failed` 和缺失 `probe_check` 的 auto mutation variant 必须进入 `excluded_variant_counts.probe_check_not_passed`，不能污染分母。
- 普通 seeded bug variant 继续进入 `excluded_variant_counts.non_auto_mutation`。
- 该测试不跑 executor，不改变 replay/metrics 主体，只锁定 B 线 mutation confirmation evidence 对 A 线 metrics 的输入边界。

### 2.8 Mutation Engine v1 第一刀：comparison_negation

本轮开始扩展 operator，但只加最窄、可确认的一类：`comparison_negation`。

- discovery 支持 `==` -> `!=`、`!=` -> `==`，仍只处理单操作符 compare，不碰 chained comparison。
- candidate 继续携带 `source_location_hash` matcher、patch evidence、operator、target symbol 和 deterministic selection reason。
- confirm-selected 仍服务端重算 discovery；测试覆盖 `comparison_negation` candidate 经 probe check 创建 confirmed auto mutation variant。
- 该切片不改变 sampled mutation score 口径；只有 confirmed + probe passed variant 才能进入分母。

### 2.9 Mutation Engine v1 第二刀：arithmetic_operator

本轮继续只扩一个可确认 operator：`arithmetic_operator`。

- discovery 支持 `+` -> `-`、`-` -> `+`，仍要求 source segment 唯一。
- candidate 继续携带 `source_location_hash` matcher、patch evidence、operator、target symbol 和 deterministic selection reason。
- confirm-selected 仍服务端重算 discovery；测试覆盖 `arithmetic_operator` candidate 经 probe check 创建 confirmed auto mutation variant。
- 该切片不改变 sampled mutation score 口径；只有 confirmed + probe passed variant 才能进入分母。

### 2.10 Mutation Engine v1 第三刀：boolean_negation

本轮继续按小步 operator 推进，只加最窄可确认的布尔取反：`boolean_negation`。

- discovery 仅支持有布尔意图证据的裸返回 `return flag` -> `return not flag`；布尔意图来自 `bool` 返回注解或谓词式函数名前缀，暂不处理复杂表达式、已有 `not`、条件分支取反或布尔常量。
- patch old text 使用整条 `return flag`，避免只替换变量名导致参数或其他引用被误改。
- candidate 继续携带 `source_location_hash` matcher、patch evidence、operator、target symbol 和 deterministic selection reason。
- confirm-selected 仍服务端重算 discovery；测试覆盖 `boolean_negation` candidate 经 probe check 创建 confirmed auto mutation variant。
- 该切片不改变 sampled mutation score 口径；只有 confirmed + probe passed variant 才能进入分母。

### 2.11 Mutation Engine v1 第四刀：constant_replacement

本轮继续只加可 probe 的最窄常量替换：`constant_replacement`。

- discovery 仅支持布尔常量返回 `return True` <-> `return False`，暂不处理数字、字符串、None 或表达式内常量。
- patch old text 使用整条 return 语句，仍要求 source segment 唯一。
- candidate 继续携带 `source_location_hash` matcher、patch evidence、operator、target symbol 和 deterministic selection reason。
- confirm-selected 仍服务端重算 discovery；测试覆盖 `constant_replacement` candidate 经 probe check 创建 confirmed auto mutation variant。
- 该切片不改变 sampled mutation score 口径；只有 confirmed + probe passed variant 才能进入分母。

### 2.12 Mutation Engine v1 第五刀：return_value

本轮继续只加返回值替换的最窄可验证切片：`return_value`。

- discovery 仅支持 `return None` -> `return False`，暂不处理其他返回值、表达式或隐式返回。
- probe confirmation 明确支持 `clean_value: null`，字段存在但值为 null 不能被当成缺失 metadata。
- patch old text 使用整条 return 语句，仍要求 source segment 唯一。
- candidate 继续携带 `source_location_hash` matcher、patch evidence、operator、target symbol 和 deterministic selection reason。
- confirm-selected 仍服务端重算 discovery；测试覆盖 `return_value` candidate 经 probe check 创建 confirmed auto mutation variant。
- 该切片不改变 sampled mutation score 口径；只有 confirmed + probe passed variant 才能进入分母。

### 2.13 B 线收束结论

本轮不继续扩 `exception_type` 或 `statement_deletion`：

- `exception_type` 需要 probe schema 先支持异常期望，否则无法通过 confirm-selected 证明 clean/buggy 行为差异。
- `statement_deletion` 容易破坏语法或制造无意义 mutant，必须先有更强 AST patch / syntax check 合同。
- 当前 operator set 已足以覆盖 V2.3 M2 的最小可验收目标：每类都带 discovery evidence、deterministic selection、server-side rediscovery 和 probe confirmation。

B 线 M1/M2 当前状态：可收束，待 V2.3-M4 A/B 联合验收。

## 3. A/B 接口约定

本轮不改变 A 线合同：

- replay 仍只执行冻结测试集，不调用 LLM。
- invalid_test_set、provider_failure、artifact_hash_mismatch、setup failure、executor failure、flaky clean replay 不能解释成 0 捕获。
- Dataset readiness 是结构检查，不是 replay 结果，也不是 captured bug 指标。
- mutation score 分母仍只能统计 confirmed + probe passed 的 auto mutation variant。
- Contract Guard 仍是系统边界，不是 prompt 装饰。
- prompt 仍只吃 prompt-safe evidence；audit-only event 默认只进 report/audit。

## 5. 后续 B 线候选

V2.3-M4 联合验收优先：

- 真实 LLM smoke 可继续沉到录制 fixture，按 Direct / Plan / ReAct 各覆盖一条 Contract Guard retry 成功或明确拒绝链路。
- Dataset readiness / confirmation smoke 可在联合验收中固定为 runbook 步骤。
- A/B 合同核对：runtime/replay failure 不压成 0 捕获，B 线 confirmed + probe passed 分母继续被 A 线 metrics 正确消费。

V2.3-M2 B 线当前可收束；已覆盖 comparison_boundary、comparison_negation、arithmetic_operator、boolean_negation、constant_replacement、return_value。后续如扩 exception_type / statement_deletion，必须先补 probe 或 AST patch 合同，不能作为当前验收阻塞项。
