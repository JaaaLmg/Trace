## 2. V2.2 已交付能力

### A 线：评测尺子 / 执行平台

- Runtime profile 已升级为执行合同，支持 executor、image、working dir、install/test command、timeout、network/resource/artifact/cleanup policy，并冻结到 clean run 与 replay evidence。
- Executor 抽象支持 `local_subprocess` 与 `docker`。
- `local_subprocess` 明确是本地进程执行，不是安全沙箱，不能宣称 enforce 网络或资源隔离。
- Docker executor MVP 已落地：检查 Docker CLI/daemon/image，构造受控 workspace mount，收集 pytest JSON/JUnit/stdout/stderr/container metadata。
- setup/install stage 是 first-class execution step；setup failure 进入 `setup_failed` EvaluationEvent，不算 0 捕获。
- Replay 不调用 LLM，只执行冻结测试集；replay evidence 包含 runtime snapshot、executor metadata、workspace manifest、artifact hash、cache key/status/source。
- Incremental replay cache 已落地，cache hit/stale/miss 可审计，失败 job 不会写成捕获结果。
- Replay scheduler 支持单机并发 variant replay；clean replay 与 flaky gate 仍先于同 clean unit 的 variant replay。
- Metrics / EvaluationEvent / 前端 replay panel 已展示 executor、runtime、setup、cache、cleanup、parallel replay evidence。
- Runtime profile API 与前端管理页支持 list/create/update/archive；实验创建页可选择 runtime profile 并展示 executor/policy summary。
- Runtime 页面已支持中英文切换；API 枚举值保留原文。

### B 线：项目理解 / 数据集 / 生成约束

- Source context 会把目标函数引用到的同文件模块级常量作为 dependency evidence。
- retrieval trace 记录 resolved / missing / truncated / failed，不静默吞证据。
- `source_context.py` 已机械拆出 failure context、trace helper、dependency slicing、support helper、route helper、target query helper。
- Contract Guard 会消费静态常量证据，可拒绝无源码依据的 route path 参数值。
- Contract Guard 已约束 route method/path、response model 字段、request field evidence、fixture evidence、弱 oracle、目标漂移和目标重定义。
- route request field guard 覆盖 `json=`、`params=`、`data=`，并限制在 HTTP verb 调用上，避免误伤普通 helper 的同名关键字。
- EvaluationEvent prompt-safe policy 已抽到 `evaluation_event_policy.py`。
- `source_context_failures.py` 低层过滤 audit-only / non-consumable EvaluationEvent；`provider_failure`、`artifact_hash_mismatch` 不进 prompt。
- Dataset UI 已接入 task / seeded bug / ordinary variant / mutation discovery / confirm-selected 最小链路。
- Dataset UI 展示结构 readiness 和 mutation discovery exclusion reason/target/message；readiness 不是 replay 结果，也不解释成捕获率。
- mutation confirmation 不信任客户端回传 audit candidate，服务端会重算 discovery 并逐字段比对 selected candidate。
- mutation confirmation 只允许 `bug_type=auto_mutation`；普通 variant API 拒绝 `ground_truth.source=auto_mutation` 绕过 probe confirmation。
- auto mutation v0 是最小 confirmation 链路，不是完整 mutation engine。

## 4. V2.3 规划与分工

V2.3 目标是从“V2.2 已验收”推进到“可持续扩展”。主线不是重新证明链路能跑，而是把 Dataset、mutation 和执行规模化三块从最小可用推进到稳定产品能力。

### 4.1 总目标

- Dataset 从最小 authoring 推进到可管理、可审计、可复用。
- auto mutation 从 v0 confirmation 链路推进到受控 mutation engine v1。
- 执行平台从单机完整链路推进到更可靠、更可观测、更适合批量实验。
- 保持 V2.2 硬口径不退化：invalid/flaky/provider/setup/artifact/runtime failure 仍不能算 0 捕获。

### 4.2 A 线责任

A 线继续负责评测尺子、执行平台和规模化运行：

- replay 调度增强：批量实验、并发上限、失败重试、资源隔离证据。
- Docker / local executor 兼容性增强。
- 多 project dataset 的 runtime profile 显式选择。
- metrics / report 的稳定性和性能优化。
- cleanup、cache、artifact 生命周期治理。
- 保证 replay 不调用 LLM，failure 不压成 0 捕获。

### 4.3 B 线责任

B 线继续负责项目理解、数据集和生成约束：

- Dataset UI 完整化：task、seeded bug、variant、mutation confirmation、readiness、filter、search。
- Mutation engine v1：扩展 operator，但必须保留 selected candidate、probe confirmation 和 server-side rediscovery。
- Source context 剩余拆分和 retrieval provider glue 收束。
- Contract Guard 继续围绕源码证据补窄规则。
- Prompt-safe evidence 回灌继续收紧，audit-only 默认不进 prompt。
- Prompt / Tool schema 管理可归 B 线，但必须和 A 线 strategy snapshot 合同对齐。

### 4.3.1 B 线补充：真实 API Contract Guard retry 稳定化

该问题属于 V2.3 B 线计划内的生成约束稳定化，不开新里程碑，不放松 Contract Guard，也不进入 V3 范围。

背景：真实 LLM/API 场景中，生成测试常被 Contract Guard 拒绝；Guard 拒绝本身正确，但一次 structured retry 需要更可操作，避免模型重复同类错误。

已完成第一步：

- retry prompt 回灌 Guard violation 原文和针对性修复动作。
- 高频 violation 已覆盖 cases 元数据对齐、fixture 证据缺失、request field 证据缺失、弱 route oracle、method/path 不匹配、目标绑定漂移、异常 oracle 缺源码证据、源码行为矛盾等。
- retry 输出不变量已冻结到 strategy snapshot：完整 `GenerationOutput` JSON、完整 pytest 文件、`cases` 与真实测试函数一一对应、不输出 `test_file_path: null`、不切换目标或新增无证据内容。
- `PIPELINE_REJECT` 仍进入 `invalid_test_set / pipeline_reject`，不能解释成 0 捕获。

后续验收候选：

- 增加真实 LLM smoke 或录制 fixture，按 Direct / Plan / ReAct 各覆盖一条 retry 成功或明确拒绝链路。
- 按真实高频 violation 聚类继续补 source context / prompt / Guard 合同测试。
### 4.3.2 B 线补充：Mutation Engine v1 第一刀

M2 已开始最小 operator 扩展，但不改变 auto mutation 的对外口径：这仍是受控 confirmation 链路，不是完整 mutation testing 平台。

已完成第一刀：

- `comparison_negation` operator：支持 `==` -> `!=`、`!=` -> `==`。
- discovery candidate 携带 operator、patch、source location hash、target symbol 和 deterministic selection reason。
- confirm-selected 继续服务端重算 discovery，不信任客户端 candidate。
- 新增 probe confirmation 测试，确认 `comparison_negation` candidate 能创建 confirmed auto mutation variant。
- sampled mutation score 分母合同保持不变：只统计 confirmed + probe passed variant。

后续 operator 扩展要求不变：每个 operator 都必须同步 discovery evidence、selection reason、probe check 和 denominator 测试。

### 4.4 共享接口

- schema / API contract / migration 必须共同评审。
- 执行证据展示归 A；Dataset / Mutation / Guard 状态归 B；跨页面导航和统一设计规范共享。
- 任何影响 metrics 解释的字段，先写合同测试，再接 UI。
- 任何跨线字段必须明确是 evidence、audit、prompt-safe 还是 metric denominator。

### 4.5 里程碑

| 里程碑 | 目标 | 主责 | 验收 |
| --- | --- | --- | --- |
| V2.3-M1 Dataset Authoring 闭环 | UI 完整创建和检查 task / seeded bug / variant；mutation confirmation 状态和失败原因可见 | B 线 | Dataset API 测试、前端 build、readiness/confirmation smoke |
| V2.3-M2 Mutation Engine v1 | 扩展 operator；每个 operator 有 discovery evidence、selection reason、probe check | B 线 | mutation discovery/confirm 合同测试、sampled mutation score 分母测试 |
| V2.3-M3 Execution Scale & Runtime UX | batch/replay/cache/runtime profile 体验增强；多 project dataset 不再含糊 | A 线 | stage runner、runtime profile、replay cache、metrics 全量回归 |
| V2.3-M4 整合验收 | A/B 合同收束，形成 V2.3 验收结论 | A/B 共同 | 后端全量、前端 build、diff check、验收报告或清单更新 |

### 4.6 暂不做

- 不把 auto mutation v1 宣传成完整 mutation engine，直到 operator 覆盖、probe、score 分母都稳定。
- 不把 local_subprocess 包装成安全沙箱。
- 不改变 captured bug、invalid_test_set、false positive、sampled mutation score 的解释口径。
- 不把 audit-only event 默认塞回 prompt。

## 5. V2.4 最终验收版规划

V2.4 的目标不是开启 V3 平台化，而是把 V2.3 的可持续能力收束成最终实训/展示验收版。判断标准不是“功能更多”，而是一次完整 benchmark 能被非开发者看懂、能复跑、能解释为什么可信。

### 5.1 总目标

- 固化一套可展示的 Dataset benchmark：至少覆盖 demo seeded bug 数据集，并优先补 1-2 个小型真实项目或扩展 demo project。
- 完成 Direct / Plan / ReAct 三策略的最终对比跑法，真实 LLM 与 MockLLM 分开解释。
- Docker runtime 至少有稳定 smoke 和一条可演示链路；local subprocess 继续只作为本地执行方式，不宣传为沙箱。
- 前端支撑最终演示：实验对比、replay 证据、Dataset 状态、mutation confirmation、runtime evidence、report/export 路径都能讲清楚。
- 文档收束：最终验收报告、演示 runbook、已知边界、V3 候选不混在一起。

### 5.2 必须交付

- Dataset 最终包：task、seeded bug、variant、probe 证据、readiness 状态和筛选视图稳定。
- Mutation v1 收束：有一组明确 operator，但只统计 confirmed + probe passed variant。
- Prompt / Tool schema 最小管理能力：至少能记录版本、查看内容/hash，并和 strategy snapshot 对齐。
- Final benchmark report：包含 capture、false positive、invalid、flaky、provider/setup/artifact/runtime failure 的解释。
- 前端最终演示路径：从 Dataset 到 Experiment，再到 Metrics / Evidence / Report，不需要手工查 DB 才能讲清楚。

### 5.3 验收门槛

| 验收项 | 标准 |

### 5.4 V2.4 不做

- 不做 V3 远程 Runner 协议、Runner 注册、heartbeat、long-poll 领任务。
- 不做分布式调度、MinIO/S3 artifact storage 或完整 OpenTelemetry。
- 不把 auto mutation 包装成完整 mutation testing 平台。
- 不引入新的大策略线；策略对比以能稳定复现和解释为主。

## 6. A/B 分工边界

V2.2/V2.3 分工按工程边界记录，不按 Direct / Plan / ReAct 策略分工。策略是共同评测对象，不是代码所有权边界。

| 分工 | 责任边界 | 稳定输出 | 红线 |
| --- | --- | --- | --- |
| A 线：评测尺子 / 执行平台 | executor、runtime profile、experiment runner、replay、metrics、DB/API evidence | `EvaluationEvent`、experiment metrics、replay artifact、dataset/variant 状态机、runtime evidence | provider failure、flaky、artifact mismatch、invalid test set、setup/executor failure 不能压成 0 捕获；执行边界不能宣传成未实现的安全沙箱 |
| B 线：项目理解 / 数据集 / 生成约束 | source context、retrieval trace、Contract Guard、Dataset UI、mutation confirmation、prompt-safe 回灌 | generated test metadata、target mapping、guard violation、source context trace、dataset/variant 编辑证据 | 证据不足时必须拒绝、降级或标记 incomplete；不能靠 prompt 自律替代 guard；auto mutation v0 不能宣传为完整 mutation engine |

协作接口：

- A 输出稳定的 `EvaluationEvent`、experiment metrics、replay artifact、runtime evidence 和 dataset/variant 状态机。
- B 消费 prompt-safe evidence，输出可 replay 的 generated test metadata、target mapping、guard violation 和 source context trace。
- A 不依赖 B 的 prompt 内部文本；B 不依赖 A 的内部 executor/metrics payload，只依赖稳定事件和 schema。
- 共同维护 schema / migration / 前端展示口径；任何跨线字段先写合同测试，再接 UI 或 runner。

## 7. 关键口径

- clean run 失败、provider failure、executor failure、setup failure、artifact hash mismatch、flaky clean replay 是系统/评测有效性问题，不是 0 捕获。
- replay 不调用 LLM；`llm_calls` 必须为 0。
- mutation score 只统计可评价测试集。
- false positive 是 clean 代码上测试失败，必须和 variant 捕获分开。
- real LLM 与 MockLLM 结果分开解释。
- prompt 输入只吃 prompt-safe evidence；审计事件不能默认塞回 LLM。
- source context 缺目标源码、缺模型字段、缺 fixture 证据时，必须拒绝、降级或标记 incomplete。
- Contract Guard 是系统边界，不是提示词装饰。
- `backend/llm.config.json` 和 `.env*` 永远不入库。
