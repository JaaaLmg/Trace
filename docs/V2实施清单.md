# TRACE V2 实施清单

## 0. 文档用途

这份文档回答一个问题：**真实 LLM 下，TRACE 的策略评测是否可复现、可审计、可对比。**

V1 证明的是单机闭环：MockLLM、三策略、trace、report、脚本评测。V2 要把它升级为评测系统：

```text
真实 LLM run -> 冻结 prompt/tool/runtime -> clean 代码生成最终测试集
-> bug 变体只 replay -> 指标落库 -> 前端展示实验结果 -> 可从 DB/artifact 重算审计
```

本文现在是 V2 的唯一状态源。已经完成的历史细节收进“实施历史”，仍然有约束力的规则收进“核心口径”。不要再在多个章节为同一件事重复打勾。

## 2. V2 边界

### 2.1 V2.1 必做

V2.1 是“真实 LLM + 评测落库”。它必须证明：

- 后端 API/worker 支持真实 LLM provider。
- Prompt、Tool Schema、Runtime Profile 正式版本化。
- 评测数据集、任务、seeded bug、bug variant、experiment、clean run、replay run 落库。
- `eval/harness/run_eval.py` 只是 CLI 包装或兼容入口；核心 clean/replay/metrics 走 service/API/DB。
- 后端提供 experiment API 和 DB 聚合 metrics。
- 前端真实 API 模式展示 experiment metrics，mock 只作为显式 fallback。
- 单次 run report 能说明测试质量，不只是运行统计。
- Agent 生成上下文能按目标读取必要源码片段，并在缺失时写风险。

### 2.2 V2.2 可做

V2.2 是加固项，不阻塞 V2.1：

- 本地 Docker executor。
- 每次 run 独立容器与独立工作目录。
- 默认禁网、限制 CPU/内存/进程数、只挂载必要目录。
- 仍然是本机执行，不做远程 Runner 注册、心跳、long-poll。

### 2.3 明确不做

这些留给 V3 或更后面：

- 远程/自托管 Runner API：`register`、heartbeat、next-job long-poll、runner job lease、presigned artifact upload。
- 多租户、权限系统、组织空间。
- GitHub App、自动拉 PR、完整 CI 集成。
- OpenTelemetry 全链路追踪。
- WebSocket 实时 trace 推送。
- Prompt 在线编辑器大后台。
- 覆盖率作为核心指标。

V2 可以为这些功能留字段和扩展点，但不能提前实现半套。

## 4. 核心口径

### 4.1 Experiment 与 LLM 配置

Experiment 是评测执行单元。strategy 列表走 `experiment_strategy_versions` 多对多表，不塞数组字段当真相。

`llm_override` 是实验级覆盖层：

```text
strategy_versions 默认模型参数
-> experiments.llm_override 覆盖层
-> repeat 派生参数
-> resolved spec 冻结进 test_runs.strategy_snapshot
```

规则：

- API 请求不传 API key。
- key 只从环境变量或 secret manager 读。
- `llm_override` 可以入库，但不能包含 secret。
- 最终 provider/model/temperature 只在每个 clean run 创建时解析一次。
- 实际调用 LLM 的参数和 `strategy_snapshot` 必须来自同一份 resolved spec。
- provider 为 `mock` 才走 MockLLM；真实 provider 走 provider factory。
- clean run 如果 provider/LLM 调用失败，experiment 必须 failed，不能伪装成 completed + 0 捕获。

### 4.2 Clean run

Clean run 负责生成和定型测试集。

流程：

1. 从 eval task 合成 `test_plan.target_scope`。
2. 创建 `test_run`，绑定 clean `project_snapshot_id`、runtime profile、strategy version。
3. 解析并冻结 runtime/strategy/LLM snapshot。
4. Agent 基于 AST 摘要 + 目标源码片段生成测试。
5. 如果允许 Reflection，只允许在 clean snapshot 上修一次。
6. 选择 final attempt。
7. 从 final attempt 生成最终测试集 artifact。
8. 在 clean snapshot 上 replay 冻结测试集，判定 clean false positive。

规则：

- clean failed / collection error / clean replay assertion failure 都属于 false positive 或系统失败语义，不属于 captured bug。
- clean run 失败时不创建有效 experiment evidence。
- 生成测试目录必须受 path policy 限制；Agent 不允许写业务源码。

### 4.3 Replay run

Replay run 只执行冻结测试集。

规则：

- replay 不规划、不生成、不调用 LLM。
- replay 可以跑 clean snapshot，也可以跑 bug variant snapshot。
- bug variant replay 前从 artifact 读取测试集，并校验 content hash。
- 如果 artifact hash mismatch，experiment failed。
- variant patch 必须从 clean snapshot 唯一应用；命中 0 次或多次都失败。
- replay 的 pytest 结果必须能映射到原始 nodeid。

### 4.4 captured bug

默认 V2.1 口径：

```text
clean replay passed nodeid
AND same nodeid has assertion failure on variant replay
=> captured_bug = true
```

只看 `experiment_replay_runs.captured_bug` 聚合捕获率。不要从前端二次解释 captured bug。

### 4.5 指标

V2.1 必算指标：

- `seeded_bug_capture_rate`
- `clean_false_positive_rate`
- `avg_tokens`
- `avg_tool_calls`
- `avg_duration_ms`
- `pytest_collection_success_rate`
- `cost_per_captured_bug`
- `reflection_contract_pass_rate`
- `reflection_acceptance_rate`

成本规则：

- 有捕获：`cost_per_captured_bug = total_tokens / total_captured`
- 0 捕获：`cost_per_captured_bug = null`，`cost_per_captured_bug_status = no_bug_captured`

### 4.6 Report quality

普通 run report 必须包含：

- 测试清单：生成了哪些测试，nodeid 是什么。
- 目标映射：测试对应哪些函数/路由/模块。
- 断言摘要：每个测试验证什么行为。
- 失败分类：assertion、fixture、import、collection、timeout 等。
- Reflection 证据：是否使用、是否通过契约、采纳了哪次 attempt。
- 上下文完整性：是否读取目标源码片段，缺失时写 `context_incomplete` 风险。

没有人工评分时，不展示 `meaningful_assertion_rate` 为已完成指标。

### 4.7 Source context

Agent 生成上下文规则：

- 先跑 `analyze_project` 得到 AST 摘要。
- 再按目标函数/路由读取源码片段。
- 读取范围受项目根、allowlist、denylist、文件大小、总预算、snippet 数量限制。
- 路径越界、找不到目标、预算截断时写 `context_incomplete` 或 partial 风险。
- 不用无边界多轮探索弥补上下文质量。

## 5. 数据模型摘要

V2 在 V1 14 张表基础上扩展，不推倒重来。

V1 基础表：

```text
projects, project_snapshots, strategy_versions, test_plans, test_runs,
run_plan_items, run_attempts, generated_test_files, generated_test_cases,
pytest_case_results, trace_steps, run_events, run_artifacts, test_reports
```

V2 新表：

| 表 | 用途 | 关键约束 |
| --- | --- | --- |
| `runtime_profiles` | 可复用执行环境配置 | run 创建时展开进 `test_runs.runtime_snapshot` |
| `strategies` | 策略逻辑父表 | Direct / Plan-and-Execute / ReAct 等父级语义 |
| `prompt_versions` | 不可变 prompt 正文版本 | 内容变化创建新 version，记录 content hash |
| `tool_schema_versions` | 不可变工具 schema 版本 | schema 来自 Pydantic 或显式 JSON schema |
| `eval_datasets` | 评测数据集版本 | dataset version 不可变 |
| `eval_tasks` | 评测任务 | `target_scope` 是捕获率分母来源 |
| `seeded_bugs` | 缺陷语义定义 | 不直接保存 patch 真相 |
| `bug_variants` | 具体 bug 变体 | V2 默认 `canonical_kind=patch`，patch 必须唯一命中 |
| `experiments` | 一次实验定义 | 失败是系统失败，不是策略捕获低 |
| `experiment_strategy_versions` | experiment 与 strategy version 多对多 | `(experiment_id, strategy_version_id)` 唯一 |
| `experiment_clean_runs` | clean 生成与定型记录 | false positive 只在 clean 层判断 |
| `test_replays` | 冻结测试集 replay 记录 | `llm_calls=0` |
| `experiment_replay_runs` | bug variant 捕获结果 | 捕获率只看这里 |

关键迁移规则：

- `project_snapshots` 补 `source_ref`、`commit_sha`、`dependency_summary`、`entrypoint_guess`、`analysis_summary`。
- `strategy_versions` 补 `strategy_id`、`version`、`prompt_version_id`、`tool_schema_version_id`、`temperature`、`is_locked`。
- `test_runs` 新 run 必须绑定 runtime profile 或冻结完整 runtime snapshot。
- `run_artifacts` metadata 必须包含 kind、created_by、experiment_id、clean_run_id/replay_id、content_role、relative_paths、hash_algorithm。

暂不建：

| 表 | 原因 |
| --- | --- |
| `runners` | V3 远程 Runner 才需要 |
| `runner_jobs` | V3 job lease 才需要 |
| `experiment_artifacts` | V2 先用 `run_artifacts` + metadata |
| `human_scores` | 可先写回 `generated_test_cases.human_meaningfulness_score` |

## 7. Runtime、Artifact 与前端

### 7.1 Runtime

V2.1 继续使用 subprocess executor，但必须诚实：

- pytest 命令来自 runtime snapshot。
- timeout 可强制。
- CPU、内存、进程数限制在 V2.1 是声明性记录，不是强制隔离。
- 不允许把 V2.1 subprocess 说成安全沙箱。
- Docker executor 后置到 V2.2。

### 7.2 Artifact

Artifact 是复现和审计的一部分，不是日志垃圾桶。

规则：

- generated test set artifact 写入后计算 sha256。
- DB 保存 URI、content hash、size、metadata。
- replay 前读取 artifact 并校验 hash。
- hash 不匹配时 experiment failed。
- artifact metadata 至少包含：
  - `kind`
  - `created_by`
  - `experiment_id`
  - `clean_run_id`
  - `replay_id`
  - `content_role`
  - `relative_paths`
  - `hash_algorithm`

阶段 4 对齐的指标细节：

- 捕获率按 repeats 取均值和总体标准差。
- 0 捕获时 `cost_per_captured_bug` 为 null / `no_bug_captured`。
- 参数化 pytest case 按完整 nodeid 判断。
- Mock/scripted DB 聚合结果与 V1 harness `metrics.py` 同口径一致。

## 9. 风险与后续处理

### 9.1 已化解风险

- MockLLM 自动化测试保留。
- experiment 支持 repeats。
- 指标展示均值和标准差。
- 指标聚合有单元测试和端到端测试。
- replay 逻辑和报告逻辑不再各算一套核心口径。
- artifact 写入后计算 hash，DB 保存 hash 和 URI。
- replay 前校验 generated test set artifact hash。
- V2.1 不依赖 Docker。
- Docker executor 明确后置到 V2.2。
- Experiment Detail 是前端主屏。
- Prompt/runtime 管理页面后置。
- clean/replay/metrics 已落到 service/API/DB。
- report quality 已展示测试质量证据。
- source context 已按目标读取源码片段，并标记 `context_incomplete`。

### 9.2 剩余风险

- 真实 LLM seeded bug experiment 需要有效 API key 后重跑。
- structured retry 还没作为 V2.1 必选闭环。
- final attempt helper、InMemoryRecorder、DB recorder 的 records 协议还可以进一步统一，减少未来口径漂移。
- Docker executor 未开始；V2.1 subprocess 不能声称是安全沙箱。
- 多 dataset / 自动 mutation / 更大规模 replay 尚未做。

## 11. 结论

V2 的价值不在“多几张表”。

它的价值在于：别人问“你说某策略更好，证据在哪”时，TRACE 能从 experiment metrics 追到 strategy、repeat、clean run、replay、seeded bug、nodeid、artifact、trace 和 pytest case result。

别人问“这份测试质量如何”时，TRACE 能说清楚测试目标、断言、失败原因、Reflection 证据和上下文来源。

这才是 V2 要打穿的东西。其他都先靠边。
