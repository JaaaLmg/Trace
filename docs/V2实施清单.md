# TRACE V2 实施清单

## 0. 这份文档是什么

`V1实施清单.md` 解决的是“单机闭环能不能跑通”：MockLLM、三策略、trace、report、脚本评测。

这份文档解决 V2 要回答的问题：**真实 LLM 下，TRACE 的策略评测是否可复现、可审计、可对比。**

- `系统设计.md` 仍是最终蓝图。
- V2 只吃掉最终蓝图里“评测模式”必须落地的部分。
- V2 不碰远程 Runner 平台化，不做多租户，不做 GitHub App，不做完整 CI 产品。
- 代码和文档都按本文档定边界；不要把 V3 的 Runner 协议提前塞进 V2。

## 0.1 协作进度用法

本文件从 V2 起同时作为实施清单和协作看板。能被执行、验收或跟踪的条目统一使用 Markdown checkbox。

每个阶段建议维护三类信息：

- 负责人：A / B / C / A+B / B+C / 全组
- 状态：未开始 / 进行中 / 阻塞 / 已完成
- 备注：只写阻塞原因、决策链接或验收证据，不写流水账

checkbox 约定：

- `- [ ]` 未完成，默认状态。
- `- [x]` 已完成，必须有代码、测试、截图、API 响应或文档证据支撑。

## 1. V2 目标（一句话）

把 V1 的 scripted MockLLM 展示，升级成**真实 LLM 可复现评测系统**：真实模型生成测试，干净代码定型测试集，bug 变体只重放，指标从数据库和 artifact 中可审计重算。

换句话说，V2 的主线不是“界面更多”，也不是“执行更炫”。V2 的主线是：

```text
真实 LLM run -> 冻结 prompt/tool/runtime -> 干净代码生成最终测试集 -> bug 变体重放 -> 指标落库 -> 前端展示实验结果
```

## 2. V2 分层：先 V2.1，再 V2.2

### 2.1 V2.1 必做：真实 LLM + 评测落库

V2.1 是必须完成的版本。

- [x] 后端 API/worker 支持配置真实 LLM provider。
- [ ] Prompt、Tool Schema、Runtime Profile 正式版本化。
- [x] 评测数据集、任务、seeded bug、bug variant、experiment、clean run、replay run 落库。
- [ ] 评测流程不再只靠 `eval/harness/run_eval.py` 写文件；脚本可以保留，但必须调用同一套服务逻辑。
- [ ] 后端必须提供 experiment API 和数据库聚合 metrics，真实 API 模式下能跑 seeded bug experiment。
- [ ] 前端能展示真实 experiment 的实验列表、实验详情、策略对比和逐 bug 捕获矩阵；`mockComparison` 只能作为离线 fallback 或开发夹具。
- [x] 单次 run report 必须补足测试质量证据，不能只展示通过/失败、tokens、工具调用和是否 Reflection。
- [x] Agent 生成上下文从“AST 摘要”升级为“AST 摘要 + 按目标函数/路由读取的源码片段”，但不引入无边界多轮自我探索。

### 2.2 V2.2 可做：本地 Docker executor

V2.2 是加固项，不允许阻塞 V2.1。

- [ ] 引入本地 Docker executor，替换或并存 V1 subprocess executor。
- [ ] 每次 run 使用独立容器和独立工作目录。
- [ ] 默认禁网、限制 CPU/内存/进程数、只挂载必要目录。
- [ ] 仍然是本机执行，不实现远程 Runner 注册、心跳、long-poll。

如果时间紧，Docker executor 就留在 V2.2 后半段或直接后置。不要为了容器化牺牲评测落库和真实 LLM 复现。

## 3. V2 明确不做

这些东西留给 V3 或更后面：

- 远程/自托管 Runner API：
  - `POST /api/v1/runners/register`
  - heartbeat
  - next-job long-poll
  - runner job lease
  - presigned artifact upload
- 多租户、权限系统、组织空间。
- GitHub App、自动拉 PR、完整 CI 集成。
- OpenTelemetry 全链路追踪。
- WebSocket 实时 trace 推送。
- Prompt 在线编辑器大后台。
- 覆盖率作为核心指标。

V2 可以为这些功能留字段和扩展点，但不能提前实现半套。

## 4. V2 成功标准

V2 结束时，必须能完成下面这条链路：

- [ ] 在 UI 或 CLI 中创建一个 experiment。
- [ ] 选择 dataset、eval task、策略版本、repeat 次数和真实 LLM provider。
- [ ] 系统在 clean snapshot 上运行 Agent，生成最终测试集。
- [ ] 如果允许 Reflection，只在 clean snapshot 上修一次。
- [ ] 系统冻结最终测试集 artifact。
- [ ] 系统把同一批测试重放到每个 bug variant。
- [ ] 系统把 clean false positive、seeded bug capture、cost、token、tool call、耗时写入数据库。
- [ ] `GET /api/v1/experiments/{experiment_id}/metrics` 返回策略对比表和逐 bug 捕获矩阵，形状兼容 V1 `comparison.json` 的 `rows` + `capture_matrix`。
- [ ] 前端能在真实 API 模式下展示 experiment metrics，并标明数据来源、provider、model、repeat count 和 mock/real 标签。
- [x] 单次 run report 能展示生成了哪些测试、对应的函数/路由、断言摘要、失败原因、Reflection 修改点和风险。
- [ ] 任意一个指标可以从 `test_runs`、`pytest_case_results`、`test_replays`、artifact 重新审计出来。

如果指标只能靠某个临时 JSON 文件解释，那就不是 V2 完成。
如果前端只能展示 `mockComparison`，也不是 V2 完成。
如果报告只能说明“跑完了”，但不能说明“测试质量如何”，也不是 V2 完成。

## 5. V2 数据模型

V1 已经有 14 张表：

```text
projects, project_snapshots, strategy_versions, test_plans, test_runs,
run_plan_items, run_attempts, generated_test_files, generated_test_cases,
pytest_case_results, trace_steps, run_events, run_artifacts, test_reports
```

V2 在这 14 张表基础上扩展，不推倒重来。

### 5.1 V2 新建表

#### 必建表（13 张）

| 表 | 用途 |
| --- | --- |
| `runtime_profiles` | 可复用执行环境配置；run 创建时冻结到 `test_runs.runtime_snapshot` |
| `strategies` | 策略逻辑父表，例如 Direct / Plan-and-Execute / ReAct |
| `prompt_versions` | 不可变 prompt 正文版本 |
| `tool_schema_versions` | 不可变工具 schema 正文版本 |
| `eval_datasets` | 评测数据集版本 |
| `eval_tasks` | 评测任务，定义 scope 和目标 |
| `seeded_bugs` | 缺陷语义定义 |
| `bug_variants` | 具体 bug 变体，canonical patch 或 snapshot |
| `experiments` | 一次实验定义 |
| `experiment_strategy_versions` | experiment 与 strategy_version 的多对多关联 |
| `experiment_clean_runs` | clean snapshot 上的生成运行 |
| `test_replays` | 把最终测试集重放到 clean/variant snapshot 的执行记录 |
| `experiment_replay_runs` | experiment 维度的 bug variant 捕获结果 |

#### V2 暂不建表

| 表 | 原因 |
| --- | --- |
| `runners` | V3 远程 Runner 协议才需要 |
| `runner_jobs` | V3 job lease 和回传幂等才需要 |
| `experiment_artifacts` | 先用 `run_artifacts` + experiment 表字段引用 artifact id |
| `human_scores` | 人工评分可以先写回 `generated_test_cases.human_meaningfulness_score` |

### 5.2 既有表迁移

#### `project_snapshots`

V1 的 snapshot 只是 root path 锚点。V2 要补到可复现，但不要求一口气做对象存储。

新增字段：

- `source_ref`
- `commit_sha`
- `dependency_summary`
- `entrypoint_guess`
- `analysis_summary`

保留：

- `root_path`
- `content_hash`

V2 规则：

- 本地 demo 项目可以继续用 `root_path`。
- 真正评测时必须计算 `content_hash`。
- 如果项目来自 git checkout，必须记录 `commit_sha`。
- 如果只是本地目录，必须记录目录 hash 和 dependency summary。

#### `strategy_versions`

V1 的 `strategy_versions` 是简化表。V2 要迁到正式版本体系。

新增字段：

- `strategy_id`
- `version`
- `prompt_version_id`
- `tool_schema_version_id`
- `temperature`
- `is_locked`

迁移规则：

- V1 三条 seed strategy 继续保留。
- 为 V1 三条策略补 `strategies` 父表。
- 为当前 prompt bundle 创建 `prompt_versions` 记录。
- 为当前工具 I/O schema 创建 `tool_schema_versions` 记录。
- `prompt_ref`、`tool_schema_ref` 暂时保留一版，标记为兼容字段；V3 或后续迁移再删。

#### `test_runs`

V2 需要正式使用 `runtime_profile_id`。

规则：

- 新 run 必须绑定 `runtime_profile_id`，除非是历史 V1 run。
- `runtime_snapshot` 仍然必须保存完整展开配置。
- `strategy_snapshot` 仍然必须保存完整 prompt、tool schema、模型参数。
- 历史 run 不回填假数据，不伪装可复现。

#### `run_artifacts`

V2 artifact 仍可用本地文件系统，但必须强化 metadata。

建议 metadata 包含：

- `kind`
- `created_by`
- `experiment_id`
- `clean_run_id`
- `replay_id`
- `content_role`
- `relative_paths`
- `hash_algorithm`

## 6. V2 表字段口径

字段细节以 `系统设计.md` §9 为准。本节只拍 V2 必须用到的字段。

### 6.1 `runtime_profiles`

关键字段：

- `id`
- `project_id`
- `name`
- `python_version`
- `install_command`
- `test_command`
- `env_template`
- `resource_limits`
- `network_policy`
- `created_at`
- `updated_at`

规则：

- `env_template` 只能保存变量名和默认占位，不保存 secret。
- `network_policy` V2 至少支持 `default`、`disabled`、`install_only`。
- run 创建时把 runtime profile 展开进 `test_runs.runtime_snapshot`。

### 6.2 `prompt_versions`

关键字段：

- `id`
- `name`
- `version`
- `content`
- `content_hash`
- `source_ref`
- `created_at`

规则：

- prompt 一旦参与 run，不允许原地修改。
- 修改 prompt 必须创建新 version。
- `content_hash` 是复现主键之一，不是装饰字段。

### 6.3 `tool_schema_versions`

关键字段：

- `id`
- `version`
- `schema_json`
- `content_hash`
- `created_at`

规则：

- schema 必须由 Pydantic model 或显式 JSON schema 生成。
- 不允许手写一份和代码漂移的 schema。
- run 创建时把展开后的 schema 存入 `test_runs.strategy_snapshot`。

### 6.4 `eval_datasets`

关键字段：

- `id`
- `name`
- `version`
- `description`
- `project_snapshot_ids`
- `created_at`

规则：

- dataset version 不可变。
- 如果数据集内容变化，创建新 version。
- `project_snapshot_ids` 可以先用 JSONB 数组，后续有复杂查询再拆关联表。

### 6.5 `eval_tasks`

关键字段：

- `id`
- `dataset_id`
- `project_snapshot_id`
- `target_scope`
- `goal`
- `expected_capabilities`

规则：

- 捕获率分母以 `eval_tasks.target_scope` 为准。
- V2 自动合成的 `test_plan.target_scope` 必须来自 eval task，不能另起一套口径。

### 6.6 `seeded_bugs`

关键字段：

- `id`
- `eval_task_id`
- `bug_type`
- `description`
- `expected_detection`

规则：

- seeded bug 描述的是缺陷语义。
- 不直接把代码 patch 塞在这里。

### 6.7 `bug_variants`

关键字段：

- `id`
- `seeded_bug_id`
- `variant_name`
- `canonical_kind`
- `patch_artifact_id`
- `mutated_snapshot_id`
- `ground_truth`

规则：

- V2 默认 `canonical_kind = patch`。
- patch 必须能从 clean snapshot 唯一应用。
- 如果 patch 命中 0 次或多次，variant 创建失败。
- V2.1 阶段 3 只写 inline canonical patch metadata：真实可应用数据在 `ground_truth.patch_artifact.patch` 的 `(file, old, new)` 三元组中，`ground_truth.patch_artifact.content_id/content_hash` 用于审计和去重。
- `patch_artifact_id` 暂作为 forward-reservation 字段；未落 `run_artifacts` 或真实 blob 前保持为空，阶段 4 replay 不允许按它解引用 patch。
- `mutated_snapshot_id` 是派生缓存，不是第二套真相。

### 6.8 `experiments`

关键字段：

- `id`
- `name`
- `dataset_id`
- `repeat_count`
- `llm_override`
- `status`
- `created_at`
- `started_at`
- `finished_at`
- `error_code`
- `error_message`

规则：

- experiment 是评测执行单元。
- strategy 列表走 `experiment_strategy_versions`，不把多对多硬塞成数组。
- experiment 失败是系统失败，不等于某个策略捕获率低。
- `status`、`started_at`、`finished_at`、`error_code`、`error_message` 是 V2 为执行态补的字段，属于对系统设计 §9.3 蓝图的落地扩展。
- `llm_override` 是实验级覆盖策略，存 JSONB；它不是最终 provider/model/temperature 的第二份真相。
- 最终 LLM 配置只在创建每个 clean `test_run` 时解析一次，并冻结进 `test_runs.strategy_snapshot`。
- 解析优先级固定为：`strategy_versions` 默认模型参数 < `experiments.llm_override` 覆盖层 < repeat 派生参数。
- 实际调用 LLM 的参数和 `strategy_snapshot` 里的参数必须来自同一份 resolved spec，禁止“快照写一套、实跑另一套”。

### 6.9 `experiment_strategy_versions`

关键字段：

- `experiment_id`
- `strategy_version_id`
- `index`
- `created_at`

规则：

- `(experiment_id, strategy_version_id)` 必须唯一。
- `index` 只用于展示和稳定遍历顺序，不参与指标语义。
- experiment 级别的模型覆盖不写在这里，统一写 `experiments.llm_override`。

### 6.10 `experiment_clean_runs`

关键字段：

- `id`
- `experiment_id`
- `eval_task_id`
- `strategy_version_id`
- `repeat_index`
- `clean_run_id`
- `generated_test_set_artifact_id`
- `false_positive`
- `clean_metrics`

规则：

- clean run 负责生成和定型测试集。
- 假阳性只在 clean 层判断。
- `generated_test_set_artifact_id` 必须指向最终测试集 artifact。

### 6.11 `test_replays`

关键字段：

- `id`
- `experiment_clean_run_id`
- `generated_test_set_artifact_id`
- `target_snapshot_id`
- `bug_variant_id`
- `status`
- `pytest_summary`
- `started_at`
- `finished_at`
- `error_code`
- `error_message`

规则：

- replay 不规划、不生成、不调用 LLM。
- replay 只执行已经冻结的测试集。
- replay 可以跑 clean snapshot，也可以跑 bug variant snapshot。
- replay 的 pytest 结果必须能映射到原始 nodeid。

### 6.12 `experiment_replay_runs`

关键字段：

- `id`
- `experiment_clean_run_id`
- `bug_variant_id`
- `replay_id`
- `captured_bug`
- `replay_metrics`

规则：

- 捕获率只看这里。
- `captured_bug = true` 必须能追到具体 failing nodeid。
- 不允许 Agent 在 bug variant 上重新生成测试。

## 7. V2 API 子集

V2 API 只补评测和版本管理必要入口。

### 7.1 Runtime Profile

```text
POST   /api/v1/projects/{project_id}/runtime-profiles
GET    /api/v1/projects/{project_id}/runtime-profiles
GET    /api/v1/runtime-profiles/{runtime_profile_id}
```

V2 暂不做复杂编辑。需要修改时创建新 profile 或新版本字段，别原地改历史语义。

### 7.2 Prompt / Tool Schema

```text
POST   /api/v1/prompt-versions
GET    /api/v1/prompt-versions/{prompt_version_id}
POST   /api/v1/tool-schema-versions
GET    /api/v1/tool-schema-versions/{tool_schema_version_id}
```

V2 不做 prompt 在线编辑 UI，但 API 要有。

### 7.3 Strategy

```text
POST   /api/v1/strategies
POST   /api/v1/strategies/{strategy_id}/versions
GET    /api/v1/strategy-versions/{version_id}
GET    /api/v1/strategies
GET    /api/v1/strategies/{strategy_id}/versions
```

规则：

- 创建 strategy version 时必须绑定 prompt version 和 tool schema version。
- `is_locked = true` 的 strategy version 不允许修改。
- 已参与 run 的 strategy version 自动视为 locked。

### 7.4 Dataset / Bug

```text
POST   /api/v1/eval-datasets
GET    /api/v1/eval-datasets
GET    /api/v1/eval-datasets/{dataset_id}
POST   /api/v1/eval-datasets/{dataset_id}/tasks
GET    /api/v1/eval-datasets/{dataset_id}/tasks
POST   /api/v1/eval-tasks/{task_id}/seeded-bugs
GET    /api/v1/eval-tasks/{task_id}/seeded-bugs
POST   /api/v1/seeded-bugs/{seeded_bug_id}/variants
GET    /api/v1/seeded-bugs/{seeded_bug_id}/variants
```

V2 可以先用 seed 脚本导入 demo dataset；API 主要为后续扩展和前端查询服务。

### 7.5 Experiment

```text
POST   /api/v1/experiments
GET    /api/v1/experiments
GET    /api/v1/experiments/{experiment_id}
POST   /api/v1/experiments/{experiment_id}/runs
POST   /api/v1/experiments/{experiment_id}/cancel
GET    /api/v1/experiments/{experiment_id}/clean-runs
GET    /api/v1/experiments/{experiment_id}/replay-runs
GET    /api/v1/experiments/{experiment_id}/metrics
```

规则：

- `POST /runs` 异步入队，不阻塞 HTTP。
- experiment retry 创建新 experiment 或新 run batch，不覆盖历史结果。
- cancel V2 可以先软取消；如果已经进入单个 subprocess/Docker 执行，尽力中断作为 V2.2 加固。

## 8. V2 评测流程

### 8.1 创建实验

输入：

```json
{
  "name": "demo-real-llm-v2",
  "dataset_id": "dataset-demo-v2",
  "strategy_version_ids": ["sv-direct-v2", "sv-plan-v2", "sv-react-v2"],
  "repeat_count": 3,
  "llm_override": {
    "provider": "openai",
    "model": "gpt-5",
    "temperature_policy": {
      "mode": "fixed",
      "value": 0.2
    }
  }
}
```

注意：

- API 请求不传 API key。
- key 只从环境变量或 secret manager 读。
- `llm_override` 可以入库，secret 不入库。
- `llm_override` 只表达覆盖策略，不保存每个 clean run 的最终展开值。
- 每个 clean run 创建前，系统用 `strategy_version` 默认配置加 `llm_override` 解析出 resolved spec；resolved spec 同时用于真实 LLM 调用和 `test_runs.strategy_snapshot` 冻结。
- 如果没有 `llm_override`，就完全使用各 strategy version 自带的模型配置。

### 8.2 Clean run

每个 `(eval_task, strategy_version, repeat_index)` 创建一个 clean run：

1. 合成或引用 `test_plan`。
2. 创建 `test_run`，绑定 clean `project_snapshot_id`、runtime profile、strategy version。
3. 为 Agent 准备上下文：先跑 `analyze_project` 得到 AST 摘要，再按目标函数/路由读取必要源码片段。
4. 调用现有 Agent orchestrator。
5. 只在 clean run 上允许 Reflection。
6. 选择 final attempt。
7. 冻结最终测试集 artifact。
8. 计算 `false_positive`。
9. 写 `experiment_clean_runs`。

Agent 上下文规则：

- V1 的 AST 摘要继续保留，它稳定、便宜、可控。
- V2 必须补“按目标读取源码片段”：例如目标函数定义、同文件依赖的小函数、FastAPI route handler、Pydantic model 定义。
- 源码片段必须经过路径白名单和大小限制，写入 trace payload 或 artifact 时仍然受脱敏规则约束。
- 不引入 SWE-agent 式无限工具循环；如果需要更多上下文，必须由明确的目标映射和预算驱动。
- 如果静态映射不到源码片段，报告和 trace 要记录 `context_incomplete`，不要假装上下文完整。

clean run 判定：

- 如果系统失败，`experiment_clean_runs.clean_metrics.error_code` 必须记录。
- 如果 pytest 在 clean 代码上有 assertion failure，记为 false positive。
- import、collection、fixture、环境错误不能算 seeded bug 捕获，只能算 clean run 失败或无效测试集。

### 8.3 Replay run

每个 clean run 对每个 in-scope bug variant 创建 replay：

1. 从 `generated_test_set_artifact_id` 读取测试内容。
2. 物化 bug variant snapshot。
3. 把测试内容写入 variant 的受控测试目录。
4. 执行 pytest。
5. 保存 replay 的 pytest summary 和 case results。
6. 计算 `captured_bug`。
7. 写 `test_replays` 和 `experiment_replay_runs`。

捕获判定必须同时满足：

- clean run 中同一 nodeid 通过。
- variant replay 中同一 nodeid 失败。
- failure_type 是 `assertion`。
- bug variant 在 eval task scope 内。
- 失败不是 import、collection、fixture、环境错误。

关于“失败与 ground truth 相关”的 V2 简化：

- 系统设计 §11.1 要求失败原因与 seeded bug ground truth 相关。
- V2 默认 `bug_variants.canonical_kind = patch`，且 patch 必须唯一命中 clean snapshot。
- replay 使用 clean 阶段已经冻结的同一批测试内容，不重新生成、不调用 LLM。
- 因此在 V2 demo/default 口径下，“clean 通过 + 单点 patch variant 上同一 nodeid assertion failure”即可视为与该 patch 的 ground truth 相关。
- 如果未来允许多点 patch、手写 mutated snapshot 或非唯一变体，必须补显式 ground truth matcher，不能继续套用这个简化。

### 8.4 指标聚合

`GET /api/v1/experiments/{experiment_id}/metrics` 必须从数据库聚合，不读取临时 JSON。

V2.1 必算指标：

- `seeded_bug_capture_rate`
- `clean_false_positive_rate`
- `cost_per_captured_bug`
- `avg_tokens`
- `avg_tool_calls`
- `avg_duration_ms`
- `pytest_collection_success_rate`
- `reflection_contract_pass_rate`
- `reflection_acceptance_rate`

V2.1 不把语义版 `reflection_valid_fix_rate` 当核心必算指标。原因很简单：系统设计里的“有效修复”需要判断修复原因是否属于 import、fixture、调用签名、测试数据或断言方向等测试自身问题，纯自动规则不一定可信。

V2.1 的自动替代口径：

- `reflection_contract_pass_rate = no_contract_violation_reflection_attempts / reflection_attempts`
- `reflection_acceptance_rate = accepted_reflection_attempts / reflection_attempts`

只有同时满足下面条件时，才允许计算并展示 `reflection_valid_fix_rate_auto`：

- reflection 后 clean run 通过；
- `contract_guard` 没有发现 skip/xfail、空洞断言、宽泛 except、断言数缩水、测试数缩水；
- reflection 输出没有标记 `suspected_code_bug`；
- reflection reason 能落入明确白名单，例如 import、fixture、调用签名、测试数据、断言方向。

如果缺少 reason 白名单或人工复核，UI 和报告只能展示自动替代指标，不能把它包装成系统设计里的语义版 `reflection_valid_fix_rate`。

V2 可保留 `meaningful_assertion_rate` 的字段，但没有人工评分流程时不要假装已完成。

### 8.5 单次 run report 增强

V2 不能只把 report 当“运行结束摘要”。普通 run report 至少要让第三方判断测试质量，而不是只看到通过/失败数量。

必须补充：

- 生成测试清单：测试函数名、文件路径、起止行、pytest nodeid。
- 目标映射：每个测试对应的函数、路由或模块；无法映射时标记 `unmatched`。
- 断言摘要：来自模型声明、AST 规则预筛或人工评分字段，不能只写“生成了 N 个用例”。
- 失败说明：按 assertion/import/collection/fixture/error 分类展示失败原因。
- Reflection 证据：是否触发、修复原因、修改点、是否被 contract guard 拒绝。
- 上下文完整性：本次生成是否只用了 AST 摘要，还是同时读取了目标源码片段；缺失时写风险。
- 成本指标：tokens、工具调用、耗时，和当前策略版本一起展示。

可以后置但要预留字段：

- 规则预筛 flags，例如 `only_status_200`、`weak_assertion`、`broad_exception`。
- 人工有效性评分 `human_meaningfulness_score`。
- 与 experiment clean/replay 的跳转关系。

## 9. 真实 LLM 接入

V1 已经有 OpenAI Responses 和 OpenAI Chat Completions 兼容 adapter。V2 要做的是把它接入 B 线 API/worker 主链路。

### 9.1 Provider 配置

支持：

- `mock`
- `openai`
- `openai_chat_compat`

必要配置：

- provider
- model
- base_url 可选
- max_output_tokens
- temperature 或 temperature_policy
- reasoning_effort 可选
- timeout_seconds
- max_retries

禁止：

- API key 入库。
- API key 写进 `.env.example` 的示例值。
- API key 出现在 trace payload。
- provider 原始错误带 Authorization header。

### 9.2 Worker 接入规则

`execute_run_sync(...)` 不再写死 `_mock_llm()`。

规则：

- 普通 run 根据 request 或 strategy snapshot 解析 provider。
- experiment run 先读取 strategy version 默认 provider/model/model_params，再应用 `experiments.llm_override`，得到 resolved spec。
- resolved spec 同时用于创建 LLM client 和冻结 `test_runs.strategy_snapshot`，不允许再从别处读第二套最终值。
- 如果 provider 配置无效，run 失败为 `LLM_PROVIDER_ERROR` 或 `TOOL_ARGUMENT_ERROR`，错误脱敏。
- MockLLM 仍保留，用于测试和演示。

### 9.3 测试策略

自动化测试不调用真实外网 LLM。

必须覆盖：

- provider factory 选择。
- API key 缺失错误。
- 错误脱敏。
- 429/5xx 退避重试。
- 4xx 快速失败。
- JSON/schema 重试记录 trace。
- provider 失败 token 口径。

真实 LLM smoke 作为手工命令，不进 CI 必过项。

## 10. Runtime 与执行器

### 10.1 V2.1 subprocess 继续可用

V2.1 可以继续使用 V1 subprocess executor，但必须更诚实、更可复现：

- runtime profile 入库。
- run 创建时冻结 runtime snapshot。
- pytest 命令来自 runtime snapshot。
- 环境变量脱敏。
- timeout 和资源限制写入 snapshot。
- artifacts 目录按 run/replay 隔离。

红线：

- V2.1 subprocess 只能强制 timeout。
- CPU、内存、进程数限制在 V2.1 只是声明性记录，不是强制隔离。
- 强制资源限制必须等 V2.2 Docker executor。
- 不允许把 V2.1 subprocess 说成安全沙箱。

### 10.2 V2.2 Docker executor

本地 Docker executor 的最低要求：

- 每次执行独立容器。
- 源码只读挂载。
- 生成测试目录可写。
- 不挂载 Docker socket。
- 不使用 privileged。
- 容器内非 root 用户。
- CPU、内存、进程数、超时限制。
- 默认禁网。
- artifacts 复制回本地 artifact store。

V2.2 不做：

- runner register。
- runner heartbeat。
- runner lease。
- 远程 artifact presigned upload。

### 10.3 依赖安装策略

依赖安装和禁网冲突，V2 这么拍板：

- V2.1 demo dataset 使用预装本地环境。
- `runtime_profiles` 指向本地已有 Python/venv 或 conda env。
- 测试执行阶段不做 `pip install`。
- `network_policy = disabled` 表示测试阶段禁网或尽力禁网；subprocess 下这是声明性策略，Docker 下才强制。
- V2.2 Docker executor 使用预构建镜像承载依赖。

不要一边写“默认禁网”，一边在测试阶段偷偷 `pip install`。

## 11. Artifact 规则

V2 artifact 是复现和审计的核心，不是日志垃圾桶。

- generated final test set。
- clean run report JSON/Markdown。
- replay pytest report。
- bug variant patch。
- experiment metrics snapshot。

建议 artifact 类型：

- `generated_test_set`
- `pytest_json`
- `junit_xml`
- `report_markdown`
- `report_json`
- `bug_patch`

规则：

## 16. V2 风险清单

### 风险 1：真实 LLM 输出不稳定

处理：

- [ ] 保留 structured retry。
- [ ] 保留 MockLLM 自动化测试。
- [ ] experiment 支持 repeats。
- [ ] 指标展示均值和标准差。

### 风险 2：评测落库后口径漂移

处理：

- [ ] 把 final attempt 选择逻辑继续复用同一个 helper。
- [ ] helper 的输入抽象成统一 records 协议，至少提供 attempts、pytest results、generated files、generated cases 的同形状读取。
- [ ] InMemoryRecorder 和 DB recorder 必须喂给 helper 同一形状的数据。
- [ ] 禁止 `InMemoryRecorder` 用属性访问一套逻辑、DB 查询重写另一套逻辑。
- [ ] 指标聚合写单元测试。
- [ ] replay 逻辑和报告逻辑不要各写一套。

### 风险 3：artifact 和数据库不一致

处理：

- [ ] artifact 写入后计算 hash。
- [ ] DB 只保存 hash 和 URI。
- [ ] 读取 replay 前校验 hash。

### 风险 4：Docker 抢主线

处理：

- [ ] V2.1 不依赖 Docker。
- [ ] Docker executor 作为 V2.2。
- [ ] 任何 Docker 工作不得改动评测指标口径。

### 风险 5：前端变成管理后台

处理：

- [ ] Experiment Detail 是主屏。
- [ ] Prompt/runtime 页面后置。
- [ ] UI 优先讲清楚“这个 bug 是被哪个测试抓到的”。

### 风险 6：评测仍停留在脚本层

处理：

- [ ] `eval/harness/run_eval.py` 只能作为 CLI 包装或兼容入口。
- [ ] clean run、replay、metrics 聚合必须落到 service/API/DB。
- [ ] `GET /api/v1/experiments/{experiment_id}/metrics` 是前端真实数据源。
- [ ] V2 验收必须从数据库和 artifact 重算指标，不能只检查 `comparison_real.json`。

### 风险 7：报告只剩运行统计

处理：

- [ ] report 必须展示测试清单、目标映射、断言摘要、失败分类、Reflection 证据和上下文完整性。
- [ ] 通过率、tokens 和工具调用只算运行统计，不算测试质量证明。
- [ ] 如果没有人工评分，就不要展示 `meaningful_assertion_rate` 为已完成指标。

### 风险 8：上下文读取仍然偏浅

处理：

- [ ] `analyze_project` 摘要后必须按目标函数/路由读取源码片段。
- [ ] 读取上下文受路径白名单、大小限制和预算控制。
- [ ] 缺失源码片段时写 `context_incomplete` 风险，而不是静默生成。
- [ ] 不用多轮无边界探索弥补上下文质量，优先把目标映射和源码片段选择做准。

## 18. 结论

V2 的价值不在“比 V1 多几张表”。V2 的价值在于：当别人问“你说 ReAct 更好，证据在哪”时，TRACE 能从数据库、artifact、trace、pytest case result 一路追到每个策略、每次重复、每个 seeded bug、每个 nodeid；当别人问“这份测试质量如何”时，报告能说清楚测试目标、断言、失败原因、Reflection 证据和上下文来源。

这才是 V2 要打穿的东西。其他都先靠边。
