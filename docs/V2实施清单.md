## 3. V2.1 已交付能力

评测尺子：

- clean run 与 variant replay 分层落库。
- replay 不调用 LLM，只执行冻结测试集。
- generated test set artifact replay 前校验 hash。
- `metric_status = ok | evaluable_zero_capture | invalid_test_set` 已落地。
- `invalid_test_set` 不再解释为 0 捕获。
- clean false positive、captured bug、cost/token/tool calls、duration 可审计。
- `capture_matrix` 保留 bool 兼容口径，同时新增 `capture_matrix_counts`。
- `sampled_mutation_score` 已有最小合同和聚合口径。

项目理解与上下文：

- target source、dependency、reference、fixture、existing tests、Pydantic model 字段、pytest failure context、EvaluationEvent context 都可进入 source context。
- 缺目标源码时 clean run 阻断为 `context_incomplete_blocking / invalid_test_set`。
- retrieval trace 记录 resolved / missing / truncated / failed，不再静默吞证据。
- rg / ast-grep / LSP v0 provider 已接入保守 fallback。

生成与契约：

- Contract Guard 拒绝弱断言、目标漂移、目标函数重定义、未知 fixture、错误 mock 参数、2xx-only route oracle。
- 已补函数/路由窄口径静态 oracle 校验。
- 已补一次结构化 Contract Guard retry；仍失败才 `PIPELINE_REJECT`。
- ReAct Reflection 可消费 pytest failure context 和 prompt-safe EvaluationEvent。

事件与回灌：

- `EvaluationEvent` 稳定契约已落地，不新增 DB 表，由现有 DB/artifact/metrics 投影。
- metrics 顶层输出 `evaluation_events` 与 `reflection_event_backfeed`。
- prompt-safe 回灌只允许 `flaky_clean_replay`、`probe_check_failed`、`replay_failure`。
- `provider_failure`、`artifact_hash_mismatch` 默认只进 report/audit，不进 LLM prompt。

Auto mutation：

- candidate discovery dry-run v0 支持单比较边界变异。
- dry-run API 与审计报告 CLI 已有。
- selected candidate 人工确认入库必须带 probe，并通过 clean/buggy 行为校验。
- confirmed auto mutation variant 可进入 sampled mutation score 分母。

前端与运行闭环：

- Experiment List / Detail、capture matrix、clean run drilldown、replay evidence、Evaluation Events timeline 已有。
- Experiment 创建入口已支持 dataset、strategy versions、repeat_count、provider/model。
- Dataset 前端目前只支持基础创建，不含完整 task / seeded bug / variant 编辑器。

## 5. V2.2 二人分工边界

V2.2 分工按工程边界记录，不按 Direct / Plan / ReAct 策略分工。策略是共同评测对象，不是代码所有权边界。具体人员、顺序和阶段主线由后续实际分工时决定。

| 分工 | 责任边界 | 稳定输出 | 红线 |
| --- | --- | --- | --- |
| A 线：评测尺子 / 执行平台 | executor、runtime profile、experiment runner、replay、metrics、DB/API evidence | `EvaluationEvent`、experiment metrics、replay artifact、dataset/variant 状态机 | provider failure、flaky、artifact mismatch、invalid test set 不能压成 0 捕获；执行边界不能宣传成未实现的安全沙箱 |
| B 线：项目理解 / 数据集 / 生成约束 | source context、retrieval trace、Contract Guard、Dataset UI、mutation confirmation、prompt-safe 回灌 | generated test metadata、target mapping、guard violation、source context trace、dataset/variant 编辑证据 | 证据不足时必须拒绝或标记 incomplete；不能靠 prompt 自律替代 guard；auto mutation v0 不能宣传为完整 mutation engine |

协作接口：

- A 输出稳定的 `EvaluationEvent`、experiment metrics、replay artifact 和 dataset/variant 状态机。
- B 消费 prompt-safe evidence，输出可 replay 的 generated test metadata、target mapping、guard violation 和 source context trace。
- A 不依赖 B 的 prompt 内部文本；B 不依赖 A 的内部 metrics payload，只依赖稳定事件和 schema。
- 共同维护 schema / migration / 前端展示口径；任何跨线字段先写合同测试，再接 UI 或 runner。

## 6. 关键口径

- clean run 失败、provider 失败、artifact hash mismatch 是系统/评测有效性问题，不是 0 捕获。
- mutation score 只统计可评价测试集。
- false positive 是 clean 代码上测试失败，必须和 variant 捕获分开。
- real LLM 与 MockLLM 结果分开解释。
- prompt 输入只吃 prompt-safe evidence；审计事件不能默认塞回 LLM。
- `backend/llm.config.json` 和 `.env*` 永远不入库。
