## 2. 已完成的关键能力

### 2.1 评测口径

- `metric_status = ok | evaluable_zero_capture | invalid_test_set` 已落地。
- `invalid_test_set` 不再解释为模型 0 捕获。
- clean false positive 和 variant captured bug 分层落库。
- replay 不调用 LLM，只执行冻结测试集。
- generated test set artifact replay 前校验 hash；篡改会让 experiment failed。
- provider/LLM 调用失败时 experiment failed，不能伪装成 completed + 0 捕获。
- 0 捕获时 `cost_per_captured_bug = null`，状态为 `no_bug_captured`。
- comparison real 报告包含“状态”列和 `experiment_id`。

### 2.2 项目理解与上下文

- `target_scope` 符号解析已修：符号型目标会扫描项目并映射到真实 source snippet。
- 缺目标源码时 clean run 标记 `context_incomplete_blocking / invalid_test_set`，不调用 LLM，不进入 mutation score。
- Source context 已包含目标源码、fixture、existing tests、Pydantic model 字段证据。
- Reflection prompt 已包含失败 nodeid、异常类型、message、file:line、traceback 摘要。

### 2.3 Contract Guard 与生成状态机

- 已拒绝明显弱测试：`assert True`、`assert x is not None`、skip/xfail、宽泛 except、空测试。
- 已拒绝目标绑定漂移、目标函数重定义、未知 fixture 参数、错误 `patch(..., new_value)` mock 参数。
- 已检查 Pydantic 请求字段来源；缺模型字段证据时禁止 `json=...` 请求体。
- 已拒绝成功路由只有 `2xx status_code` 的测试。
- 已修 route target 归一：`route:GET /path`、`GET /path`、`/path` 可按规则匹配。
- 已修 pytest 参数化和类方法 case 名称归并：`test_x[...]`、`TestClass.test_x` 对齐真实函数名。
- 已补源码行为窄口径 oracle 检查：函数/路由在可静态求值时，矛盾断言拒绝。
- 已补一次结构化 Contract Guard retry；仍失败才 `PIPELINE_REJECT / invalid_test_set`。

### 2.4 Probe 自检

- Demo hand-crafted seeded bug 的 `probe / clean_value / buggy_value` 已接入 DB experiment 与 script harness。
- variant replay 前验证 clean/buggy 两侧行为是否分别等于 ground truth。
- probe 失败会让 experiment failed，避免等价/失效 mutant 混进分母。
- probe 成功结果写入 `replay_metrics.probe_check` / harness variant 记录。
- 函数 probe 只接受 AST 限定的“函数名 + 常量参数”；route probe 通过 FastAPI `TestClient` 取状态码。

## 3. 当前未完成

优先继续做评测尺子，其次做项目理解。

1. Flaky check：final test set 冻结前在 clean snapshot 连跑 N 次，nodeid 和结果稳定才进入 mutation score。
2. Repeat 捕获比例：capture matrix 从 bool OR 升级到 `captured_count / repeats`，避免“任一 repeat 捕获”夸大稳定性。
3. 更深业务 oracle：Contract Guard 目前只覆盖可静态求值的窄口径函数/路由。
4. 多源 retrieval trace：rg / ast-grep / LSP / framework scanner 的统一证据输出还未做。
5. final attempt / InMemoryRecorder / DB recorder records 协议还可继续统一。
6. Auto mutation、sampled mutation score、selected mutant matcher 尚未做。
7. Docker executor 后置到 V2.2；V2.1 subprocess 不能声称是安全沙箱。
8. 前端 Experiment Detail / Comparison 仍需保存截图或等价 evidence。

## 4. 二人分工

不要按 Direct / Plan / ReAct 拆，也不要按 prompt / 代码拆。策略不是模块边界，prompt 不是系统状态机。V2.1 最稳的拆法是：A 管评测尺子，B 管项目理解。

### 4.1 A：评测尺子 / Runner 状态机

目标：保证“结果到底能不能信”。

负责范围：

- Flaky check：final test set 在 clean snapshot 连跑 N 次，nodeid 和结果稳定才冻结。
- Repeat 捕获比例：capture matrix 从 bool OR 改成 `captured_count / repeats`。
- Probe / mutation 口径继续加固：当前 demo probe 已接入，后续接 auto mutant 分层。
- Metrics / DB 聚合 / artifact replay 审计。
- final attempt、InMemoryRecorder、DB recorder 协议统一。
- 真实评测脚本、comparison 输出、文档证据更新。

边界：

- 不靠 prompt 调参修指标。
- 不改测试生成策略的业务语义。
- 所有指标必须能从 DB/artifact 重算。
- invalid / flaky / provider failure 不能被压成 0 捕获。

### 4.2 B：项目理解 / Contract Guard / Context Builder

目标：保证“模型拿到的证据够不够写强测试”。

负责范围：

- 多源 retrieval trace：`rg / ast-grep / LSP / framework scanner` 的结构化证据。
- Context Builder：目标源码、Pydantic schema、fixture、existing tests、failure context 的排序和预算。
- Contract Guard 更深业务 oracle：继续拦源码无证据的异常/边界假设。
- route / model / fixture 证据链增强。
- Reflection 输入质量和修复约束。
- Generate prompt 只表达系统约束，不把 prompt 当核心修复。

边界：

- 不改 mutation score 口径。
- 不改 experiment metrics 聚合。
- 生成测试过不了 Guard 就系统拒绝，不靠“下次模型注意”。

### 4.3 协作接口

- B 输出的 test case metadata / target mapping 必须满足 A 的 replay 和 metrics 需要。
- A 的 flaky / invalid / probe 结果要回写成 B 能读的 failure context。
- 文档由合并者更新 `docs/V2实施清单.md`；长历史和旧流水进 `docs/V2实施归档.md`。

推荐排期：

1. 第一轮：A 做 flaky gate；B 做 retrieval trace schema。
2. 第二轮：A 做 repeat 捕获比例；B 做更深 oracle Guard。
3. 第三轮：A 做 recorder 协议统一；B 做 micro tool-calling 最小状态机设计。

## 5. 核心口径

### 5.1 Experiment 与 LLM

- Experiment 是评测执行单元。
- `llm_override` 只是覆盖层，不含 secret。
- 最终 provider/model/temperature/max_output_tokens 在每个 clean run 的 `strategy_snapshot.resolved_llm` 冻结。
- 实际调用 LLM 的参数必须与 `strategy_snapshot` 同源。
- provider 为 `mock` 才走 MockLLM；真实 provider 走 provider factory。

### 5.2 Clean Run

- Clean run 只在 clean snapshot 上生成和定型测试集。
- Reflection 只能修测试，不能改业务源码。
- final attempt 生成最终测试集 artifact。
- clean replay assertion failure 属于 false positive。
- clean replay collection/import error 或缺目标源码属于 `invalid_test_set`。

### 5.3 Replay Run

- Replay run 只执行冻结测试集，不规划、不生成、不调用 LLM。
- replay 前必须读取 artifact 并校验 hash。
- variant patch 必须从 clean snapshot 唯一应用；命中 0 次或多次都失败。
- replay 结果必须能映射到原始 pytest nodeid。

### 5.4 Captured Bug

默认 V2.1 口径：

```text
clean replay passed nodeid
AND same nodeid has assertion failure on variant replay
=> captured_bug = true
```

只从 `experiment_replay_runs.captured_bug` 聚合捕获率。

### 5.5 Source Context

- seeded bug experiment 中，目标必须能解析到真实 source path 和源码片段。
- 缺目标源码是 blocking invalid，不允许靠 prompt 猜。
- 模型字段、fixture、existing tests 是辅助证据，不能掩盖目标源码缺失。
- 源码片段上下文只服务测试生成，不允许 Agent 修改业务源码。

## 7. 下一步建议

下一刀建议做 flaky check。理由很简单：如果 clean 测试本身不稳定，mutation score 就是坏尺子；这比继续调 prompt 更基础。

推荐执行顺序：

1. 设计 final test set 的 clean repeated replay 口径。
2. 先在 DB experiment runner 上实现最小 `N=3` flaky gate。
3. 把不稳定 nodeid 写入 `clean_metrics` 和 report quality。
4. 更新 metrics：不稳定测试不进入 captured bug 分母。
5. 再做 repeat 捕获比例展示。
