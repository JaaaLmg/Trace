## 2. 已完成的关键能力

### 2.1 评测口径

- `metric_status = ok | evaluable_zero_capture | invalid_test_set` 已落地。
- `invalid_test_set` 不再解释为模型 0 捕获。
- clean false positive 和 variant captured bug 分层落库。
- replay 不调用 LLM，只执行冻结测试集。
- generated test set artifact replay 前校验 hash；篡改会让 experiment failed。
- provider/LLM 调用失败时 experiment failed，不能伪装成 completed + 0 捕获。
- eval CLI `--smoke` 中 clean run 未 completed 时会非 0 退出、跳过 comparison table / capture matrix 输出；真实 provider 认证失败不再被 legacy harness 误报成“可评价 0 捕获”。
- 已新增最小稳定 `EvaluationEvent` 契约与 metrics 顶层 `evaluation_events` 投影：覆盖 `invalid_test_set`、`context_incomplete_blocking`、`flaky_clean_replay`、`probe_check_passed/failed`、`replay_captured/uncaptured/failure`、`artifact_hash_mismatch`、`provider_failure`；事件带确定性 `event_id`、稳定 `source_ids`、`nodeids`、`artifact_ids` 和精简 payload；事件投影已抽到独立 `evaluation_events` service，由现有 DB/artifact/metrics 合成事件，不新增 DB 表，不让 B 线直接依赖 A 内部 `clean_metrics` / `replay_metrics` dict。

- comparison real 报告包含“状态”列和 `experiment_id`。
- `capture_matrix` 保留兼容 bool，新增 `capture_matrix_counts` 表达每个 bug/策略的 `captured_count / repeat_count / capture_rate`。
- Experiment metrics API 已绑定 `ExperimentMetricsResponse` response model；`capture_matrix_counts`、`evaluation_events` 和 `reflection_event_backfeed` 在 HTTP 出口受合同校验且不会被过滤。`reflection_event_backfeed` 由稳定事件投影重算，记录每个 clean repeat 的 prompt-safe included / audit-only filtered 决策和 reason；`clean_metrics` / `replay_metrics` / runtime / strategy snapshot 这类审计 payload 允许保留额外字段，但稳定 `EvaluationEvent` 契约仍严格。

### 2.2 项目理解与上下文

- `target_scope` 符号解析已修：符号型目标会扫描项目并映射到真实 source snippet。
- 缺目标源码时 clean run 标记 `context_incomplete_blocking / invalid_test_set`，不调用 LLM，不进入 mutation score。
- Source context 已包含目标源码、同文件直接 dependency、project-local `from ... import helper` / relative import alias 跨文件直接 dependency、模块属性调用 dependency、函数到函数的多跳 call slice v1、`self.method(...)` / `ClassName.method(...)` 类方法最小 dependency、LSP reference 调用点、fixture、existing tests、Pydantic model 字段证据；Reflection 前可追加 pytest failure context 证据；A 线 `EvaluationEvent` 可经 B adapter 转成 `failure_context / evaluation_event` evidence。
- Source context 已补最小 retrieval trace schema：目标源码、同文件 / 跨文件直接 dependency、LSP reference 调用点、Pydantic model、fixture、existing tests、pytest failure context、EvaluationEvent failure context 片段带 `source_kind / retrieval_source / source_path / symbol / target / line range / confidence / content_hash / retrieval_trace_id`；缺失、截断、读失败会进入 `retrieval_trace` 和 risk notes。
- Context Builder 已补最小排序/预算规则：`target_source > dependency > reference > model_schema > fixture > existing_test`；多目标场景会先预算全部目标源码，再追加 dependency / reference / support context，避免辅助证据挤掉目标源码；support 候选会在预算前按 `source_path + symbol` 去重，保留高优先级/高置信证据，重复项写入 resolved retrieval trace，不静默吃掉 support 预算；reference / support context / dependency 因预算被裁掉时保留目标源码，并把整体标为 `partial/context_incomplete`，不再伪装 complete。
- 已新增最小 `rg_search` lexical retrieval provider，输出结构化 match：`source_path / line_range / line_text / content_hash / trace_id / confidence / engine`；Context Builder 已保守接入 rg fallback：只在 analysis/framework 解析失败时 exact lexical 召回，且必须能用 AST 切出同名 symbol 或 route decorator handler 才升级为目标源码，否则只写 `missing` trace / risk note。
- 已新增最小 `ast_grep_search` 结构化 AST provider，输出函数/类/路由 handler 的 `source_path / line_range / matched_text / symbol / content_hash / trace_id / confidence / engine`；本机存在 `ast-grep`/`sg` 时会机会性走真实 binary，并用项目内路径、行号和 Python AST 再确认候选后才输出 `engine=ast_grep`；binary 缺失、失败或无可验证匹配时降级到 `python_ast_fallback`。
- 已新增 `lsp_definition` / `lsp_references` v0 工具契约和 provider：输出 definition / reference 的 `source_path / line_range / matched_text / symbol / content_hash / trace_id / confidence / engine`；当本机可用 `pyright-langserver` 且输入提供 `from_path/line/column` 时走 pyright JSON-RPC definition / references，否则结构化记录 fallback 并使用 Python AST fallback。Context Builder 对 analysis/framework/direct 命中仍优先；analysis 缺失时会收集 `lsp_definition / ast_grep / rg` 目标候选，按 `source_kind`、confidence band、retrieval source priority 排序，低置信早到候选不能仅因链路顺序压过高置信结构证据；`lsp_references` 已作为 `reference` support evidence 窄接入目标源码之后，默认最多 2 条、跳过 generated tests，预算截断会标 partial。LSP missing/error 会写 retrieval trace 与 fallback risk note，并继续降级。
- Report quality 的 target mapping 已按 case 目标匹配对应 source snippet，不再默认拿第一个 snippet 冒充证据。
- Contract Guard 的请求字段证据已按当前 plan item 收窄：route/function 只使用目标 handler/函数签名引用到的 Pydantic model 字段，避免无关 model 字段给当前请求体背书。
- Reflection prompt 已包含失败 nodeid、异常类型、message、file:line、traceback 摘要；ReAct Reflection 前会把 `RunPytestOutput.failures / collection_errors` 整理成 `failure_context / pytest_result` evidence，进入 Context Builder 的排序、预算和 retrieval trace；`PlanInput.evaluation_events` 可把稳定 `EvaluationEvent` 一并送入 Reflection 前 source context，形成 `failure_context / evaluation_event` evidence；experiment runner 已补最小事件回灌编排，会把同一 `eval_task_id + strategy_version_id` 的历史 prompt-safe 事件喂给后续 repeat，不跨任务/策略污染。

## 4. 二人分工

不要按 Direct / Plan / ReAct 拆，也不要按 prompt / 代码拆。策略不是模块边界，prompt 不是系统状态机。V2.1 最稳的拆法是：A 管评测尺子，B 管项目理解。

### 4.1 A：评测尺子 / Runner 状态机

目标：保证“结果到底能不能信”。

负责范围：

- Flaky check：final test set 在 clean snapshot 连跑 N 次，nodeid 和结果稳定才进入 variant replay。
- Repeat 捕获比例：capture matrix 提供 `captured_count / repeats` 证据。
- Probe / mutation 口径继续加固：当前 demo probe 已接入，后续接 auto mutant 分层。
- Metrics / DB 聚合 / artifact replay 审计。
- final attempt、InMemoryRecorder、DB recorder 协议统一。
- 真实评测脚本、comparison 输出、文档证据更新。

边界：

- 不靠 prompt 调参修指标。
- 不改测试生成策略的业务语义。
- 所有指标必须能从 DB/artifact 重算。
- invalid / flaky / provider failure 不能被压成 0 捕获。

### 4.2 B：项目理解 / Contract Guard / Context Builder

目标：保证“模型拿到的证据够不够写强测试”。

负责范围：

- 多源 retrieval trace：`rg / ast-grep / LSP / framework scanner` 的结构化证据。
- Context Builder：目标源码、Pydantic schema、fixture、existing tests、failure context 的排序和预算。
- Contract Guard 更深业务 oracle：继续拦源码无证据的异常/边界假设。
- route / model / fixture 证据链增强。
- Reflection 输入质量和修复约束。
- Generate prompt 只表达系统约束，不把 prompt 当核心修复。

边界：

- 不改 mutation score 口径。
- 不改 experiment metrics 聚合。
- 生成测试过不了 Guard 就系统拒绝，不靠“下次模型注意”。

### 4.3 协作接口

- B 输出的 test case metadata / target mapping 必须满足 A 的 replay 和 metrics 需要。
- A 的 flaky / invalid / probe 结果要先约定稳定事件契约，再由 B adapter 转成 failure context；不要直接依赖 A 内部 metrics payload。

## 5. 核心口径

### 5.1 Experiment 与 LLM

- Experiment 是评测执行单元。
- `llm_override` 只是覆盖层，不含 secret。
- 最终 provider/model/temperature/max_output_tokens 在每个 clean run 的 `strategy_snapshot.resolved_llm` 冻结。
- 实际调用 LLM 的参数必须与 `strategy_snapshot` 同源。
- provider 为 `mock` 才走 MockLLM；真实 provider 走 provider factory。

### 5.2 Clean Run

- Clean run 只在 clean snapshot 上生成和定型测试集。
- Reflection 只能修测试，不能改业务源码。
- final attempt 生成最终测试集 artifact。
- clean replay assertion failure 属于 false positive。
- clean replay collection/import error 或缺目标源码属于 `invalid_test_set`。

### 5.3 Replay Run

- Replay run 只执行冻结测试集，不规划、不生成、不调用 LLM。
- replay 前必须读取 artifact 并校验 hash。
- variant patch 必须从 clean snapshot 唯一应用；命中 0 次或多次都失败。
- replay 结果必须能映射到原始 pytest nodeid。

### 5.4 Captured Bug

默认 V2.1 口径：

```text
clean replay passed nodeid
AND same nodeid has assertion failure on variant replay
=> captured_bug = true
```

只从 `experiment_replay_runs.captured_bug` 聚合捕获率。

### 5.5 Source Context

- seeded bug experiment 中，目标必须能解析到真实 source path 和源码片段。
- 缺目标源码是 blocking invalid，不允许靠 prompt 猜。
- 模型字段、fixture、existing tests、failure context 是辅助证据，不能掩盖目标源码缺失。
- 源码片段上下文只服务测试生成，不允许 Agent 修改业务源码。
