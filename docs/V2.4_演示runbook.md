# TRACE V2.4 演示 runbook

更新日期：2026-06-21

本文是 V2.4 最终演示的复现手册。目标是让操作者不用查数据库、不手找隐藏路由，也能从 UI 走完：

```text
Dataset -> Experiment -> Start -> Progress -> Metrics / Evidence -> Report / Export
```

默认使用 API 数据源和 MockLLM 复现。MockLLM 用于稳定验证系统链路；真实 LLM 只用于额外演示 provider/model 效果，不能和 MockLLM 指标混在一起解释。

## 1. 一次性环境准备

### 1.1 依赖前提

1. Python 环境可运行后端依赖。本机通常使用：

   ```powershell
   conda activate trace
   ```

2. Node.js 与前端依赖已安装。首次运行前执行：

   ```powershell
   cd D:\17524\MyCode\Trace\frontend
   npm install
   ```

3. 仓库根目录存在 `.env.local`，至少包含：

   ```text
   TRACE_DB_URL=postgresql+psycopg://postgres:postgres@127.0.0.1:5432/trace_test
   TRACE_REDIS_URL=redis://127.0.0.1:6379/0
   TRACE_API_HOST=127.0.0.1
   TRACE_API_PORT=8000
   ```

4. PostgreSQL 和 Redis 必须可用。推荐方式：

   ```powershell
   cd D:\17524\MyCode\Trace
   docker compose up -d
   ```

   如果 Docker Desktop 没启动，Docker executor smoke 会显示 unavailable；这不影响 local_subprocess 演示，也不能算 0 捕获。

### 1.2 初始化数据库和 demo benchmark

在后端目录执行：

```powershell
cd D:\17524\MyCode\Trace\backend
conda activate trace
python scripts/init_db.py
python scripts/seed_strategies.py
python scripts/seed_eval_demo.py
```

期望结果：

- `init_db.py` 输出 `running alembic upgrade head` 或 `schema already exists`。
- `seed_strategies.py` 输出 `strategy seeds ready`。
- `seed_eval_demo.py` 输出 JSON，里面包含：
  - `dataset_id`: `dataset-demo-v2`
  - `task_id`: `task-demo-shop-pricing-v2`
  - `project_id`: `project-demo-shop-v2`
  - `seeded_bug_count`: `6`

## 2. 前端基础设置

1. 打开 `http://127.0.0.1:5186`。
2. 右上角数据源切到 `API`。
3. 语言可选中文或英文；本 runbook 按中文 UI 描述。
4. 如果页面报 API 错误，先确认 API 终端还在运行，且前端代理目标是 `http://127.0.0.1:8000`。

## 3. Dataset 页面

### 3.1 选择已有 demo benchmark

1. 顶部导航点击“数据集”。
2. API 模式下会进入 `#/datasets`，页面读取 `GET /api/v1/eval-datasets`。
3. 找到 `TRACE demo benchmark` 或 ID 为 `dataset-demo-v2` 的卡片。
4. 点击卡片右侧箭头进入 Dataset detail：`#/datasets/dataset-demo-v2`。

期望看到：

- 顶部显示 Dataset 名称、版本、ID。
- metadata 中 snapshot 数量不为 0。
- readiness 显示结构完整或列出明确缺口。
- 左侧 task 列表至少有 `task-demo-shop-pricing-v2`。
- 右侧能看到 seeded bug、variant、confirmation 状态。

### 3.2 Demo 数据源路径

如果右上角切到 `Demo`，点击“数据集”也会显示固定 demo benchmark。这个路径只用于前端静态展示，不代表数据库、pytest、worker 或真实 API 状态。最终验收必须使用 API 模式。

### 3.3 API 中没有 Dataset 时

如果 `#/datasets` 显示“暂无 Dataset”：

1. 先回到 1.2 执行 `python scripts/seed_eval_demo.py`。
2. 刷新 Dataset 页面。
3. 如果仍为空，检查 API 连接的 `TRACE_DB_URL` 是否和 seed 命令一致。

也可以点击“创建数据集”进入 `#/datasets/new`，但这只创建 Dataset metadata 和 snapshot IDs；task、seeded bug、variant/probe 仍需要在详情页 authoring 区补齐。最终演示建议优先使用 seeded demo。

### 3.4 讲解口径

- readiness 是结构检查，不是 replay 结果。
- ordinary variant 不进入 sampled mutation score 分母。
- 只有 `ground_truth.source=auto_mutation` 且 `probe.probe_check.status=passed` 的 variant 才进入 sampled mutation score 分母。

## 4. Experiment 创建

### 4.1 从 Dataset 进入创建页

1. 在 Dataset detail 页面右上方点击“创建实验”。
2. 页面进入 `#/experiments?dataset=dataset-demo-v2`。
3. 创建面板里的 Dataset 应自动预选 `dataset-demo-v2`。

如果没有自动预选：

- 确认 URL query 里有 `dataset=dataset-demo-v2`。
- 确认 `GET /api/v1/eval-datasets` 返回了该 Dataset。
- 手动从 Dataset 下拉框选择 `dataset-demo-v2`。

### 4.2 填写创建表单

推荐填写：

- 实验名：`V2.4 demo benchmark`
- Dataset：`dataset-demo-v2`
- Runtime profile：默认 `Default Local Subprocess`
- Strategy versions：勾选 `sv-direct-v1`、`sv-plan-v1`、`sv-react-v1`
- Repeat count：`1`
- LLM option：`MockLLM / mock-1`

页面会自动读取：

- Dataset options：`GET /api/v1/eval-datasets`
- Strategy versions：`GET /api/v1/strategies` 和各 strategy version API
- Runtime profiles：`GET /api/v1/eval-datasets/{dataset_id}/runtime-profiles`
- LLM options：`GET /api/v1/llm-options`

### 4.3 创建按钮不可用时

按页面 checklist 排查：

- 缺 Dataset：返回 Dataset 页面 seed 或选择已有 Dataset。
- 缺 Runtime profile：确认 Dataset 至少有 task；首次访问会自动创建默认 local profile。
- 缺 Strategy：确认已执行 `python scripts/seed_strategies.py`。
- 缺 LLM option：确认 `GET /api/v1/llm-options` 中 `mock` 为 selectable。
- Repeat count 小于 1：改成 `1`。

点击“创建实验”后，页面应跳转到 `#/experiments/<experiment_id>`，状态为 `draft`。

## 5. Start 与 Progress

1. 在 Experiment detail 中确认状态为 `draft`。
2. 点击 `Start`。
3. 状态应变为 `queued`，随后进入 `running`。
4. 页面 progress 区会刷新：
   - strategy 当前进度。
   - run stage。
   - latest trace step。
   - current task/repeat。
   - clean/replay completed counts。
5. 等待状态进入 `completed`。

期望结果：

- `draft` 才显示 Start。
- `queued/running` 不重复显示 Start。
- `completed/failed/cancelled` 不支持原地重跑。
- 使用 MockLLM 和 demo benchmark 时，三策略通常在数分钟内完成。

如果一直 queued：

- 确认 Worker 终端仍在运行。
- 确认 Redis 可用。
- 查看 Worker 日志是否连接 `trace_test_runs` 队列。

如果 failed：

- 打开 EvaluationEvent 和 progress 详情，区分 provider/setup/executor/artifact/runtime failure。
- 这些系统失败不能解释为 0 捕获。

## 6. Metrics / Evidence

Experiment completed 后，在详情页依次检查：

1. Metrics table：
   - 应有三行，对应 Direct / Plan / ReAct。
   - 查看 capture rate、false positive、metric status、cost per bug。
2. Capture matrix：
   - 查看每个 bug variant 是否被各策略捕获。
3. EvaluationEvent timeline：
   - 查看 invalid、flaky、probe、provider/setup/executor/artifact/runtime 事件。
4. Clean run table：
   - 查看 clean baseline 是否通过。
   - false positive 必须和 variant capture 分开讲。
5. Replay evidence：
   - 确认 replay 不调用 LLM，`llm_calls=0`。
   - 查看 cache hit/stale/miss。
   - 查看 runtime snapshot、executor metadata、setup/replay error。
6. Runtime execution JSON：
   - 查看 executor kind distribution。
   - 查看 runtime profiles。
   - 查看 replay cache counts、setup status counts、retries、artifact inventory。

关键解释：

- `invalid_test_set` 不是 0 捕获。
- provider/setup/executor/artifact/flaky/runtime failure 不是 0 捕获。
- false positive 是 clean baseline 失败，必须和 variant capture 分开。

## 7. Runtime / Preflight

1. 顶部导航点击“运行时”。
2. 确认 executor status：
   - `local_subprocess` 应 available。
   - `docker` 如果 Docker Desktop 未启动，会显示 unavailable reason。
3. 选择或查看默认 local runtime profile。
4. 点击 preflight。
5. 检查 preflight panel：
   - checks 数量。
   - status：passed/warning/failed。
   - replay policy。
   - audit-only limitations。

讲解口径：

- `local_subprocess` 是本地进程执行，不是安全沙箱。
- Docker executor 才是容器隔离路径，但依赖宿主 Docker CLI/daemon/image。
- Docker 不可用是 runtime 状态，不进入捕获率分母。

## 8. Cleanup / Replay Cache Cleanup

在 Experiment detail 的 lifecycle / cleanup 区：

1. 点击 `Preview workspace cleanup`。
2. 页面出现 workspace cleanup candidates 表格。
3. 检查候选字段：
   - replay ID。
   - status。
   - exists。
   - path。
4. 如需真正删除，再点击 `Delete workspace candidates`。
5. 浏览器会弹出二次确认；确认后才执行删除。
6. Replay cache cleanup 同理：
   - 先点击 `Preview replay cache cleanup`。
   - 检查 replay cache cleanup candidates。
   - 再点击 `Delete cache candidates`。

安全边界：

- 默认 `keep_failed=true`，失败证据不会被普通 workspace cleanup 删除。
- replay cache cleanup 由后端保护 source replay evidence / reused source workspace。
- cleanup 只影响 workspace/cache 文件，不重写历史 metrics。

## 9. Report / Export

在 Experiment detail 的 `REPORT / EXPORT` 区：

1. 点击 `Download metrics JSON`。
2. 浏览器会下载 `<experiment_id>.metrics.json`。
3. 点击 `Open first run report`。
4. 页面跳转到对应 run detail，查看 run-level report。
5. 点击 Dataset 回指，确认能回到原 Dataset。
6. 展开 Runtime execution JSON，检查 metrics 证据不需要查数据库。

V2.4 最小闭环是：experiment metrics JSON + run-level report + Dataset 回指。若后续需要 experiment-level Markdown report，应作为 V3 或增强项单独实现。

## 10. Comparison

1. 顶部导航点击“对比”。
2. API 模式只展示 completed experiment metrics。
3. 如果有多个 completed experiment，用选择器切换。
4. 如果没有 completed experiment，按页面提示回到 Dataset / Experiments 创建并启动。
5. 如果 API 失败，页面显示错误，不会回退 demo metrics。

验收口径：

- Comparison 展示的是单 experiment 内策略对比。
- Demo fallback 不能作为真实 benchmark 证据。

## 11. 最终证据记录

一次验收至少记录：

- Dataset ID / version。
- Experiment ID。
- Strategy version IDs。
- LLM option。
- Runtime profile。
- completed 状态截图或 API 响应。
- Metrics JSON。
- Runtime execution JSON。
- cleanup dry-run candidates 截图或响应。
- Docker preflight / smoke 结果；若 Docker 不可用，记录 unavailable reason。
