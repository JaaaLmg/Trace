# TRACE V2.4 演示 runbook

更新日期：2026-06-22

本文是 V2.4 最终展示的复现手册。目标是让操作者不用查数据库、不手找隐藏路由，也能严格从 UI 走完：

```text
启动环境 -> Dataset -> Experiment -> Start -> Progress -> Metrics / Evidence -> Report / Export -> Comparison
```

默认演示使用 API 数据源和 MockLLM。MockLLM 用于稳定验证系统链路；真实 LLM 只用于额外展示 provider/model 配置效果，不能和 MockLLM 指标混在一起解释。

当前 demo benchmark 已扩容为：

- Dataset：`dataset-demo-v2`
- Task：3 个，覆盖 shop pricing、inventory stock、accounts policy
- Seeded bug variants：16 个
- Strategy：`sv-direct-v1`、`sv-plan-v1`、`sv-react-v1`

## 1. 演示路线选择

V2.4 支持两种启动路线。二选一即可，后续 UI 展示流程相同。

| 路线 | 适用场景 | 启动方式 | 默认执行器 |
| --- | --- | --- | --- |
| 本地开发分步启动 | 开发、调试、看后端/前端日志 | DB/Redis 用 Docker，API/Worker/Frontend 本机启动 | `local_subprocess` |
| Docker 一键启动 | 交付展示、验收、最少手工步骤 | `docker compose up --build` 启动完整产品栈 | `local_subprocess`，运行在 worker 容器内 |

注意：Docker Compose 一键启动是“TRACE 产品栈容器化运行”，不是 Docker executor 验收。默认 runtime profile 仍是 `Default Local Subprocess`。它在本机或 worker 容器内启动 pytest 子进程，不等于 Docker 里再启动 Docker。

## 2. 路线 A：本地开发分步启动

### 2.1 前提

1. 安装并启动 Docker Desktop，PostgreSQL 和 Redis 会通过 compose 启动。
2. Python 环境可运行后端依赖。本文示例使用：

   ```powershell
   conda activate trace
   ```

3. Node.js 18+ 可用。
4. 设置仓库根目录变量。按实际路径修改：

   ```powershell
   $TRACE_ROOT = "D:\17524\MyCode\Trace"
   ```

5. 首次运行安装依赖：

   ```powershell
   cd $TRACE_ROOT
   conda activate trace
   pip install -r backend/requirements.txt

   cd $TRACE_ROOT\frontend
   npm install
   ```

6. 准备本地环境变量：

   ```powershell
   cd $TRACE_ROOT
   if (-not (Test-Path .env.local)) { Copy-Item .env.example .env.local }
   ```

   `.env.local` 至少包含：

   ```text
   TRACE_DB_URL=postgresql+psycopg://postgres:postgres@127.0.0.1:5432/trace_test
   TRACE_REDIS_URL=redis://127.0.0.1:6379/0
   TRACE_API_HOST=127.0.0.1
   TRACE_API_PORT=8000
   TRACE_LLM_PROVIDER=mock
   TRACE_LLM_MODEL=mock-1
   ```

### 2.2 启动 PostgreSQL 和 Redis

本地开发只启动基础服务，不启动 API/Worker/Frontend 容器：

```powershell
cd $TRACE_ROOT
docker compose up -d postgres redis
```

检查服务：

```powershell
docker compose ps postgres redis
```

### 2.3 初始化数据库和 demo benchmark

```powershell
cd $TRACE_ROOT\backend
conda activate trace
python scripts/init_db.py
python scripts/seed_strategies.py
python scripts/seed_eval_demo.py
```

期望 `seed_eval_demo.py` 输出包含：

```text
dataset_id: dataset-demo-v2
task_count: 3
seeded_bug_count: 16
project_id: project-demo-shop-v2
snapshot_id: snapshot-demo-shop-clean-v2
```

如果重复执行，脚本应保持可重复 seed，不需要手动清库。

### 2.4 启动 API、Worker、Frontend

需要三个终端。

终端 1，启动 API：

```powershell
cd $TRACE_ROOT\backend
conda activate trace
python scripts/run_api.py
```

期望地址：`http://127.0.0.1:8000`

终端 2，启动 Worker：

```powershell
cd $TRACE_ROOT\backend
conda activate trace
python scripts/run_worker.py
```

Windows 下 worker 默认使用 Celery `solo` pool，适合本地演示和开发。

终端 3，启动前端：

```powershell
cd $TRACE_ROOT\frontend
npm run dev
```

期望地址：`http://127.0.0.1:5186`

### 2.5 本地启动后检查

```powershell
Invoke-RestMethod http://127.0.0.1:8000/healthz
Invoke-RestMethod http://127.0.0.1:8000/api/v1/llm-options
Invoke-RestMethod http://127.0.0.1:8000/api/v1/eval-datasets
Invoke-RestMethod http://127.0.0.1:8000/api/v1/runtime-profiles/executors/status
```

期望结果：

- `/healthz` 返回 `ok=true`。
- `/api/v1/llm-options` 中 `mock` 为 `selectable=true`。
- `/api/v1/eval-datasets` 包含 `dataset-demo-v2`。
- `local_subprocess` 可用；`docker` 可用与否取决于宿主 Docker CLI/daemon，不影响默认演示。

## 3. 路线 B：Docker Compose 一键启动

### 3.1 前提

1. 已安装 Docker Desktop。
2. Docker Desktop 已启动，并切到 Linux containers。
3. 本机端口 `5186`、`8000`、`5432`、`6379` 未被占用。
4. `docker info` 能看到 Server 信息。

可先检查：

```powershell

docker compose config
```

### 3.2 一键启动完整产品栈

在仓库根目录执行：

```powershell

```

首次启动会自动完成：

1. 构建后端镜像和前端镜像。
2. 启动 PostgreSQL / Redis。
3. API 等待 DB/Redis ready。
4. API 容器执行 `scripts/init_db.py`、`scripts/seed_strategies.py`、`scripts/seed_eval_demo.py`。
5. 启动 API、Worker 和 nginx 前端。

访问地址：

```text
Frontend: http://127.0.0.1:5186
API:      http://127.0.0.1:8000
```

如果希望后台运行：

```powershell
docker compose up --build -d
docker compose ps
```

查看日志：

```powershell

docker compose logs -f worker
```

### 3.3 Compose smoke 检查

基础 smoke：

```powershell

powershell -ExecutionPolicy Bypass -File scripts/compose_smoke.ps1
```

期望输出：

```text

```

### 3.4 Compose 三策略 benchmark

如果要生成正式展示证据，建议 repeat=3：

```powershell

  -ExperimentId $id `
```

期望：

- status 为 `completed`。
- `task_count=3`。
- `bug_variant_count=16`。

- `replay_llm_calls=[0]`。
- repeat=3 时 expected clean runs 为 `3 tasks * 3 strategies * 3 repeats = 27`。
- repeat=3 时 expected variant replay records 为 `16 variants * 3 strategies * 3 repeats = 144`。

```text

```


## 4. 真实 LLM 可选配置

默认演示使用 MockLLM，不需要密钥。

### 4.1 本地开发配置真实 LLM

本地直接运行后端时，可以使用 shell 环境变量或 `backend/llm.config.json`。推荐临时环境变量：

```powershell
$env:TRACE_LLM_PROVIDER="openai_chat_compat"
$env:TRACE_LLM_MODEL="your-model"
$env:TRACE_LLM_API_KEY="your-api-key"
$env:TRACE_LLM_BASE_URL="https://example.com/v1"
```

重新启动 API 和 Worker 后检查：

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/v1/llm-options
```

真实 LLM option 只有后端配置完整且有 API key 时才会 `selectable=true`。

### 4.2 Compose 配置真实 LLM

Compose 默认读取仓库根目录 `.env`。不要把真实密钥写入 `.env.example`。

```powershell
cd D:\17524\MyCode\Trace
if (-not (Test-Path .env)) { Copy-Item .env.example .env }
```

在 `.env` 中填写：

```env
TRACE_LLM_PROVIDER=openai_chat_compat
TRACE_LLM_MODEL=your-model
TRACE_LLM_API_KEY=your-api-key
TRACE_LLM_BASE_URL=https://example.com/v1
```

让 API/Worker 重新读取：

```powershell
docker compose up -d --force-recreate api worker
```

展示口径：

- MockLLM 是确定性链路验证。
- 真实 LLM 是额外模型效果展示。
- 两者指标不能混合解释，真实 LLM 的成本、稳定性和捕获率以实际完成的 experiment 为准。

## 6. Dataset 展示流程

### 6.1 进入 demo benchmark

1. 顶部导航点击“数据集”。
2. API 模式下进入 `#/datasets`，页面读取 `GET /api/v1/eval-datasets`。
3. 找到 `TRACE demo seeded bugs` 或 ID 为 `dataset-demo-v2` 的卡片。
4. 点击进入 Dataset detail：`#/datasets/dataset-demo-v2`。

期望看到：

- Dataset ID：`dataset-demo-v2`。
- Dataset version：`v2`。
- metadata 中 snapshot 数量不为 0。
- readiness 为 ready 或列出明确缺口。
- Task 列表有 3 个：
  - `task-demo-shop-pricing-v2`
  - `task-demo-inventory-stock-v1`
  - `task-demo-accounts-policy-v1`
- seeded bug / variant 总量为 16 个。

### 6.2 Dataset 讲解口径

- Dataset 是 benchmark 分母来源，包含 task、seeded bug、variant、readiness 和 confirmation 状态。
- Dataset readiness 是结构检查，不是 replay 结果，也不是捕获率。
- 当前 seeded demo 的 variants 来自 canonical patch，seed 前会校验 patch 唯一命中。
- ordinary variant 不进入 sampled mutation score 分母。
- 只有 `ground_truth.source=auto_mutation` 且 probe passed 的 confirmed mutation variant 才进入 sampled mutation score 分母。
- Demo authoring preset 只减少手填，不等于自动完成真实 Dataset 生产。

### 6.3 API 中没有 Dataset 时

本地开发路线：

```powershell
cd $TRACE_ROOT\backend
conda activate trace
python scripts/seed_eval_demo.py
```

Compose 路线：

```powershell
docker compose up -d --force-recreate api
```

如果仍为空，检查 API 连接的数据库是否和 seed 命令一致。本地开发常见问题是 `.env.local` 中 `TRACE_DB_URL` 与当前 PostgreSQL 实例不一致。

## 7. Runtime / Preflight 展示流程

1. 顶部导航点击“运行时”。
2. 查看 executor status：
   - `local_subprocess` 应 available。
   - `docker` 是否 available 取决于当前环境。
3. 选择或查看 `Default Local Subprocess` runtime profile。
4. 点击 preflight。
5. 检查 preflight panel：
   - checks 列表。
   - status：`passed` / `warning` / `failed`。
   - replay policy。
   - audit-only limitations。

讲解口径：

- `local_subprocess` 是进程级执行，不是安全沙箱。
- Docker Compose 启动 TRACE 产品栈时，默认 pytest 仍由 worker 容器内的 local subprocess 执行。
- Docker executor 是可选 runtime profile 能力，需要 Docker CLI/daemon/image；当前 compose 交付路径不默认启用 Docker executor，也没有把 Docker socket 挂给 worker。
- Docker 不可用是 runtime 状态，不进入捕获率分母。

## 8. Experiment 创建

### 8.1 从 Dataset 进入创建页

1. 在 Dataset detail 页面点击“创建实验”。
2. 页面进入 `#/experiments?dataset=dataset-demo-v2`。
3. 创建面板里的 Dataset 应自动预选 `dataset-demo-v2`。

如果没有自动预选：

- 确认 URL query 中有 `dataset=dataset-demo-v2`。
- 确认 `GET /api/v1/eval-datasets` 返回该 Dataset。
- 手动从 Dataset 下拉框选择 `dataset-demo-v2`。

### 8.2 填写创建表单

快速 UI 展示推荐：

- 实验名：`V2.4 demo benchmark`
- Dataset：`dataset-demo-v2`
- Runtime profile：`Default Local Subprocess`
- Strategy versions：勾选 `sv-direct-v1`、`sv-plan-v1`、`sv-react-v1`
- Repeat count：`1`
- LLM option：`MockLLM / mock-1`

正式验收或稳定 benchmark 证据推荐：

- Strategy versions：同上三策略
- Repeat count：`3`
- LLM option：`MockLLM / mock-1`
- 或直接使用 `scripts/compose_benchmark.ps1 -RepeatCount 3` 生成 JSON 证据

页面会自动读取：

- Dataset options：`GET /api/v1/eval-datasets`
- Runtime profiles：`GET /api/v1/eval-datasets/{dataset_id}/runtime-profiles`
- Strategy versions：`GET /api/v1/strategies` 和 strategy version API
- LLM options：`GET /api/v1/llm-options`

### 8.3 创建按钮不可用时

按页面 checklist 排查：

- 缺 Dataset：返回 Dataset 页面 seed 或选择已有 Dataset。
- 缺 Runtime profile：确认 Dataset 至少有 task；首次访问会自动创建默认 local profile。
- 缺 Strategy：确认已执行 `python scripts/seed_strategies.py` 或 API bootstrap 已完成。
- 缺 LLM option：确认 `GET /api/v1/llm-options` 中 `mock` 为 selectable。
- Repeat count 小于 1：改成 `1` 或 `3`。

点击“创建实验”后，页面应跳转到 `#/experiments/<experiment_id>`，状态为 `draft`。

## 9. Start 与 Progress

1. 在 Experiment detail 中确认状态为 `draft`。
2. 点击 `Start`。
3. 状态应变为 `queued`，随后进入 `running`。
4. 页面 progress 区会刷新：
   - strategy 当前进度。
   - run stage。
   - latest trace step。
   - current task / repeat。
   - clean / replay completed counts。
5. 等待状态进入 `completed`。

期望：

- `draft` 才显示 Start。
- `queued/running` 不重复显示 Start。
- `completed/failed/cancelled` 不支持原地重跑；如需重跑，新建 experiment。
- 三策略、3 tasks、16 variants、repeat=1 通常在数分钟内完成。
- repeat=3 会更慢，适合正式证据而不是现场临时等待。

如果一直 queued：

- 本地路线：确认 Worker 终端仍在运行，Redis 可用。
- Compose 路线：执行 `docker compose ps worker` 和 `docker compose logs -f worker`。
- 查看 Worker 日志是否连接 `trace_test_runs` 队列。

如果 failed：

- 打开 EvaluationEvent 和 progress 详情。
- 区分 provider/setup/executor/artifact/runtime failure。
- 这些系统失败不能解释为 0 捕获。

## 10. Metrics / Evidence

Experiment completed 后，在详情页依次检查：

1. Metrics table：
   - 应有三行，对应 Direct / Plan / ReAct。
   - 查看 capture rate、false positive、metric status、cost per bug。
2. Capture matrix：
   - 查看 16 个 bug variant 是否被各策略捕获。
3. EvaluationEvent timeline：
   - 查看 invalid、flaky、probe、provider/setup/executor/artifact/runtime 事件。
4. Clean run table：
   - 查看 clean baseline 是否通过。
   - false positive 必须和 variant capture 分开讲。
5. Replay evidence：
   - replay 不调用 LLM，确认 `llm_calls=0`。
   - 查看 cache hit/stale/miss。
   - 查看 runtime snapshot、executor metadata、setup/replay error。
6. Runtime execution JSON：
   - 查看 executor kind distribution。
   - 查看 runtime profiles。
   - 查看 replay cache counts、setup status counts、retries、artifact inventory。

关键解释：

- `metric_status=ok` 表示该行可评价。
- `invalid_test_set` 不是 0 捕获。
- provider/setup/executor/artifact/flaky/runtime failure 不是 0 捕获。
- false positive 是 clean baseline 失败，必须和 variant capture 分开。
- replay 只执行冻结测试集，不重新规划、不重新生成、不调用 LLM。

参考 benchmark 结果口径，来自 `docs/evidence/compose-benchmark-suite-r3-20260622112519.json`：

| Strategy | metric_status | capture_rate_mean | false_positive_rate |
| --- | --- | --- | --- |
| Direct v1 | `ok` | 0.875 | 1.0 |
| Plan-and-Execute v1 | `ok` | 1.0 | 0.0 |
| ReAct+Reflection v1 | `ok` | 1.0 | 0.0 |

这只是已验证证据，不要求每次真实 LLM 演示都复现相同数值。

## 11. Cleanup / Replay Cache Cleanup

在 Experiment detail 的 lifecycle / cleanup 区：

1. 点击 `Preview workspace cleanup`。
2. 页面出现 workspace cleanup candidates 表格。
3. 检查候选字段：
   - replay ID。
   - status。
   - exists。
   - path。
4. 如需真正删除，再点击 `Delete workspace candidates`。
5. 浏览器会弹出二次确认，确认后才执行删除。
6. Replay cache cleanup 同理：
   - 先点击 `Preview replay cache cleanup`。
   - 检查 replay cache cleanup candidates。
   - 再点击 `Delete cache candidates`。

安全边界：

- 默认 `keep_failed=true`，失败证据不会被普通 workspace cleanup 删除。
- replay cache cleanup 由后端保护 source replay evidence / reused source workspace。
- cleanup 只影响 workspace/cache 文件，不重写历史 metrics、DB 记录或 report。

现场演示建议只做 preview，除非明确需要展示二次确认和删除结果。

## 12. Report / Export

在 Experiment detail 的 `REPORT / EXPORT` 区：

1. 点击 `Download metrics JSON`。
2. 浏览器会下载 `<experiment_id>.metrics.json`。
3. 点击 `Open first run report`。
4. 页面跳转到对应 run detail，查看 run-level report。
5. 点击 Dataset 回指，确认能回到 `#/datasets/dataset-demo-v2`。
6. 展开 Runtime execution JSON，确认 metrics 证据不需要查数据库。

V2.4 最小 report/export 闭环是：

```text
experiment metrics JSON + run-level report + Dataset 回指 + Runtime execution JSON
```

experiment-level Markdown report 不属于 V2.4 验收承诺。

## 13. Comparison

1. 顶部导航点击“对比”。
2. API 模式只展示 completed experiment metrics。
3. 如果有多个 completed experiment，用选择器切换。
4. 如果没有 completed experiment，按页面提示回到 Dataset / Experiments 创建并启动。
5. 如果 API 失败，页面显示错误，不回退 demo metrics。

验收口径：

- Comparison 展示的是单 experiment 内策略对比。
- Demo fallback 不能作为真实 benchmark 证据。
- 最终截图和报告必须来自 API completed experiment。

## 16. 常见问题

### 16.1 页面没有 API 数据

- 确认右上角数据源是 `API`。
- 确认 API 地址 `http://127.0.0.1:8000/healthz` 可用。
- 本地路线确认 `.env.local` 的 DB/Redis 地址正确。
- Compose 路线确认 `docker compose ps` 中 API healthy。

### 16.2 Dataset 不存在

- 本地路线执行 `python scripts/seed_eval_demo.py`。
- Compose 路线检查 API bootstrap 日志，或重建 API 容器。
- 确认连接的是同一个数据库。

### 16.3 Experiment 一直 queued

- Worker 未启动或无法连 Redis。
- 本地路线看 Worker 终端。
- Compose 路线看 `docker compose logs -f worker`。

### 16.4 Docker executor 显示 unavailable

这不影响默认 V2.4 演示。当前默认执行器是 `local_subprocess`。Docker executor 是 opt-in runtime profile 能力；Docker Compose 交付路径没有默认挂载 Docker socket，也不承诺 Docker executor 验收。

### 16.5 Direct false positive 为 1.0 怎么讲

这是 MockLLM direct 策略在参考 benchmark 中产生的 clean baseline 失败证据。必须和 variant capture 分开解释，不能抹成 0，也不能和 Plan/ReAct 的结果混讲。

## 18. V2.4 边界

以下不属于 V2.4 验收承诺：

- 远程 Runner、register、heartbeat、long-poll。
- 分布式调度与跨机器 replay。
- Docker executor 默认交付。
- MinIO/S3 artifact storage。
- 完整 OpenTelemetry。
- experiment-level Markdown 报告生成服务。
- failed/cancelled experiment 的原地 rerun；后续应设计 clone/new experiment 合同。
