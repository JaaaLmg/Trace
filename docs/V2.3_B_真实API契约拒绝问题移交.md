# V2.3 B 线真实 API 契约拒绝问题移交

更新日期：2026-06-21

## 1. 问题现象

使用真实 LLM/API 跑实验时，实验详情或 run 详情中经常出现：

- `生成测试契约重试`
- `生成测试（契约重试）`
- `生成测试契约拒绝`
- 随后 run 级状态显示 `运行失败`

截图中的典型链路是：模型已生成测试内容，但 `Contract Guard` 检查失败；系统给模型一次结构化重试机会，重试后仍违反生成契约，于是抛出 `PIPELINE_REJECT`。

## 2. 对照 V2 实施清单的归属判断

结论：该问题本体属于 B 线范围，不是 A 线 runtime / replay / cache / executor 问题。

依据 `docs/V2实施清单.md`：

- A 线责任是“评测尺子、执行平台和规模化运行”，包括 executor、runtime profile、experiment runner、replay、metrics、artifact/cache/cleanup。
- B 线责任是“项目理解、数据集和生成约束”，包括 source context、retrieval trace、Contract Guard、prompt-safe 回灌、generated test metadata 和 target mapping。
- 清单明确写到：`Contract Guard` 继续围绕源码证据补窄规则；证据不足时必须拒绝、降级或标记 incomplete；不能靠 prompt 自律替代 guard。

因此：

- “真实 LLM 生成的测试被 Contract Guard 拒绝”是 B 线生成约束与项目理解质量问题。
- “拒绝后不能算作 0 捕获，必须进入 invalid test set 口径”是 A 线评测口径责任。

## 3. 当前代码链路

相关代码位置：

- `backend/app/agents/contract_guard.py`
  - `check_generation_contract(...)` 负责生成测试契约检查。
- `backend/app/agents/strategies/common.py`
  - `do_generate(...)` 首轮生成后调用 Contract Guard。
  - 若违反契约，会追加 violation feedback 并执行一次生成重试。
  - 重试仍失败时，抛出 `TraceError(ErrorCode.PIPELINE_REJECT, ...)`。
- `backend/app/services/experiments.py`
  - experiment runner 遇到 `run.error_code == "PIPELINE_REJECT"` 时，应该把该 clean unit 记录为 `invalid_test_set`，而不是让 capture rate 变成 0。

实验层当前预期落库口径：

```json
{
  "validity_status": "invalid_test_set",
  "invalid_reason": "pipeline_reject",
  "pipeline_reject_error": "Contract Guard violation details..."
}
```

## 4. A 线应保持的边界

A 线不应绕过、放松或吞掉 Contract Guard。

A 线只需要保证：

- `PIPELINE_REJECT` 不被解释为 replay/runtime/cache/executor failure。
- `PIPELINE_REJECT` 不被压成 0 捕获。
- metrics / evaluation event 能清楚表达该 clean unit 不可评价。
- run 详情可以显示 run 级 failed，但 experiment metrics 必须能区分 `invalid_test_set / pipeline_reject`。

如果整个 experiment 因 `PIPELINE_REJECT` 直接变成 failed，而不是继续其它 task / strategy / repeat，则属于 A 线兜底或错误码传播风险，需要另行修复。

## 5. B 线建议处理方向

建议 B 线优先从“真实 LLM 为什么经常被拒”入手，而不是要求 A 线放宽评测口径。

候选方向：

1. 改善 violation feedback
   - 当前重试只给一次机会。
   - 建议检查 prompt 中是否明确要求只修复 violation、保持 cases 元数据、保持 target binding。
   - 对常见 violation 增加更可操作的修复说明。

2. 增强生成输出结构稳定性
   - 确保 `cases[*].test_name` 与真实 pytest 函数一一对应。
   - 确保 `assertion_summary`、`target_function` / `target_route` 必填且与 plan item 对齐。
   - 避免生成 `test_file_path=None` 这类对写入无用但容易暴露模型漂移的字段。

3. 强化 source context / evidence
   - 真实 API 场景下，模型更容易编造 route、fixture、request field 或 oracle。
   - 若 source context 缺目标源码、缺模型字段、缺 fixture 证据，应明确标记 incomplete 或降级，而不是让模型自由猜。

4. 细分 Contract Guard violation 类型
   - 将高频失败聚类，例如：
     - cases 元数据不匹配真实测试函数。
     - 弱 oracle / 只有 2xx status。
     - request field 缺证据。
     - fixture 缺证据。
     - route method/path 不匹配。
     - target binding 漂移。
   - 每类补一条失败测试，再优化 guard 或 prompt。

5. 引入更可审计的 structured retry
   - 重试输入应只包含必要 violation、上轮测试内容和目标证据摘要。
   - 重试输出仍必须完整通过 Guard。
   - 不建议把 Guard 改成“警告但放行”，否则会污染评测分母。

## 6. B 线验收建议

建议新增或补强以下测试：

- `backend/tests/test_contract_guard.py`
  - 覆盖真实 API 中出现的高频 violation。
- 真实 LLM smoke 或录制 fixture
  - 至少覆盖 direct / plan / react 各一条生成路径。
- experiment runner 集成测试
  - 确认 `PIPELINE_REJECT` 在 experiment metrics 中呈现为 `invalid_test_set / pipeline_reject`。

验收口径：

- Contract Guard 不放松核心红线。
- 常见真实 LLM 输出能通过更明确的证据和 retry 修正。
- 仍无法修正的输出必须被拒绝，并进入 invalid test set，而不是 0 捕获。

## 7. 给 B 线的最终结论

这不是 A 线执行平台跑坏了，而是真实 LLM 输出在 B 线生成约束边界上被正确拦截。

B 线下一步应重点提升 source context、prompt-safe evidence、生成契约提示和 structured retry 的成功率；A 线只需继续保证拒绝结果在实验指标中被解释为不可评价测试集。
## 8. 本轮处理记录

已按 B 线范围处理，不放松 Contract Guard：

- `backend/app/agents/prompts.py` 新增 `generation_retry_fix_hints()`，把高频 Guard violation 转成可执行修复动作。
- `build_generate_retry_messages()` 现在同时回灌 violation 原文、针对性修复动作、上轮完整测试文件和 retry 输出不变量。
- retry 输出不变量要求完整 `GenerationOutput` JSON、完整 pytest 文件、`cases` 与真实测试函数一一对应、不输出 `test_file_path: null`、不切换目标或新增无证据 fixture/request field/oracle。
- `prompt_bundle()` 冻结 retry 输出不变量，保证 strategy snapshot 可审计。
- 已回归确认 `PIPELINE_REJECT` 仍进入 `invalid_test_set / pipeline_reject`，不解释成 0 捕获。
