# TRACE V1 实施清单

## 0. 这份文档是什么

`系统设计.md` 是**最终形态蓝图**（~28 张表、Docker、远程 Runner、完整版本体系）。这份文档是**第一版真正要建的子集和落地决策**。

- 写 V1 代码以**本文档**为准：建哪些表、砍哪些、外键怎么处理、API 取哪几个。
- 字段语义、工具契约、状态机规则仍以 `系统设计.md` 为准。本文档只做减法和拍板，**不改任何字段语义**。
- `初步构想.md` 是旧 6 表草稿，**已废弃**，不作为技术依据，只留作取舍记录和求职材料（分工、排期、简历、参考论文）。

## 1. V1 目标（一句话）

跑通端到端闭环 + Agent trace + 用例级结果 + 最小 seeded bug 评测（脚本）。

**V1 不碰**：Docker、远程/自托管 Runner、快照重放、完整 Prompt/Tool Schema 版本表、评测落库、多 Runner 调度。

### 2.2 延后（不在 V1 建）

- **V2 建**：`runtime_profiles`、`strategies`（父表）、`prompt_versions`、`tool_schema_versions`、`eval_datasets`、`eval_tasks`、`seeded_bugs`、`bug_variants`、`experiments`、`experiment_clean_runs`、`test_replays`、`experiment_replay_runs`。
- **V3 建**：`runners`、`runner_jobs`。

V1 的最小 seeded bug 评测**不落这些表**，走脚本 + 固定目录：`eval/demo/clean`、`eval/demo/variants/*`，指标用脚本算 `seeded_bug_capture_rate` 和 `clean_false_positive_rate`。

### 2.3 三个外键的拍板

`test_runs` 有三个指向「延后/最简」表的外键，V1 这么处理：

| 外键                    | V1 处理                                                                             | 理由                                                                               |
| ----------------------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| `project_snapshot_id` | **建最简 `project_snapshots`，`NOT NULL`**                                | 它是复现根，也是「这次跑的是哪份代码」的锚；就一张轻表；nullable 反而 V2 要回改    |
| `runtime_profile_id`  | **不建表，FK 置 `NULL`**；环境信息写进 `test_runs.runtime_snapshot`(JSON) | V1 单机固定环境，不存在「多套可复用运行配置」需求；V2 有多环境时再把 JSON 提升为表 |
| `strategy_version_id` | **建最简 `strategy_versions`，`NOT NULL`，种 3 条**                       | V1 就要做策略对比，策略身份必须能落库、能 group by                                 |

**V1 `project_snapshots` 最简字段**：`id`、`project_id`、`source_kind`、`root_path`、`content_hash`(可空)、`created_at`。

## 3. V1 策略怎么表示

种 3 条 `strategy_versions`（对应三种策略），不建 `strategies` 父表，不建 prompt/tool schema 版本表。

**V1 `strategy_versions` 字段**：`id`、`name`、`workflow_type`(`direct`/`plan_execute`/`react_reflection`)、`model_provider`、`model_name`、`model_params`、`allow_reflection`、`max_tool_calls`、`prompt_ref`(字符串)、`tool_schema_ref`(字符串)。

- `prompt_ref`/`tool_schema_ref` 是**临时字段**，记录 prompt 和工具 schema 在代码仓库里的版本号（如 `react_v1`）。V2 引入 `prompt_versions`/`tool_schema_versions` 表后，迁移成 `prompt_version_id`/`tool_schema_version_id`。
- **V1 可复现怎么保证**：prompt 正文和工具 schema 放在 git 仓库里（用 commit 兜底）；**同时**创建 `test_runs` 时把解析后的 prompt 正文、模型参数冻结进 `test_runs.strategy_snapshot`(JSON)。这满足「能找回当时正文」的实质要求，只是没用专门的表。**不允许只存 `prompt_ref` 字符串就完事**——那等于退回到上一轮被否掉的「只存版本号」。

## 4. V1 执行模式

按 `系统设计.md` §8 的 V1 口径：

- 独立 subprocess + 独立工作目录；
- 独立 venv，尽力禁网；
- 路径白名单校验（canonical path）；
- 硬超时 + 进程清理。

**明确不做**：Docker、容器加固、seccomp/AppArmor、远程 Runner。

**口径红线**：跑的是 LLM 生成的任意 Python，subprocess + 白名单**不是安全沙箱**。README 和面试都按这个讲，不声称没有的安全性。真正的容器隔离是 V2。

## 6. V1 硬边界（对齐 README / 面试口径）

- 只支持 Python/FastAPI + pytest。
- 只跑 `examples/` 或白名单目录，不接 GitHub App、不接 OpenAPI。
- Reflection 只一次。
- 评测走脚本，不进 API、不落评测表。
- 执行是 subprocess 隔离，不是安全沙箱。
- 前端 V1 优先 trace + report 两页，其余页面有余力再做。

```text
- strategy_versions 用最简字段，含 workflow_type(direct/plan_execute/react_reflection)、allow_reflection、max_tool_calls、

  prompt 正文和模型参数冻结进 test_runs.strategy_snapshot(JSON)。

  test_runs.project_snapshot_id 和 strategy_version_id 为 NOT NULL。

list_files、read_file、analyze_project（AST + FastAPI 路由内省，不把整仓喂给 LLM）、
write_test_file（canonical path 白名单校验，禁写业务源码）、
run_pytest（返回 exit_code/passed/failed/skipped/duration_ms/case_results/failures，失败日志即在 failures 里，不另设 read_test_log）。

```

## 8. 开工顺序建议

1. **第 1-2 天**：建上面 14 张表 + Alembic 迁移；锁工具 I/O 契约和 trace 事件 schema；docker-compose 起 pg/redis；MockLLM 打桩。
2. **第 3-5 天**：工具层真实实现 + 执行器 subprocess + 状态机 + run/steps/report 接口，MockLLM 端到端打通（M1）。
3. **之后**：接真实 LLM、三策略、Reflection 契约落地、脚本评测出捕获率。详见 `初步构想.md` 的两周计划（排期部分仍有效）。

## 9. 三人分工与协作

按「能靠契约解耦的子系统」切，每条线对应一类简历能力。最懂 LLM 的去 A，偏前端的去 C。本项目 A 兼做评测（评测语义和策略强相关，A 写最顺），C 专注前端与可视化。

### 9.1 三条线

**A · Agent + 评测工程师**

- 工具层契约与实现：`analyze_project`（AST + FastAPI 路由内省）、`write_test_file`、`run_pytest` 的 I/O 契约与结果解析；
- 三策略 agent loop：Direct / Plan-and-Execute / ReAct + Reflection；
- Reflection 契约落地（只修测试、`suspected_code_bug` 标记）；
- Pydantic 结构化输出 + 重试；prompt 工程；MockLLM 设计；
- 种子 bug 埋点设计（翻比较符、改边界、漏校验、错状态码 + ground truth）；
- 评测 harness：污染隔离流程、捕获率/假阳性/各指标计算、3 次重复统计、产出对比表数据。
- **简历点**：手写 agent loop、ReAct + Reflection、工具调用、结构化输出、策略对比与评测方法论。
- **依赖**：B 的执行器实际跑 `run_pytest`（A 定契约，B 实现执行）。

**B · 后端 / 平台工程师**

- FastAPI API（V1 子集，见 §5）；
- 14 张表 SQLAlchemy 模型 + Alembic 迁移（见 §2）；
- Celery/RQ 异步编排；
- 运行状态机（status/stage 分离、失败处理、retry 建新 run、幂等）；
- subprocess 隔离执行器（独立 venv、路径白名单、硬超时）；
- `trace_steps`/`run_events` 持久化（先写过程再改状态）、`run_artifacts` 本地存储；
- docker-compose（pg + redis）。
- **简历点**：任务状态机、异步任务、隔离执行、可观测 trace 持久化、数据建模。
- **依赖**：跟 A 共定 trace 事件 schema 与工具 I/O 契约。

**C · 前端 + 可视化工程师**

- Vue 3 + TypeScript + Vite 前端，V1 优先 trace + report 两页，余力做项目/计划/运行页；
- 评测结果可视化：策略对比表、捕获率/假阳性图表（吃 A 的 harness 输出）；
- 3-5 个 FastAPI demo 项目的**应用骨架**（被测对象；bug 埋点由 A 设计 ground truth，C 协助落地 `variants/*`）；
- README、架构图、demo GIF。
- **简历点**：前端工程、Agent trace 可视化、数据可视化、项目门面。
- **依赖**：A 的 harness 数据 + B 的 API；前期用 mock 数据先跑前端。

### 9.2 并行前提：第 1-2 天先锁 4 个契约

不锁契约，「并行」就是各写各的对不上：

1. **14 表 schema**（B 定，A/C 用）；
2. **工具 I/O 契约**（A 定，B 实现 `run_pytest` 执行）；
3. **API 契约**（B 定，C 用）；
4. **trace 事件 schema**（A + B 共定）。

**MockLLM 是解耦器**：契约 + Mock 一就位，A/B/C 三线全速跑，最后集成。

### 9.3 时序要点

- **A**：第 1 周做 Agent 智能层；第 2 周做评测 harness 与实验。两块不同时压。
- **C**：前期做不依赖别人的——demo 应用骨架、前端骨架读 mock 数据；中期接 B 的真实 API；后期做评测结果可视化。
- **B**：第 2 周主体稳定后，腾手支援 C 的前端页面或 A 的实验集成。
- **进度紧张时**：先砍前端到只剩 trace + report 两页，**绝不砍评测深度**——评测是灵魂。
