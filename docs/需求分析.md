# TRACE 需求分析文档

## 1. 文档定位

本文档是 TRACE 的开发蓝图，用于统一产品边界、核心流程、验收口径和后续系统设计。文档描述的是系统最终形态，但实现必须分阶段推进。任何新增功能如果不能服务“测试生成、执行、反思、追踪、评测”这条主线，都默认不进入核心范围。

## 2. 一句话定位

TRACE 是一个面向 Python/FastAPI 后端项目的测试生成 Agent 平台。系统通过受控工具分析项目、生成 pytest 测试、执行测试、读取失败信息并进行一次有约束的 Reflection 修正，最终保存 Agent 轨迹、测试报告和策略评测结果。

核心判断：

- 管理系统是 control plane，不是产品主体。
- Agent 执行闭环是 execution plane，是系统主体。
- 评测不是装饰，必须能回答“生成的测试能不能抓到真实缺陷”。

## 3. 核心原则

1. 真实执行优先：测试必须在真实项目和真实 pytest 环境中运行，不能只停留在生成文本。
2. 轨迹可审计：Agent 的规划、工具调用、观察、反思和报告必须结构化落库。
3. 工具边界清晰：Agent 只能通过白名单工具访问项目，不能直接执行任意 shell。
4. Reflection 有红线：只修生成测试，不修业务代码，不允许通过削弱断言来“变绿”。
5. 评测看 bug 捕获：覆盖率和通过率只能辅助判断，种子 bug 捕获率是核心指标。
6. Docker 不是万能安全：Docker 是隔离边界之一，不等于天然安全沙箱，文档和展示都不能夸大。

## 4. 目标用户

- 后端开发者：为 FastAPI 模块生成初版回归测试。
- 测试开发者：获得可运行、可追踪、可复盘的自动化补测流程。
- Agent 工程人员：比较 Direct、Plan-and-Execute、ReAct + Reflection 等策略。
- 项目展示评审者：判断系统是否同时体现 AI Agent 能力和后端工程能力。

## 5. 非目标

TRACE 第一主线不包括：

- 通用聊天机器人；
- 通用 RAG 平台；
- Jira、禅道、TestRail 替代品；
- 任意语言、任意框架测试平台；
- 自动修改业务源码的全自动编程 Agent；
- 训练、LoRA、DPO 或模型微调平台；
- 复杂组织权限和审批流。

这些能力可在未来扩展，但不能压过测试生成闭环。

## 6. 分阶段范围

### 6.1 V1：单机闭环

V1 要跑通最小但完整的闭环：

- Python/FastAPI + pytest；
- 本地 examples 或白名单目录；
- 内部 Runner 或独立 subprocess 执行，先保证路径白名单、硬超时和结果结构化；
- 项目、测试计划、测试运行、报告、轨迹；
- MockLLM 和至少一种真实 LLM 接入方式；
- `analyze_project`、`write_test_file`、`run_pytest` 等核心工具；
- 一次 Reflection；
- Markdown/JSON 测试报告；
- 最小 seeded bug 评测：1 个 demo 项目、5-6 个 bug 变体、`seeded_bug_capture_rate` 和 `clean_false_positive_rate` 两个指标；
- V1 评测可用脚本和固定目录实现，干净代码与 bug 变体可以先是 `eval/demo/clean`、`eval/demo/variants/*`，不要求完整项目快照和评测表；
- 基础策略对比和人工评分入口。

### 6.2 V2：评测与复现

V2 强化 Compared & Evaluated：

- 项目快照；
- 运行配置；
- 策略版本和 Prompt 版本；
- 多项目种子 bug 数据集；
- 正式评测数据模型和重放记录；
- bug 变体批量运行；
- 策略排行榜；
- 成本、耗时、稳定性统计；
- 测试用例采纳和人工有效性评分。

### 6.3 V3：平台化 Runner

V3 扩展执行面：

- 本地或自托管 Runner；
- Runner 注册、心跳、标签和调度；
- Runner 主动拉任务、鉴权回传结果、预签名 URL 上传 artifacts；
- Docker 沙箱加固；
- 产物存储；
- 多项目、多策略、多 Runner 实验；
- 更完整的日志、指标和链路追踪。

## 7. 核心概念

- Project：被测 FastAPI 项目，是业务容器。
- Project Snapshot：项目在某一时刻的可复现代码版本。
- Runtime Profile：Python 版本、依赖安装、测试命令、环境变量和资源限制。
- Strategy Version：不可变策略版本，绑定模型、Prompt、工具权限和工作流。
- Test Plan：用户定义的测试目标、范围和预算，可保存默认策略，但不固化策略。
- Test Run：一次实际 Agent 执行流程，运行级固化策略版本、模型配置和工具 Schema。
- Run Plan Item：某次运行中 Agent 拆出的测试任务。
- Run Attempt：某个任务的一次生成和执行尝试。
- Generated Test File：Agent 生成或修复后的测试文件版本。
- Generated Test Case：文件中的测试用例逻辑单元，用于采纳、评分和评测。
- Pytest Case Result：pytest 收集到的单个测试项执行结果。
- Trace Step：Agent 或系统执行过程中的有序事件。
- Run Artifact：日志、JUnit XML、coverage、diff、报告导出等大体积产物。
- Test Report：最终人类可读和机器可读的结果汇总。
- Eval Dataset：评测数据集。
- Eval Task：评测任务，例如某个项目中的某个接口或模块。
- Seeded Bug：人工设计的已知缺陷类型。
- Bug Variant：某个缺陷在代码中的一个具体变体。
- Experiment：一组策略、任务、快照、bug 变体和重复次数构成的评测实验。

## 8. 核心业务流程

### 8.1 普通测试生成流程

1. 用户创建项目。
2. 用户创建或选择项目快照。
3. 用户创建测试计划。
4. 用户选择运行配置和策略版本；若不选择，则使用测试计划默认策略。
5. 系统创建测试运行并在 run 级固化策略版本，然后进入异步编排。
6. Agent 通过工具分析项目结构、路由、函数和已有测试。
7. Agent 拆分运行任务并生成 pytest 测试文件。
8. Runner 在受控环境中执行 pytest。
9. 系统解析 pytest 结果，保存测试用例级结果。
10. 若失败且允许 Reflection，Agent 读取结构化失败上下文并修复一次。
11. Runner 重跑修复后的测试。
12. 系统保存轨迹、产物和报告。

### 8.2 Reflection 流程

Reflection 只处理生成测试自身的问题，例如：

- import 路径错误；
- fixture 缺失；
- 调用签名错误；
- 测试数据构造错误；
- 断言写反；
- 对 FastAPI 客户端使用方式不匹配。

Reflection 禁止：

- 修改业务源码；
- 删除核心断言；
- 把具体断言改成空洞断言；
- 使用 `pytest.skip` 或直接跳过失败；
- 伪造通过结果；
- 为了通过而适配明显错误的业务行为。

如果不削弱断言就无法通过，系统应标记为“疑似代码缺陷”，保留失败并写入报告。

### 8.3 策略评测流程

评测必须避免污染。正确流程是：

1. 在干净代码上运行 Agent，生成测试。
2. 若允许 Reflection，先在干净代码上完成一次修复，得到最终通过的测试集。
3. 保存最终测试集，不再让 Agent 看到 bug 变体。
4. 将同一批最终测试重放到 seeded bug 变体上。
5. 如果至少一个测试在干净代码上通过、在 bug 变体上发生与缺陷相关的 assertion failure，判定捕获该 bug。
6. import、collection、fixture、环境错误或运行时 error 不算捕获，只能算测试无效或环境失败。
7. 捕获率分母只包含本次 `eval_task` 目标范围内的 bug 变体；范围外 bug 不计入捕获率。评测中合成的 `test_plan` 范围必须与 `eval_task` 一致，不另起一套。
8. 如果最终测试集在干净代码上失败，判定为假阳性。
9. 多次重复运行，统计均值、方差、成本和耗时。

禁止让 Agent 在带 bug 的代码上重新生成测试，否则模型可能把错误行为当成正常行为。

## 9. 功能需求

### 9.1 项目与快照

- 创建、查看、归档项目；
- 记录项目语言、框架、路径、仓库地址和描述；
- 创建项目快照，记录来源、提交哈希、依赖摘要和入口识别结果；
- 快照一旦参与运行，不允许被原地修改。

### 9.2 运行配置

- 保存 Python 版本、依赖安装命令、测试命令、环境变量模板和资源限制；
- 支持禁网、受控代理、离线依赖包或预构建镜像等依赖方案；
- 每次运行绑定运行配置快照，避免历史结果漂移。

### 9.3 测试计划

- 指定目标范围，例如目录、模块、路由、函数或自然语言目标；
- 保存默认策略版本；实验或单次运行可以覆盖该默认值；
- 配置最大 token、最大工具调用次数、最大运行时间和是否允许一次 Reflection；
- 配置是否保存完整轨迹、是否采集 coverage、是否允许人工采纳。

策略归属规则：策略在 `Test Run` 级固化，`Test Plan` 中的策略只作为默认值。这样同一个测试计划才能被多个策略复用，实验对比才是真正的“同一任务换策略”。

### 9.4 Agent 测试生成

- 通过工具分析项目，不直接把整仓塞给 LLM；
- 输出结构化测试设计和生成测试文件；
- 生成测试必须写入受控目录；
- 生成结果必须记录目标、假设、覆盖点和风险说明；
- 所有模型输出应通过 Pydantic 或 JSON Schema 校验。

### 9.5 测试执行

- 执行 pytest 并采集 stdout、stderr、退出码、耗时和失败详情；
- 解析 pytest 收集项，保存测试用例级结果；
- 支持 JUnit XML、coverage 和生成文件副本作为 artifacts；
- 超时、进程异常、依赖安装失败都必须转成结构化错误。

### 9.6 轨迹与报告

- Trace Step 至少记录序号、类型、输入摘要、输出摘要、工具名、耗时、token、状态和错误；
- 报告必须包含生成用例、执行结果、失败原因、Reflection 变更、风险点、成本和工具调用统计；
- 报告同时支持页面展示、JSON 输出和 Markdown 导出。

### 9.7 策略评测

评测能力至少支持：

- 数据集和任务定义；
- seeded bug 和 bug variant 管理；
- 多策略、多任务、多重复运行；
- 干净代码假阳性统计；
- bug 捕获率统计；
- 成本、耗时、稳定性和人工评分统计；
- 失败案例聚合。

核心指标：

- `seeded_bug_capture_rate`：种子 bug 捕获率，头号指标；
- `clean_false_positive_rate`：干净代码假阳性率；
- `test_collection_success_rate`：pytest 收集成功率；
- `valid_test_generation_rate`：语法、导入、fixture 层面可运行比例；
- `meaningful_assertion_rate`：人工量规判断的有效断言比例；
- `reflection_valid_fix_rate`：Reflection 正确修复测试错误的比例，必须满足修复后在干净代码上通过，且没有删除、跳过或削弱核心断言；
- `cost_per_captured_bug`：每捕获一个 bug 的平均成本；
- `avg_tool_calls`、`avg_tokens`、`avg_duration_ms`：工程成本指标。

覆盖率只能作为辅助指标。

人工断言有效性评分采用系统设计文档中的统一 1/3/5 量规。`meaningful_assertion_rate` 可按 `human_meaningfulness_score >= 3` 的测试用例数除以有效生成测试用例数计算。规则检查只能做辅助预筛，不能替代人工量规。

### 9.8 系统管理与审计

- 查看异步任务状态；
- 取消运行和重试运行；
- 查看 Runner 状态；
- 查看策略、模型、Prompt 和工具 Schema 版本；
- 查看运行事件、错误分类和审计日志。

## 10. 非功能需求

### 10.1 可复现性

每次运行必须绑定：

- 项目快照；
- 运行配置快照；
- 策略版本；
- Prompt 版本；
- 模型配置；
- 工具 Schema 版本；
- 依赖安装方式；
- 评测数据集版本。

Prompt 版本和工具 Schema 版本必须能找回当时正文，不能只保存字符串版本号。可以使用不可变数据库记录、artifact，或仓库 commit + path + content hash 映射。

### 10.2 安全性

- Agent 只能通过白名单工具操作项目；
- 工具层必须做路径规范化和白名单校验；
- 生成测试默认只能写入受控目录；
- Runner 默认不使用特权容器；
- 禁止挂载 Docker socket；
- 敏感环境变量和日志必须脱敏；
- Docker 只是隔离层之一，不能作为绝对安全保证。

### 10.3 可靠性

- 状态机必须显式；
- 关键阶段必须幂等；
- 任何失败路径都要先保存 trace 和事件，再更新最终状态；
- Worker 异常、Runner 超时、模型调用失败、非法 JSON 都必须有明确错误分类；
- 重试不能重复创建同一业务记录。

### 10.4 可观测性

- 结构化日志必须带 `run_id`、`attempt_id`、`stage`、`event_type` 和 `duration_ms`；
- 运行过程必须能按时间线回放；
- 报告应能链接到关键 trace step 和 artifacts；
- 评测结果必须能追溯到具体数据集、bug 变体和策略版本。

### 10.5 可扩展性

系统后续应能扩展到：

- 更多 Python Web 框架；
- OpenAPI 辅助测试生成；
- 更多模型提供方；
- 多 Runner 调度；
- 更严格容器安全策略；
- 更细粒度的失败归因。

## 11. 验收标准

### 11.1 功能验收

- 能导入或绑定一个 FastAPI 项目；
- 能创建测试计划并触发运行；
- 能生成 pytest 测试并真实执行；
- 能保存 Agent trace、测试结果和报告；
- 能进行一次受约束 Reflection；
- 能展示测试用例级结果；
- 能执行至少一组 seeded bug 评测。

### 11.2 工程验收

- API、Worker、数据库、队列和执行器职责清晰；
- 核心数据模型能支撑复现和评测；
- 运行状态机无悬空状态；
- artifacts 不直接塞进主业务表；
- 核心工具返回结构化结果；
- Docker 和 Runner 边界没有夸大安全能力。

### 11.3 效果验收

- 干净代码上生成测试应具备较低假阳性；
- seeded bug 上应能量化捕获率；
- Reflection 应提升有效运行率，而不是削弱测试；
- 报告足以让第三方判断失败原因和测试质量。

## 12. 主要风险

- 生成测试能跑但断言无意义；
- 通过率和覆盖率被误当成核心效果；
- Reflection 为了通过而删断言；
- 策略版本漂移导致实验不可复现；
- pytest 结果只存日志，无法做用例级分析；
- Docker 沙箱边界设计不足；
- Runner 与中心后端协议不清导致状态错乱。

这些风险都必须通过数据模型、工具契约和评测流程解决，不能靠 Prompt 祈祷。

## 13. 结论

TRACE 的最终形态是一个可管理、可执行、可追踪、可评测的测试生成 Agent 平台。它的价值不在于“模型写了多少测试”，而在于系统能否证明这些测试在真实项目中可运行、可解释，并且能抓到已知缺陷。
