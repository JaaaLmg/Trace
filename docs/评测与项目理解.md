# 评测算法与项目理解方向决策稿

## 0. 用途

这份文档解释 TRACE V2 为什么这么做，以及后续分工时需要守住哪些边界。它不是历史日志，也不是验收清单。

当前状态看 `docs/V2实施清单.md`。
V2.1 收工证据看 `docs/V2实施归档.md`。

## 1. 总判断

TRACE 的核心不是“让 LLM 多写几个测试”，而是建立一把能信的尺子：

```text
测试生成能力 = 在 clean 代码上不误报 + 在已知 bug 变体上能捕获 + 证据链可复查
```

所以 V2 的判断顺序是：

1. 先保证评测有效。
2. 再提高项目理解质量。
3. 最后比较策略优劣。

如果 clean 测试集本身不可复现，capture rate 没有意义。这个判断已经在 V2.1 中落地为 `metric_status`、clean replay、flaky gate、probe check、EvaluationEvent 和 source context gate。

## 2. 评测算法原则

### 2.1 本质是 mutation testing

TRACE 的 seeded bug / auto mutation 评测，本质是 mutation testing：

- clean snapshot：没有 bug 的基线。
- bug variant：带已知缺陷的变体。
- generated test set：模型生成并被冻结的测试集。
- capture：同一测试在 clean 上通过、在 variant 上失败。

这比“模型说它测到了什么”可靠。评测只认可执行结果和可审计证据。

### 2.2 无效测试集不能算 0 捕获

这条是 V2.1 的关键口径：

- provider failure 不是 0 捕获。
- artifact hash mismatch 不是 0 捕获。
- clean replay collection error 不是 0 捕获。
- context incomplete 不是 0 捕获。
- flaky clean replay 不是 0 捕获。

它们都应该进入 `invalid_test_set` 或 experiment failed。

### 2.3 Probe 是分母卫生

如果 mutant 本身无效或等价，把它放进分母会污染 mutation score。

因此：

- hand-crafted seeded bug 需要 probe。
- auto mutation confirmed variant 需要 probe。
- probe 必须在 clean/buggy 两侧分别符合期望。
- probe failed 是评测构造问题，不是模型能力问题。

### 2.4 Mock 与真实 LLM 分开解释

MockLLM 适合验证 harness、状态机、schema、UI 和确定性流程。

真实 LLM 适合验证 provider、prompt、token、模型输出和真实失败形态。

两者都需要，但不能混成一个结论。

### 2.5 指标要表达“不知道”

好的指标不应该用 0 掩盖未知状态。

因此：

- `cost_per_captured_bug = null` 表示没有捕获，不写 0。
- `metric_status` 明确表达可评价性。
- `capture_matrix_counts` 表达 repeat 层面的稳定性。
- `sampled_mutation_score.status` 区分 `ok / not_applicable / incomplete_replay`。

## 3. 项目理解原则

### 3.1 问题不是上下文越大越好

真实问题是证据是否准：

- 目标源码是否正确。
- helper dependency 是否相关。
- fixture / existing tests 是否真实。
- request/response model 字段是否来自代码。
- failure context 是否能定位到当前失败。

盲目扩大 token window 只会把噪声塞给模型。

### 3.2 Context Builder 是证据预算器

Context Builder 的职责不是拼 prompt，而是做预算内的证据选择：

1. target source 优先。
2. dependency / reference 次之。
3. model schema、fixture、existing tests 是 support evidence。
4. failure context 和 EvaluationEvent 只在需要时进入。
5. 缺失、截断、降级必须进入 retrieval trace。

### 3.3 Agentless 宏观，Tool-calling 微观

宏观流程应保持确定性：创建 plan、生成、guard、replay、metrics，不让模型控制系统状态机。

微观检索可以工具化：rg、ast-grep、LSP、source slice、failure context 都可以是工具，但输出必须结构化、可审计。

### 3.4 Contract Guard 是系统边界

Prompt 只能减少坏输出，不能替代 guard。

Contract Guard 要继续承担：

- 弱断言拒绝。
- 目标绑定检查。
- fixture / request model / response model 证据检查。
- 明显矛盾 oracle 拒绝。
- 结构化 retry 的 violation feedback。

## 4. V2.2 待选能力边界

本节只记录后续可选能力边界，不规定执行次序。具体选择由实际分工时决定。

### 4.1 执行安全

目标：让 TRACE 能更可信地运行外部项目。

范围：

- Docker executor。
- runtime profile executor 配置。
- artifact / workspace mount 策略。
- timeout / resource limit / cleanup 规则。

收益：工程可信度提升。
风险：平台适配和本机环境成本高。

### 4.2 数据集与 mutation

目标：让评测分母更丰富。

范围：

- 完整 Dataset UI。
- task / seeded bug / variant 编辑器。
- auto mutation operator 扩展。
- batch dry-run、人工确认、probe 编排。

收益：评测更像真实 benchmark。
风险：若 UI 和 mutation engine 同时扩，容易过大。

### 4.3 项目理解

目标：减少模型因为上下文错而写坏测试。

范围：

- 拆 `source_context.py`。
- 强化 retrieval trace。
- 更准确 call slice。
- 更深 response/oracle evidence。
- EvaluationEvent / failure context 的 prompt-safe 策略升级。

收益：直接改善真实 LLM 输出质量。
风险：容易变成“再加一种 provider”的堆叠，要先拆模块。

### 4.4 产品化管理

目标：让实验配置可视化、可复现。

范围：

- Prompt version 管理页。
- Runtime profile 管理页。
- Tool schema version 管理页。
- experiment config diff / clone。

收益：演示和使用体验提升。
风险：如果底层执行和数据集还不稳，页面会先于能力成熟。

## 5. V2.2 二人分工原则

V2.2 的分工应按失败模式和工程边界切，不按策略名切。Direct / Plan / ReAct 都要经过同一把尺子，所以不能把策略拆成不同所有权。

A 线负责“尺子和执行”：

- Docker executor / subprocess executor 边界。
- runtime profile、timeout、workspace mount、artifact cleanup。
- clean replay、variant replay、parallel / incremental replay。
- metrics、EvaluationEvent、artifact hash、DB/API evidence。
- 目标是让一次实验是否可评价、为什么不可评价、证据在哪里，都能被审计。

B 线负责“证据和生成”：

- source context 模块化和 retrieval trace。
- fixture/model/existing tests/failure context/EvaluationEvent 的证据选择。
- Contract Guard、oracle evidence、structured retry。
- Dataset UI、seeded bug / variant 编辑、auto mutation confirmation。
- 目标是让模型只基于可信证据生成测试，证据不足时拒绝或降级，而不是编造。

两条线的稳定接口：

- `EvaluationEvent`：A 产生系统/评测事件，B 只消费 prompt-safe 子集。
- `SourceContextBundle`：B 产生 source evidence，A 只看 completeness / trace / metadata，不解析 prompt 文本。
- dataset / variant schema：B 维护任务和缺陷定义，A 负责 replay 与 metrics 解释。
- experiment metrics API：A 维护聚合口径，B 提供 target/test metadata 让聚合可解释。

协作红线：

- invalid、flaky、provider failure、artifact mismatch 不能变成 0 捕获。
- 缺目标源码、缺模型字段、缺 fixture 证据时不能静默编造测试。
- schema 变更必须有迁移和合同测试。
- UI 只能展示已有能力，不能替底层能力背书。

## 6. 决策表

| 问题 | 决策 |
| --- | --- |
| 评测核心 | mutation testing + clean validity gate |
| 无效测试集 | `invalid_test_set`，不算 0 捕获 |
| replay 是否调用 LLM | 不调用，只执行冻结测试 |
| 真实 LLM 与 Mock | 分开解释 |
| Project understanding | 证据驱动，不靠大上下文硬塞 |
| 检索体系 | rg / ast-grep / LSP / AST slice 多源，但必须结构化 trace |
| Reflection 输入 | 只吃 prompt-safe failure evidence |
| Auto mutation | 先 dry-run + 人工确认 + probe，后续再自动化 |
| 二人分工 | A 管评测尺子和执行平台，B 管项目理解、数据集和生成约束 |
