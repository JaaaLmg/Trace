## 0. 总判断

TRACE 的上限取决于两件事：

- **评测算法**：它是尺子。尺子不准，所有“Agent 变聪明了”的说法都是幻觉。
- **项目理解**：它是能力天花板。模型读不懂真实项目，测试生成和 Reflection 都是在盲写。

顺序必须是：**先铸造可信评测尺子，再优化项目理解能力**。

这条主线不变。但结合 2025-2026 年 Agent 演进趋势，项目理解方案需要从“自研静态大图 + 固定 pipeline”修正为：

```text
宏观 Agentless 管线主导
    +
局部受限 Tool-Calling 小循环
    +
LSP / ast-grep / rg 驱动的动态代码情报
    +
Budgeted Context Builder 组装证据
```

一句话：**不要让模型全局乱逛，也不要把系统写死到只能盲喂上下文。把高质量代码情报封装成强类型工具，让模型在预算和边界内按需查询。**

这条路线有三个工程边界必须提前写清楚：

```text
代码情报不能押宝单一 LSP：必须多源召回、可降级、可审计
mutation 不能默认全量执行：必须分层预算、采样、按目标增量 replay
Tool-Calling 不能无边界探索：必须有状态机、去重、超时和审计日志
```

## 1. 评测算法：先把尺子做硬

### 1.1 TRACE 的评测本质是 mutation testing

`seeded_bug_capture_rate` 在测试生成领域对应 **mutation score**：

```text
mutation score = killed mutants / total mutants
```

当前 seeded bug、bug variant、clean replay 的设计，本质就是 mutation testing。这个方向是正确的，而且比覆盖率更能解释“测试是否真的发现 bug”。

覆盖率只说明代码被执行过，不说明行为被验证过。生成式 AI 很容易刷高覆盖率，但弱断言、空断言、只测不校验都会让覆盖率失真。V2 以后，TRACE 对外应优先讲 mutation score，而不是 coverage。

### 1.2 Probe 是等价变异体排除机制

mutation testing 最大的问题是 **equivalent mutants**：代码变了，但行为没变，任何测试都杀不掉。

`eval/demo/bugs.py` 里的 `probe / clean_value / buggy_value` 正好能做排除：

1. 数据集自检：物化 clean 和 variant 后，验证 probe 在两边输出不同；
2. 捕获二次校验：确认 variant failure 对应目标行为变化，而不是其它副作用；
3. 手工 seeded bug 和核心校准集必须带 probe 或可执行 ground truth matcher。

这不是锦上添花，而是 mutation score 可信的前提。

但 probe 不能被误用成自动 mutation 的硬门槛。`mutmut` / `cosmic-ray` 这类工具会批量生成大量 AST 变异体，不可能为每个自动 mutant 手写 `buggy_value`。因此数据集要分层处理：

- **Hand-crafted / Seeded Bugs**：必须有 probe，用作核心能力定标和论文/展示口径；
- **Auto Mutants**：允许没有 probe，用静态启发式、超时、编译失败、重复 mutant、稳定性检查等过滤噪声；
- **Selected Auto Mutants**：对少量高价值 mutant 可以补行为 matcher，作为自动 mutation 到 seeded bug 的升级路径。

也就是说，probe 保证核心校准集的纯度；auto mutation 提供规模化趋势信号，两者不要混成一个质量承诺。

### 1.3 弱断言问题：评测只认 mutation score，生成阶段加静态预筛

弱断言和 test oracle 是测试生成最难的问题。不要单独发明一套复杂“弱断言评分器”，那会变成另一把不准的尺子。

拍板：

- **评测算法只认 mutation score**：能杀 mutant 的断言就是强断言。
- **生成阶段加廉价静态预筛**：拦截 `assert True`、`assert result is not None`、断言数量下降、skip、吞异常等明显垃圾输出。
- **Reflection 的有效性用行为验证兜底**：如果修复后 clean pass 但 mutation score 下降，不能视为有效修复。

这和 Meta TestGen-LLM 的 filter 思路一致：LLM 输出不是直接采纳，而是必须通过可执行过滤器。

### 1.4 Mock 与真实 LLM 必须分开解释

当前 scripted MockLLM 适合证明 harness 管道连通，不适合证明策略优劣。

V2 报告必须区分：

- `mock/scripted`：工程自检、契约测试、回归测试；
- `real_llm`：真实模型评测结论；
- `provider/model/repeat_count`：每个 clean run 的 `strategy_snapshot.resolved_llm` 是最终真相。

前端和报告都不能把 mock comparison 当真实结论展示。

### 1.5 数据集扩展：V2 优先自动 mutation

数据集路线拍板：

1. **V2 主线优先自动 mutation**：使用 `mutmut` 或 `cosmic-ray` 扩展 mutant 数量。
2. **真实 bug 集后置**：BugsInPy、SWE-bench、SWT-Bench 类数据集价值高，但环境复现成本大。
3. **防数据污染**：真实 LLM 评测集应尽量来自训练截止后、私有、刚开源或经过改写的代码。
4. **未来可做对抗生成**：一个 agent 生成 mutant，另一个 agent 生成 test，用 mutation score 互相压强度。

理由很简单：先用自动 mutation 把评测框架磨硬，再接真实 bug。别一上来被环境依赖、历史项目构建问题拖死。

展示和验收时要分清两种数字：

- **calibrated score**：来自带 probe 的 seeded bugs，适合讲“这个 bug 是否被可靠捕获”；
- **sampled mutation score**：来自自动 mutants，适合看趋势、回归和策略相对变化。

自动 mutation 的分数不能直接替代带 probe 的核心定标分数。

### 1.6 增量与并行 replay 不能太晚

mutation testing 天然昂贵：

```text
策略数 × repeat 数 × task 数 × mutant 数 × pytest 执行成本
```

如果每个 mutant 都跑全量测试，V2 很快会慢到无法研发迭代。

拍板：

- V2.1 可以先跑小数据集全量 replay，目的是锁口径；
- 一旦接自动 mutation，必须做增量 replay 和 mutant sampling；
- 并行 replay 是工程优化，不改变 captured_bug 定义；
- 指标必须记录 sampling strategy，不能把 sampled score 伪装成 full score。

执行分层：

```text
PR smoke
  少量高价值 mutants + target 相关 generated tests

nightly eval
  扩大 mutant sample + 并行 replay + 稳定性统计

release / paper result
  固定数据集全量 replay + 完整审计 artifact
```

`pytest-testmon` 或 coverage.py 只能作为召回信号，不能当唯一过滤器。原因很现实：`a + b` 变成 `a - b` 这种行内 mutation 行号不变，行级 coverage 无法区分。

更稳的增量单位应是 **mutated symbol / target scope**：

```text
mutant -> mutated function/method/route/model field
generated test -> target_function / target_route / covered symbols
replay candidates = tests targeting or covering that symbol
fallback = module-level tests
last resort = full replay
```

最小工程策略：

- mutant operator 白名单：先保留 comparison、boundary、boolean、exception/status 这类业务价值高的 operator；
- 每个 target 限制 mutant 数；
- seeded bug 用 probe / behavior matcher 剔除等价变异体；
- auto mutant 用编译检查、超时、重复 mutant 去重、静态启发式和稳定性检查过滤明显噪声；
- nodeid 到 target 的映射写入 artifact，方便重算；
- 对无法映射的 mutant 明确降级为 module/full replay，不静默漏测。

### 1.7 评测指标需要从 OR 聚合升级

当前 capture matrix 用 “任一 repeat 捕获即 true” 容易夸大稳定性。

后续应升级为：

- 逐 bug 显示 `captured_count / repeats`；
- capture rate 报均值、样本标准差或 bootstrap CI；
- false positive 不只按 run，也可补充按 test case；
- cost per captured bug 在 0 捕获时必须是 `null` 或 `no_bug_captured`，绝不写 0。

### 1.8 Flaky check 是评测可信度的一部分

AI 生成测试很容易出现不稳定测试：依赖当前时间、随机数、dict 顺序、外部网络、共享状态或执行顺序。它可能在一次 clean run 里通过，也可能偶然杀掉 mutant，但这种捕获不能作为可信证据。

因此最终测试集冻结前必须做稳定性检查：

```text
生成测试
  -> clean snapshot 连续运行 N 次
  -> 结果一致且 nodeid 稳定
  -> 冻结 generated_test_set artifact
  -> replay 到 mutants
```

V2 可以先用 `N=3` 作为默认值，允许 runtime profile 覆盖。规则：

- clean 上不稳定的测试不进入 final test set；
- 不稳定原因写入 report quality evidence；
- 如果 Reflection 生成了新版本，也必须重新做 flaky check；
- captured_bug 只能来自 clean 稳定通过、variant 稳定 assertion failure 的 nodeid；
- 为控制成本，variant 侧可只对捕获候选 nodeid 做二次确认，而不是全量重复 replay。

这一步会增加一点时间，但能显著提高 mutation score 的可信度。

## 2. 项目理解：从固定 AST 摘要升级为动态代码情报

### 2.1 问题不是 context 不够大，而是证据不够准

当前 Agent 的核心问题不是 token window 小，而是上下文质量低：

- `analyze_project` 主要给函数名、类名、路由摘要；
- Pydantic 字段、fixture、已有测试风格、跨文件引用缺失；
- prompt 里存在 `[:40]` 类盲截；
- LLM 很少看到足够的行为证据，只能写弱断言。

测试生成需要的不是“更多文件”，而是：

- 目标函数/路由的实现片段；
- 请求/响应模型字段；
- fixture 和已有测试构造方式；
- 边界条件、异常路径、状态码；
- 失败 traceback 对应源码上下文。

### 2.2 Agentless vs Tool-Calling：宏观管线，微观 Agent

2024 年 Agentless 的结论仍然有价值：固定 pipeline 便宜、可控、可审计，适合评测系统。

但 2025-2026 年 Claude Code、SWE-agent、o3、DeepSeek-R1 等趋势说明：具备 test-time compute 的模型，在强约束工具接口下，局部自主探索已经非常有价值。

拍板：TRACE 采用 **Macro-Agentless, Micro-Agent**。

宏观流程仍然固定：

```text
创建 plan
  -> 收集上下文
  -> 生成测试
  -> 执行 pytest
  -> 必要时 Reflection
  -> report / replay / metrics
```

微观阶段允许受限 tool-calling：

- 上下文收集阶段：模型可调用 `find_symbol`、`get_definition`、`find_references`、`get_route_context` 等高层工具；
- Reflection 阶段：模型可按失败 nodeid 查询相关源码、fixture、模型字段；
- 禁止全局漫游、禁止任意写业务源码、禁止无预算循环。

也就是说，不做“让 LLM 自己打开仓库乱翻”的通用 agent，而是做“限定目标、限定工具、限定预算”的项目理解小循环。

### 2.3 多源代码情报：LSP 只是其中一层

旧方案倾向构建轻量 Code Context Graph，并考虑 PageRank。这个方向比盲截好，但仍然有风险：

- Python 动态特性复杂，自研 AST scanner 很容易漏；
- FastAPI、Pydantic、fixture、typing、跨文件引用都容易写成半吊子；
- 大项目里全量图维护和刷新成本高；
- PageRank 容易偏向通用工具函数，而测试生成通常有明确目标。

同时也不能把 LSP 当成唯一真相源。pyright / jedi 在真实工程里有清晰边界：

- 缺 `requirements.txt`、虚拟环境或 `pyproject.toml` 时，definition / references 质量会快速下降；
- 中大型仓库冷启动和 indexing 可能需要分钟级时间；
- `**kwargs`、装饰器、`getattr`、动态注册、FastAPI/Django 的框架魔法会让 references 不完整；
- 沙箱环境和开发机环境不同，LSP 看到的类型世界可能是错的。

因此这里的拍板不是“用 LSP 替代一切”，而是：

```text
不做重型知识图谱。
不优先做 target-biased PageRank。
LSP 作为高置信度证据来源之一。
rg / ast-grep / framework scanner / analyze_project 共同组成降级链。
```

查询优先级按“高精度可用则先用，失败后逐级降级”设计：

```text
cached LSP definition / references / hover
  -> framework scanner / pytest collect（route、fixture、test discovery）
  -> ast-grep / lightweight AST
  -> rg / path / lexical search
  -> runtime probe / import-based introspection（可选，沙箱内）
```

如果 LSP 未就绪、缺依赖或超时，系统不等待全仓 indexing；直接跳到后续层。反过来，冷启动时可以先用 `rg` 做低成本 seed discovery，但一旦定位到候选 symbol，仍应优先尝试 LSP / framework scanner 做高置信度确认。

工程规则：

- LSP 查询必须有 timeout，超时不阻塞主流程；
- LSP 结果带 `confidence/source/latency/stale`；
- project snapshot hash、dependency summary、LSP config hash 作为缓存 key；
- 大仓优先 target_scope 附近的局部查询，不强制全仓 indexing；
- 缺依赖或动态特性导致查询失败时，Context Builder 继续使用 rg / ast-grep / framework scanner；
- 降级必须写入 `context_incomplete` 或 `retrieval_trace`，不能静默假装完整。

这更接近 Claude Code / SWE-agent 的工程路线：不是把世界建成一张大图，而是让 Agent 通过可靠工具动态拿到需要的事实。

### 2.4 LSP as a Tool：给模型高层代码情报接口

不要给模型裸 `cat` 和无限 `read_file`。应该给它面向测试生成的工具：

```text
search_symbol(query)
get_definition(symbol_or_location)
find_references(symbol_or_location)
get_fastapi_route_context(method, path)
get_pydantic_model_schema(model_name)
get_pytest_fixture_context(fixture_or_test_path)
get_call_slice(function, budget)
get_failure_context(nodeid, traceback)
```

这些工具内部可以用 LSP、ast-grep、rg、AST fallback 组合实现，但对 LLM 暴露的是稳定 JSON：

```json
{
  "target": "GET /items/{item_id}",
  "handler": {
    "symbol": "get_item",
    "path": "shop/api.py",
    "start_line": 22,
    "end_line": 41
  },
  "models": [
    {
      "name": "ItemResponse",
      "fields": [
        {"name": "price", "type": "float", "required": true}
      ]
    }
  ],
  "fixtures": ["client"],
  "risk_notes": []
}
```

工具返回必须带：

- `source_path`；
- `start_line/end_line`；
- `content_hash`；
- `retrieval_trace`；
- `context_incomplete` 标记。

这和阶段 0 已锁定的 report/context schema 对齐。

工具失败时必须有固定降级路径。例如 `get_fastapi_route_context(method, path)` 失败：

```text
route cache / framework scanner exact match 失败
  -> LSP 查候选 handler definition / references（如果索引就绪）
  -> ast-grep 搜 @router.get/post 装饰器
  -> rg 搜 path / handler name
  -> 仍失败则 context_incomplete=true，停止该目标查询
```

降级后的生成策略也要收缩：

- 缺目标源码：中断该 plan item，标记 `context_incomplete_blocking`；
- 缺模型字段：禁止编造请求体；
- 缺 fixture：允许生成纯函数测试，但不生成 FastAPI client 测试；
- 缺业务 oracle：禁止生成 `assert result is not None` 伪强断言；
- 证据不足但仍可生成 smoke test 时，该测试不得计入 mutation score 的强断言证据。

### 2.5 Budgeted Context Builder 是核心，不是装饰

模型不是没有推理能力，真正的问题是证据质量和证据组织。

Context Builder 的职责：

1. 把 LSP / rg / ast-grep 结果统一成结构化证据；
2. 在 token budget 内排序和裁剪；
3. 明确标记哪些目标缺证据；
4. 输出给 planner/generator 的上下文必须可审计。

优先级：

```text
1. 目标源码片段
2. Pydantic model / request response schema
3. fixture / existing tests
4. traceback 相关源码
5. 直接依赖的小函数
6. repo map / global summary
```

不要把 RepoMap 当主要理解来源。RepoMap 适合提供全局背景，不适合生成强 oracle。

### 2.6 证据驱动测试生成

Planner/Gernerator 的约束应变成：

- 没有源码证据，不规划异常路径；
- 没有模型字段，不编造请求体；
- 没有明确业务 oracle，不接受 `assert result is not None`；
- 如果上下文缺失，写入 `context_incomplete` 风险；
- Reflection 只能修测试，不允许修改业务源码；
- 失败修复不能删除核心断言、skip、吞异常。

强断言不是 prompt 喊出来的，是上下文里有行为证据才写得出来。

这些约束不能只写在 prompt 里，必须由系统层 Contract Guard 执行。生成测试后至少做三类检查：

1. **断言强度检查**：AST 扫描 `assert True`、`assert x is not None`、断言数量下降、skip、吞异常等明显弱断言；
2. **上下文字段检查**：对 FastAPI/Pydantic 请求体，生成代码使用的字段必须来自 Context Builder 提供的 model schema；无法确认时标 `context_incomplete`，不能静默放行；
3. **目标绑定检查**：生成测试声明的 `target_function / target_route` 必须能映射回 plan item 或 retrieval trace。

不通过时不是“提醒模型下次注意”，而是 `PipelineReject`：

```text
generated_test
  -> contract_guard
  -> pass: write_test_file / run_pytest
  -> reject: Reflection rewrite or fail item with structured reason
```

这样才能把“证据驱动”变成工程约束，而不是 prompt 愿望。

### 2.7 Micro-Agent 的状态机和停止条件

局部 Tool-Calling 只能用于收集证据和修复测试，不能变成自由探索。实现上必须是有限状态机：

```text
START
  -> NEED_CONTEXT
  -> TOOL_CALL
  -> EVALUATE_EVIDENCE
  -> ENOUGH_CONTEXT -> BUILD_CONTEXT -> DONE
  -> NEED_MORE_CONTEXT -> TOOL_CALL
  -> BUDGET_EXHAUSTED -> CONTEXT_INCOMPLETE -> DONE
  -> BLOCKING_MISSING_CONTEXT -> ABORT_ITEM -> DONE
```

硬约束：

- 每个 plan item 最大工具调用次数，例如 6 次；
- 单个工具同一参数最多重试 1 次；
- 重复查询直接拒绝，并写入 trace；
- 总 token budget 和 wall-clock timeout 必须可配置；
- 工具只读，业务源码不可写；
- 每次工具调用必须记录 reason、input、output hash、latency、confidence；
- 工具失败进入 `failed_query_log`，不允许模型原地反复调用同一个失败查询。

上下文不足时按严重程度处理：

- 缺目标源码：中断该 plan item，标记 `context_incomplete_blocking`；
- 缺请求/响应模型字段：禁止编造请求体；
- 缺 fixture：允许生成纯函数测试，但不生成依赖 fixture 的集成测试；
- 缺业务 oracle：禁止生成弱断言，或者生成 smoke test 但不计入 mutation score 的强断言证据；
- 缺次要依赖：允许生成，但 report 必须标出 `context_incomplete` 风险。

这样做的目的不是让模型少思考，而是让 test-time compute 发生在可审计的边界里。

## 3. 关键决策点：已拍板

### 3.1 项目理解走 Agentless 还是自主 Agent？

拍板：**管线主导 + 局部 Tool-Calling**。

宏观流程固定，微观信息收集和 Reflection 可允许模型调用高层代码情报工具。禁止无边界探索。

### 3.2 做轻量图还是完整知识图谱？

拍板：**都不做成主线**。

不做 Neo4j 式完整知识图谱，也不优先做自研全量 Code Context Graph。V2 优先做 LSP / ast-grep / rg 驱动的动态查询 API，加轻量缓存。

### 3.3 检索排序做到哪一层？

拍板：**lexical + LSP references/definition + token budget 截断**。

target-biased PageRank 暂不作为 V2 主线。测试生成通常有明确目标，顺藤摸瓜找两层依赖比全局排序更有效。

### 3.4 数据集扩充走哪条？

拍板：**自动 mutation 优先**。

V2 先用 `mutmut/cosmic-ray` 或自定义 mutation 扩展 mutant；真实 bug 数据集作为 V2 后半段或 V3。带 probe 的 seeded bugs 负责核心定标，auto mutants 负责规模化趋势，不混成同一种可信度。

### 3.5 弱断言怎么处理？

拍板：**评测靠 mutation score，生成靠 Contract Guard**。

不要用启发式评分器替代 mutation score。静态预筛、字段校验、目标绑定检查只作为系统层过滤器，用来拒绝明显垃圾测试和省执行成本。

### 3.6 Scalability 要不要做增量/并行重放？

拍板：**要，但和指标口径解耦**。

V2 接自动 mutation 后必须做增量 replay；并行化和 coverage 过滤不能改变 captured_bug 定义。

## 4. 后续落地路线

### 阶段 A：铸造金尺子

目标：让评测结果可信。

- probe 接入 dataset seed / variant materialize；
- probe 接入捕获二次校验；
- 自动 mutation 数据集 POC；
- sampled mutation score 与 calibrated score 分开展示；
- final test set 冻结前加入 flaky check；
- mock/scripted 与 real_llm 展示彻底分离；
- capture matrix 从 bool 升级到 repeat 命中比例；
- 增量 replay 设计，不改变指标口径。

### 阶段 B：动态代码情报局

目标：用成熟工具替代半吊子自研 scanner。

- 封装 `rg` lexical search；
- 封装 `ast-grep` 结构化搜索；
- 封装 Python LSP 的 definition/references/hover/symbol；
- 保留 `analyze_project` 作为 cheap summary；
- 明确 LSP timeout、缓存 key、降级顺序和 `context_incomplete` 记录；
- 所有工具输出统一带路径、行号、hash、trace。

### 阶段 C：Context 弹药库

目标：把项目事实变成模型能用的证据。

- Budgeted Context Builder；
- route context；
- Pydantic model schema；
- fixture / existing test context；
- failure context；
- Contract Guard：断言强度、字段来源、目标绑定；
- context_incomplete 风险记录；
- report 展示测试清单、目标映射、断言摘要、失败分类、Reflection 证据和上下文完整性。

### 阶段 D：微型 Tool-Calling Agent

目标：在上下文收集和 Reflection 中引入受限自主查询。

- 工具调用预算；
- 最大轮数；
- 同参数重复查询拒绝；
- `PipelineReject` / `context_incomplete_blocking` 状态；
- 禁止业务源码写权限；
- 工具 schema 版本化；
- trace 记录每次查询、理由、结果 hash；
- 对 mock/real LLM 都可回归测试。

## 5. 和 V2 当前清单的关系

当前 `V2实施清单.md` 阶段 0 已经锁定契约：

- `captured_bug` 只属于 variant replay；
- `false_positive` 只属于 clean run；
- replay 不调用 LLM；
- `llm_override` 只是覆盖层；
- resolved spec 冻结进 clean run 的 `strategy_snapshot`；
- report schema 能表达质量证据；
- context schema 有路径边界和 `context_incomplete`。

本文档不是要推翻 V2 清单，而是给阶段 2 之后的 A 线实现提供方向：

```text
先接 report/context schema
再接 probe/mutation
再接动态代码情报
最后接受限 Tool-Calling
```

不要为了追 Claude Code 的形，提前做一个不可审计的通用 Agent。TRACE 的价值是：**每个指标、每个测试、每次上下文读取都能追溯。**
